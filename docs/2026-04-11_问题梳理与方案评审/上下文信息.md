# 雷达信息项目概况 —— Prompt编写

## 你的项目是做什么的？
我的项目是一个高性能高实时性的三阵面ku波段雷达信号处理系统。详细信息如下：
1. 项目的系统定位：QDGZ300 是三面阵雷达后端系统。
2. 软件目标运行平台：麒麟 V10 ARM64
3. 项目软件大体划分（单指运行在后端服务器上的信号处理系统）：
    - 信号处理软件（数据接收模块+信号处理算法模块+通信检查等其他功能）注解：我负责数据接收模块的实现和信号处理软件的初步设计（架构+数据流+信号流），具体算法我不是很了解。
    - 数据处理软件（数据处理算法+波束安排表生成与发送+波束调度/任务调度）注解：此模块不是我负责，具体职责不清楚。
    - 显控软件（负责下发控制参数，显示航迹信息，提供各种校准检查的界面和入口，提供所需调试界面和数据展示的可视化支持） 注解：此软件与信号处理系统耦合性较弱，边界职责和接口清晰，不在我们的讨论范围之内。
    > 注：当前我们主要关注信号处理数据流和整体架构的设计工作。
4. 服务器使用四口万兆网卡和前端阵面（AD采样+FPGA+同10GbE网卡光口）进行数据交互，三个阵面对应服务器网卡的三个网卡，预留一个作为备用。网络拓扑采用光纤直连（无交换机），每条链路支持全双工 10 Gbps（发送 + 接收）。
5. 雷达时序已经安排，一个PRT为119us，为三脉冲三接收区构型，每个PRT采样带你为5755，IQ为int16，四通道（和差差辅助，当前不做辅助），所以一个PRT数据包大小为 5755 * 4 * 3 = 66900 Bytes，每秒钟需要处理的PRT数量为 1s / 119us = 8403 PRT/s，所以每秒钟需要处理的数据量为 66900 Bytes * 8403 PRT/s = 562.5 MB/s，约等于 4.5 Gbps。具体的数据包如何划分，如何安排字段见 docs\2026-04-11_问题梳理与方案评审\回传字段表v2_0.md.
6. 数据接收模块我已经初步实现了。对信号处理算法实现模块输出的单位为一个完整的数据单元+前端回传的波位安排表（或者可能是接收模块解析波束安排表之后自己拼的CPI上下文，共信号处理算法流水线消费），他们之间的传递关系使用SPSC无锁队列实现。
7. 我翻到了资料：

### 💡 核心架构逻辑总结

1.  **同节点数据交换 (Local Affinity):**
    *   GPU (MR-V100) 与 Intel 10G 网卡均位于 **NUMA Node 1**。
    *   这种物理布局极大降低了雷达原始数据从网卡到 GPU 显存的传输延迟，符合高性能信号处理的“确定性”要求。

2.  **生态兼容性 (Compatibility):**
    *   采用 **CoreX 软件栈** 提供类 CUDA 10.2 的 API 接口。
    *   原有的 CUDA 代码（如 FFT 算法）可通过 `clang++` 配合特定 Target 指令直接编译迁移，无需重写底层算子逻辑。

3.  **容器化支持 (Container Ready):**
    *   通过 `iluvatar-node-init.service` 自动创建设备节点并赋予 `666` 权限。
    *   服务被配置为在 Docker 启动前执行，解决了容器内无法识别宿主机的 `/dev/ix*` 设备的常见问题。

4.  **开发工具链:**
    *   主要依赖 `/usr/local/corex/bin/clang++` 进行编译。
    *   头文件位于 `/usr/local/corex/include`，动态库位于 `/usr/local/corex/lib64`。
