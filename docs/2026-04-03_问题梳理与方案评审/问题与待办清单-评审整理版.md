## 二、必须先明确的设计原则

### 1. 不要把“统计”整体当成调试功能一刀切掉
生产环境仍然必须保留低开销的最低可观测性：

- 基本收包数 / 丢包数
- backend error
- pool exhaustion
- ring / mempool 高水位
- 当前 CPI 基本状态

真正应该裁剪的是：

- invalid packet preview
- 详细 reject dump
- 原始帧全量录制
- 高频状态面板
- 高频详细日志

---

### 2. 动态 PRT 检测不能只改一个赋值语句
必须先补齐状态机规则，否则很容易返工。至少要明确：

- 控制包先到时如何绑定
- 数据包先到、控制包后到时如何处理
- 同一 CPI 多个控制包冲突时如何处理
- 控制包给出的 `n_prt` 小于已观测最大 `prt` 时如何处理
- timeout 以什么时间点为锚点

---

### 3. 双缓冲 CPI 不能只按 `%2` 简单实现
双缓冲方向是对的，但还必须补：

- CPI 跳变处理
- 迟到包策略
- previous / current / next 的容忍边界
- finalize 触发规则与告警口径

---

### 4. 内联优化不是当前最优先事项
在没有完成热路径裁剪、零拷贝收口、资源参数稳定之前，不建议优先做解释层内联优化。
当前更可能影响性能的是：

- 深拷贝
- 文件写入
- 状态面板
- 调试输出
- cache miss / 分支预测失败
