## 物理拓扑与链路背景

### 阵面与网口
- 系统有 3 个固定阵面
- 每个阵面覆盖约 120° 扫描范围
- 每个阵面前端都有 ADC 采样与 FPGA 逻辑
- FPGA 将原始回波采样组装为上行 UDP 数据包发往服务器
- 服务器有 4 口 10GbE 网卡
- 其中：
  - receiver0 ↔ 1号阵面
  - receiver1 ↔ 2号阵面
  - receiver2 ↔ 3号阵面
  - receiver3 预留
- 网络拓扑为光纤直连，无交换机
- 每条链路支持全双工 10Gbps

### 数据规模（当前理解）
- 一个 PRT = 119us
- 三脉冲三接收区构型
- 每个 PRT 采样带点数 = 5755
- IQ = int16
- 四通道（当前辅助通道先不处理）
- 单个 PRT 数据量按当前口径约为 66900 Bytes
- 每秒 PRT 数约为 8403 PRT/s
- 每秒处理数据量约为 562.5 MB/s，约 4.5 Gbps
- 前端发送单位是数据包，一个 PRT 可能拆为多个数据包
- 接收端会把包片组装成完整数据，再进一步形成完整 CPI 数据单元

### CPI 相关
- 当前计划以 50 个 PRT 作为一个 CPI 基准
- 一个完整 CPI 可能由 2 / 4 / 8 / 16 个 CPI 基准拼接而成
- 扫描边界能量不足时，计划通过拼波位/拼泊位方式提高检测精度
- 在当前设想中，接收模块输出的是“完整 CPI 任务对象”，包含：
  - 原始 IQ 数据
  - 波束安排表相关信息 / 或解析后形成的 CPI 上下文
  - 必要元数据
  - 供算法流水线直接消费

## 当前对系统边界的理解

### 三阵面关系
- 三阵面在信号处理阶段基本独立，数据不交叠
- 波束安排表下发也是独立的
- 更后面的阵面融合可能属于数据处理阶段，而非当前重点
- 但一个重要事实是：三路数据最终要在同一个 GPU 上处理
- 因此，前端接收是独立流，GPU 资源使用可能是共享竞争关系

### 控制面 / 数据面
- 我希望信号处理软件中明确区分控制面与数据面
- 显控/外部控制会影响：
  - 工作模式
  - 参数装载
  - 启停控制
  - 校准流程
- 波束安排表/任务调度信息，我目前更倾向将其理解为“介于控制面与数据面之间的运行时上下文输入”

## 运行平台与资源约束

### CPU / NUMA
- CPU：Phytium S5000C B-EN32-C
- 32 核，单插槽，每核 1 线程
- 频率约 50MHz - 2.3GHz
- NUMA 分为 2 个节点：
  - Node 0: CPU 0-15
  - Node 1: CPU 16-31
- 内存：64GB DDR5 ECC

### GPU / 加速器
- 当前 1 张 GPU，未来可能扩展到 2 张
- 型号：Iluvatar MR-V100
- GPU NUMA 亲和性：Node 1
- GPU 与万兆网卡同 NUMA 节点，适合同节点数据交换
- 软件栈：CoreX 4.3.8
- 类 CUDA 10.2 兼容层
- 编译器：clang 18.1.8
- 可能存在 GPU 特性支持边界，需要以后查官方文档确认
- 当前我不知道 CPU→GPU 的最佳数据路径细节
- 我认为 GPU 很可能成为系统瓶颈资源

### 当前接收技术栈
- 数据接收层当前支持 DPDK 和 Linux Socket 策略模式
- 两种方式在进入业务层时统一为完整数据包处理逻辑
- 当前鼓励高性能方案，但最终可行性仍需结合平台能力确认
