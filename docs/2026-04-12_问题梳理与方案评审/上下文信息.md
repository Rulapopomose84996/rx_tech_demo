# 与服务器交互的工作流设计

## 修改一下server via这个技能，修改方向：
1. 对于普通的项目开发流程（开发新功能），必须先检查分支情况（保留远程分支），询问是否开新分支，还是在某条分支上继续。用户选定分支以后，进入开发流程的第二步。
2. 第二步是拿到服务器的环境基线：你需要找到关于服务器的环境信息和构建说明，明确有些不能做的事情，比如没有 std::filesystem 依赖等问题，以免返工。这些信息大多都在 AGENTS.md 中，或者在项目的 README.md 中，又或者是docs下的环境/配置/说明等文档中。你需要从这些文档中提取相关信息，并在用户进入开发流程之前提供给他们，以确保他们了解项目的环境和构建要求。
    > 我在考虑要不要直接把项目基础环境放入技能中？ 这个怎么抉择？

3. 当你拿到环境基线之后，你就正式进入新建功能/修改代码/debug/重构/测试的开发流程了，原则是本地工作区修改代码，服务器上进行验证。具体实际操作方式可以分为以下几种：

    1. 直接在服务器上修改代码，然后进行验证
    2. 在本地修改代码后scp上传到服务器，然后验证。
    3. 在本地修改代码之后，使用git push到远程分支，然后在服务器上pull下来进行验证。（一般用于新建分支开发了完整的功能/或者进行了一次大重构，这个时候改动的文件比较多了，直接scp上传可能不太合适了）
    4. 在本地修改代码之后，在项目的同级别目录下新建一个完全隔离的项目进行验证（一般是大动的时候使用，改的文件很多，触及到项目根基。）
    5. 除此之外，没有想到其他同步方式了。
    > 有一点很重要，这些用法他们应对不同的场景，需要根据场景和需求不同，选择不同的用法。比较推荐1和2，3不推荐。

4. 本地开发依照项目里针对不同开发方向的工作流和SKILL即可。我这里不多做要求了。
5. 因为本地修改不满足服务器环境和配置的，需要记录到AGENTS.md中，或者项目文档中。并且在技能中也更新说明。
6. 项目运行在 devuser 下，权限问题需要注意。我已经为部分devuser 免密 sudo 配置了，后续如果需要增加权限的操作，可以直接申请，我会帮你配置的。
