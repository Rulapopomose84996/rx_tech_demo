
# Server Via - 本地开发与服务器验证协同技能

## 目的
本技能用于指导你在“本地开发 + 服务器验证”的模式下稳定工作。

你的目标不是把服务器当作主开发环境，而是：
- 在本地完成主要代码修改
- 在服务器上完成环境确认、构建验证、运行验证、问题复现
- 在开始开发前先确认分支状态与服务器约束
- 在发现新环境限制、新权限要求、新构建条件时及时沉淀到文档
- 时刻保持服务器工作区干净、可解释、可恢复

---

## 适用范围
以下任务都适用本技能：
- 新功能开发
- 缺陷修复
- 重构
- 调试
- 测试验证
- 环境相关兼容性排查

---

## 不可违背的总原则
1. **运行与部署兼容性，以服务器真实环境为准。**
2. **代码修改与提交，以本地仓库为准。**
3. **环境说明若与实际服务器冲突，以服务器实际结果为准，并回写文档。**
4. **任何服务器上的直接修改，验证结束后必须立即回流到本地工作区，并纳入版本控制；不允许服务器成为代码真源。**
5. **没有完成连接确认、分支确认、环境基线确认之前，不进入正式开发。**
6. **服务器优先用于验证，不作为默认主开发环境。**
7. **时刻保持服务器工作区干净整洁。**
8. **如果环境基线信息获取不足 80%，不得开始开发，必须直接向用户申请补充。**

## 默认连接方式
默认通过以下方式进入服务器：

```bash
ssh kds
```

---

## 第一阶段：连接检查与止损

### 1. 目标

先确认能否进入服务器。若不能进入，当前任务只做连接排障，不继续后续开发流程。

### 2. 若无法进入服务器，按以下顺序排查

1. 检查本地网络连接
2. 检查 SSH 配置文件
3. 检查 Host / User / Port / IdentityFile / ProxyJump 等配置
4. 记录并分析具体报错信息
5. 判断问题属于：

   * 网络不可达
   * DNS / Host 配置错误
   * SSH 配置错误
   * 密钥 / 认证失败
   * 权限不足
   * 目标机器异常

### 3. 止损规则

如果 `ssh kds` 无法进入服务器：

* 只做连接问题定位
* 输出明确报错与定位结论
* **不要继续执行分支检查、环境检查、同步、构建、验证、部署**

## 第二阶段：开发前必须检查分支状态

### 1. 进入开发前必须确认

至少检查以下内容：

* 当前本地分支
* 当前远程跟踪分支
* 本地是否存在未提交修改
* 本地是否领先/落后远程
* 远程是否存在目标分支
* 当前工作树是否干净

### 2. 你必须给用户明确的分支选择

在开始开发前，你必须给出用户以下选择，并标明你的推荐：

#### 选项 A：继续在当前分支上开发

适用场景：

* 小改动
* 当前任务是对现有工作连续推进
* 当前分支本来就是该任务分支

**推荐度：中**
当任务连续性强且改动规模小，可以推荐。

#### 选项 B：基于某个分支新建本地开发分支

适用场景：

* 新功能开发
* 风险较高的修复
* 中大型重构
* 需要独立验证与回滚

**推荐度：高**
这是默认优先推荐项，尤其适用于新功能、重构、实验性尝试。

#### 选项 C：切换到已有分支继续开发

适用场景：

* 已存在专门对应当前任务的分支
* 当前任务需要延续之前工作

**推荐度：中**
仅当已有分支职责明确时推荐。

### 3. 你的默认推荐逻辑

* **新功能 / 中大型修改 / 重构 / 不确定影响范围的修复：推荐新建分支**
* **小改动 / 延续当前任务：推荐继续当前分支**
* **已有明确任务分支：推荐切换到已有分支**

### 4. 禁止继续条件

出现以下情况时，不得进入正式开发：

* 当前分支用途不清晰
* 本地工作区有来源不明的脏改动
* 远程/本地关系不清楚
* 用户尚未决定用哪个分支继续

## 第三阶段：环境基线检查 —— 开发前必须先拿到“服务器约束”

### 1. 目标

在正式修改代码前，先确认服务器环境、构建要求、运行要求、权限边界和已知限制，避免本地开发后在服务器返工。

### 2. 信息来源优先级

按以下优先级收集环境基线：

1. 本技能中的服务器级公共基线
2. 项目内 `AGENTS.md`
3. 项目内 `README.md`
4. 项目内 `docs/` 下的环境说明、配置说明、部署说明、构建说明
5. 构建脚本、启动脚本、测试脚本
6. CI 配置、工具链配置文件
7. 服务器实际检查结果

### 3. 必须覆盖的信息项

你必须尽量拿到以下信息：

1. 服务器入口与访问方式
2. 运行用户与权限边界
3. 项目仓库路径 / 工作目录
4. 编译器或运行时版本
5. 构建命令
6. 运行命令 / 启动方式
7. 测试命令 / 验证方式
8. 关键依赖与依赖限制
9. 已知不兼容点 / 禁用特性
10. 配置文件位置 / 环境变量要求
11. 日志位置 / 排错入口
12. 服务管理方式（如有）
13. 临时文件 / 构建产物位置
14. 不能做的事情
15. 已知历史坑点

### 4. 80% 规则

只有当你拿到 **80% 以上的环境基线信息** 时，才允许开始开发。

执行标准：

* 以上 15 项里，至少明确 12 项，视为达标
* 或虽未达到 12 项，但已覆盖以下核心 8 项，也可视为达标：

  * 运行用户与权限边界
  * 项目仓库路径 / 工作目录
  * 编译器或运行时版本
  * 构建命令
  * 运行命令 / 启动方式
  * 测试命令 / 验证方式
  * 已知不兼容点 / 禁用特性
  * 日志位置 / 排错入口

### 5. 不足 80% 时的动作

如果环境基线获取不足 80%，你必须：

* 明确告诉用户当前缺失哪些关键项
* 说明这些缺失会带来什么风险
* **直接向用户申请补充**
* 在补充前，不得开始正式开发

### 6. 强制输出

在进入开发前，你必须向用户输出一份简明环境摘要，至少包含：

* 当前服务器约束
* 构建方式
* 运行方式
* 验证方式
* 已知限制
* 明确不能做的事情

## 第四阶段：选择同步与验证策略

### 总原则

默认采用：

* **本地修改**
* **服务器验证**

服务器不应成为默认主开发环境。

### 可选策略与推荐级别

#### 策略 A：本地修改后，用 `rsync` 同步到服务器验证

适用场景：

* 小到中等规模改动
* 多文件改动，但还不到需要专门走完整远程分支流程
* 需要快速反复验证

优点：

* 比 `scp` 更适合目录级同步
* 可只同步差异
* 可排除构建产物和无关文件
* 更稳、更快、更适合反复验证

**推荐度：最高**
这是默认优先推荐策略。

#### 策略 B：本地修改后，用 `scp` 上传少量文件到服务器验证

适用场景：

* 只改了极少数文件
* 需要快速替换单个或少量文件

优点：

* 简单直接

局限：

* 不适合多文件和目录级同步
* 不适合频繁增量同步

**推荐度：中**
少量文件时可用，但通常 `rsync` 更优。

#### 策略 C：本地修改后，推送到远程分支，服务器上 pull 后验证

适用场景：

* 已完成阶段性开发
* 改动文件较多
* 需要保留完整版本轨迹
* 需要让服务器工作区与仓库状态一致

优点：

* 版本轨迹清晰
* 更适合阶段性成果验证

局限：

* 不是最快的实验闭环
* 不适合高频试错的小步调试

**推荐度：中**
适用于阶段性成果，不作为默认快速验证手段。

#### 策略 D：在本地建立隔离副本 / 独立工作区进行大改

适用场景：

* 大重构
* 大范围试验
* 会触及项目基础结构
* 不希望污染当前主工作区

**推荐度：高**
适合高风险改动。

#### 策略 E：直接在服务器上修改并验证

适用场景：

* 很小的临时修复
* 强环境相关排查
* 快速定位问题时的极小范围修改

风险：

* 容易造成服务器工作区漂移
* 容易丢失修改
* 容易绕过本地版本管理

**推荐度：低**
仅在确有必要时使用，不应成为默认方式。

### 默认推荐顺序

1. `rsync`
2. 本地隔离副本 / 独立工作区
3. `git push + server pull`
4. `scp`
5. 服务器直接修改

> 注：当只涉及 1~2 个文件的极小改动时，`scp` 可以临时优先于 `git push/pull`。

## 第五阶段：服务器工作区卫生规则

项目由单人负责，不需要复杂协作管理，但必须保持基础秩序。

### 基本要求

1. **尽量不要在服务器工作区创建分支**
2. 服务器工作区应主要用于验证，而不是长期开发
3. 不保留来源不明的临时改动
4. 不保留长期未清理的临时文件、测试文件、构建垃圾
5. 不保留用途不明的未跟踪文件
6. 一旦某分支已确认合并完成，服务器上的该分支应删除
7. 时刻保持服务器工作区“干净、可解释、可恢复”

### 推荐实践

* 在服务器上尽量只检出用于当前验证的目标分支
* 验证完成后，及时清理临时产物
* 不在服务器工作区做长期实验
* 不把服务器工作区当作“临时代码仓库”

### 禁止行为

* 在服务器上长期堆积多个无用分支
* 在服务器上保留无版本管理承接的关键改动
* 在服务器上留下无法解释来源的差异
* 将服务器工作区作为代码真源

---

## 第六阶段：验证必须拆成四层

你不能只说“已经验证过”。必须明确验证完成了哪几层，缺了哪几层。

### 1. 构建验证

检查：

* 是否能成功编译/打包
* 是否有新的依赖问题
* 是否触发工具链兼容性问题

### 2. 启动验证

检查：

* 程序 / 服务是否能启动
* 启动过程中是否有权限、路径、配置问题
* 是否出现运行时动态链接或环境变量问题

### 3. 功能验证

检查：

* 当前目标功能是否按预期工作
* 当前缺陷是否已修复
* 修改目标是否达成

### 4. 回归验证

检查：

* 是否影响已有功能
* 是否破坏已有接口 / 配置 / 行为
* 是否引入明显副作用

### 输出要求

每次验证结束后，你必须明确说明：

* 已完成哪些验证层级
* 哪些层级尚未完成
* 当前结论属于“构建通过”还是“功能通过”还是“完整通过”

---

## 第七阶段：服务器实际环境优先级

以下规则必须严格遵守：

1. **运行与部署兼容性，以服务器真实环境为准**
2. **代码修改与提交，以本地仓库为准**
3. **环境说明若与实际服务器冲突，以服务器实际结果为准，并回写文档**
4. **若文档、脚本、服务器现状三者冲突，优先确认服务器真实结果，再决定文档修正**

---

## 第八阶段：服务器直接修改的回流规则

如果确实发生了服务器直接修改，必须满足以下要求：

1. 修改范围尽量小
2. 仅用于必要场景
3. 验证结束后，必须立即把修改回流到本地工作区
4. 回流后必须纳入版本控制
5. 必须说明服务器直接改动的原因
6. 回流完成前，不得把该服务器修改视为最终成果

**硬性规则：不允许服务器成为代码真源。**

---

## 第九阶段：权限与运行用户规则

### 1. 默认运行用户

项目运行在：

`devuser`

### 2. 你必须注意

* 构建、运行、验证、日志查看、文件写入是否都在正确用户下进行
* 不要默认认为当前 shell 用户就具备目标权限
* 不要默认认为所有目录都可写
* 不要默认认为所有系统操作都可执行

### 3. sudo 规则

* 已为部分 `devuser` 场景配置免密 sudo
* 需要 sudo 时先判断当前操作是否真的必要
* 若权限不足，必须明确记录：

  * 需要执行什么操作
  * 需要什么权限
  * 当前为何失败

### 4. 禁止默认假设

不得默认假设以下事情一定允许：

* 安装系统级依赖
* 修改系统配置
* 重启系统服务
* 写入系统目录
* 切换其他用户
* 修改系统守护进程或启动项

---

## 第十阶段：失败恢复策略

任何同步、验证、服务器直接修改动作，都必须考虑恢复路径。

### 1. 基本恢复原则

在做以下操作前，先确认存在恢复路径：

* 覆盖文件
* 批量同步
* pull / reset / clean
* 构建目录清理
* 服务器直接修改

### 2. 推荐的恢复前动作

在关键操作前，至少确认并记录：

* 当前分支
* `git status`
* 关键文件差异
* 当前服务器工作区状态
* 当前构建目录状态

### 3. 常见失败恢复规则

#### 文件被错误覆盖

处理原则：

* 优先从本地仓库或可追溯版本恢复
* 若是服务器验证文件被覆盖，尽快恢复到可解释状态

#### 服务器 pull 后冲突

处理原则：

* 服务器不是解决复杂冲突的首选地点
* 若冲突复杂，应回到本地解决
* 服务器工作区应恢复到清晰、可验证状态

#### 构建目录污染

处理原则：

* 优先清理构建目录后重新构建
* 必要时重新创建干净构建目录

#### 服务器直接修改后验证失败

处理原则：

* 保留必要日志
* 回滚临时修改或恢复干净状态
* 把有价值的排查结论带回本地

#### 临时文件和测试残留

处理原则：

* 验证后清理
* 不让临时文件长期驻留在服务器工作区

### 4. 硬性要求

验证失败后，服务器工作区也必须恢复到：

* 干净
* 可解释
* 可继续下一轮验证

## 第十一阶段：文档沉淀规则

当你发现以下信息时，必须更新文档：

* 本地与服务器环境不一致
* 服务器存在新的依赖限制
* 构建或运行存在新的隐含前提
* 权限边界与预期不同
* 某种同步策略在特定场景下容易失败
* 某个坑点具备重复出现的可能性

### 更新位置

优先更新以下位置：

1. 项目内 `AGENTS.md`
2. 项目内 `README.md`
3. 项目内 `docs/` 下相关文档
4. 如有必要，再更新本技能中的服务器级公共规则

### 原则

**同一个坑不要踩第二次。**

---

## 第十二阶段：你每次开始任务时的执行顺序

每次接到与服务器协同开发相关任务时，按以下顺序行动：

1. 确认是否能 `ssh kds`
2. 若失败，只做连接排障并停止
3. 检查本地分支状态与工作区状态
4. 给用户明确的分支选择，并指出推荐项
5. 收集环境基线
6. 判断环境基线是否达到 80%
7. 不足则向用户申请补充，并停止正式开发
8. 达标后，输出环境基线摘要
9. 根据改动规模选择同步/验证策略，并说明推荐理由
10. 进入开发与验证
11. 按四层验证法给出验证结论
12. 若发现新约束，更新文档
13. 清理服务器工作区，保持干净整洁

---

## 第十三阶段：输出风格要求

### 1. 在分支选择阶段，你必须这样输出

你必须告诉用户：

* 当前分支情况
* 可选方案
* 你的推荐方案
* 推荐理由

### 2. 在环境基线阶段，你必须这样输出

你必须告诉用户：

* 已确认的信息
* 尚缺失的信息
* 是否达到 80%
* 若未达到，明确申请补充

### 3. 在同步策略阶段，你必须这样输出

你必须告诉用户：

* 当前推荐使用哪种同步方式
* 为什么推荐这种方式
* 为什么不推荐其他方式

### 4. 在验证结论阶段，你必须这样输出

你必须明确区分：

* 构建通过
* 启动通过
* 功能通过
* 回归通过

不要模糊地说“验证通过”。

## 第十五阶段：服务器级公共基线与项目级基线获取规则

### 1. 服务器级公共基线

以下信息属于服务器级公共基线，可直接作为本技能的默认前提：

* 默认 SSH 入口：`ssh kds`
* 默认运行用户：`devuser`
* 服务器主要用于验证，不作为默认主开发环境
* 默认优先使用 `rsync` 进行本地到服务器的增量同步
* 尽量不要在服务器工作区创建分支
* 一旦某分支确认合并完成，服务器上的该分支应删除
* 服务器工作区必须长期保持干净整洁、可解释、可恢复
* 任何服务器上的直接修改，验证结束后必须立即回流到本地并纳入版本控制
* 环境基线获取不足 80% 时，不得开始正式开发，必须向用户申请补充

### 2. 项目级基线不在本技能中硬编码

本技能面向整台服务器上的多个项目，因此不在此处硬编码项目特定信息，例如：

* 仓库路径
* 构建目录
* 运行目录
* 日志目录
* 编译器版本
* 构建命令
* 启动命令
* 测试命令
* 项目独有依赖限制
* 项目独有历史坑点

这些信息必须在进入具体项目后动态获取。

### 3. 项目级基线获取顺序

进入具体项目后，必须按以下顺序获取项目级环境基线：

1. 项目内 `AGENTS.md`
2. `README.md`
3. `docs/` 下的环境、配置、部署、构建说明
4. 构建脚本、启动脚本、测试脚本
5. CI 配置、工具链配置文件
6. 服务器上的实际检查结果
7. 若仍不足 80%，直接向用户申请补充

### 4. 项目级基线的判断原则

项目级基线必须至少覆盖以下核心信息中的大多数：

* 项目仓库路径 / 工作目录
* 编译器或运行时版本
* 构建命令
* 启动方式
* 测试 / 验证方式
* 关键依赖限制
* 已知不兼容点
* 日志位置 / 排错入口

若未达到 80%，不得进入正式开发。

### 5. 冲突处理原则

若项目文档与服务器实际结果冲突：

* 以服务器实际结果为准
* 继续开发前先向用户说明冲突点
* 将正确结果回写到项目文档

## 最终执行准则

你必须始终记住：

* 先连通，再开发
* 先选分支，再动代码
* 先拿环境基线，再进入正式开发
* 环境信息不足 80%，就停止并申请补充
* 本地改代码，服务器做验证
* 默认优先 `rsync`
* 验证必须拆成四层
* 服务器真实环境高于文档描述
* 服务器直接修改必须回流本地
* 服务器工作区必须长期保持干净整洁
* 新坑必须沉淀到文档
