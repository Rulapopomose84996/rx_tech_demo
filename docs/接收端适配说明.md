# 接收端适配说明

> 这份文档只描述“当前 sender 实际发出来的是什么”和“接收端现在应该如何适配”。
> 它不替代接收端自身实现文档，也不把 sender 未来方案写成既成事实。

## 1. 当前 sender 事实

当前 sender 已确认的事实是：

- 发送端采用三口并发：
  - `sender0`
  - `sender1`
  - `sender2`
- 运行模型：
  - 单进程
  - 三线程
  - 每口一个完整发送上下文
- 每口单队列发送：
  - `queue-id=0`
- 当前默认 sender 候选实现：
  - `block-batch fast path`
- 当前 sender 默认口径：
  - 三条流的源端口都是 `9999`
  - 三条流的目标端口都是 `9999`

接收端需要适配的是这些 sender 已经成立的事实上线行为，而不是 sender 内部的优化细节。

## 2. 三条独立流

接收端应把三口流量视为三条独立 UDP 数据流：

| 发送口 | sender 源地址 | receiver 目标地址 | sender 源端口 | receiver 目标端口 |
|------|------|------|------|------|
| `sender0` | `172.20.11.11` | `172.20.11.100` | `9999` | `9999` |
| `sender1` | `172.20.12.12` | `172.20.12.100` | `9999` | `9999` |
| `sender2` | `172.20.13.13` | `172.20.13.100` | `9999` | `9999` |

当前建议：

- 一口一个 socket
- 一口一个接收线程或一口一个接收队列
- 不要先把三口流量混成一条再处理

## 3. 线上的包格式

每个 UDP 数据报当前格式是：

1. Ethernet
2. IPv4
3. UDP
4. `DemoHeader`
5. IQ payload fragment

`DemoHeader` 定义见 [`include/demo/protocol.h`](/home/klein/WorkSpace/sender/include/demo/protocol.h)。

接收端当前最应使用的字段是：

- `magic`
- `version`
- `flags`
- `stream_id`
- `block_id`
- `block_bytes`
- `frag_idx`
- `frag_count`
- `frag_payload_bytes`

## 4. 接收端应如何重组

接收端当前应按“每口独立重组”处理。

建议规则：

- 以 `(port, block_id)` 作为重组单元
- 用 `frag_idx` 决定 fragment 在 block 内的位置
- 用 `frag_count` 判断一个 block 是否完整
- 用 `frag_payload_bytes` 处理尾片的真实长度
- `flags` 中的 first/last 只作为辅助校验，不应替代 `frag_idx/frag_count`

也就是说：

- `sender0` 的 block 只在 `sender0` 自己的接收流里重组
- `sender1` 的 block 只在 `sender1` 自己的接收流里重组
- `sender2` 的 block 只在 `sender2` 自己的接收流里重组

## 5. 接收端不需要适配的 sender 细节

接收端当前不需要关心 sender 的这些内部实现细节：

- sender 是否用了 `block-batch fast path`
- sender 是否复用了 packet 缓冲区
- sender 内部如何批量 submit / poll
- sender 终端日志现在是否显示轻摘要还是详细块

这些都属于 sender 实现策略，不属于线上协议语义。

对接收端来说，真正稳定的事实只有：

- 三条独立 UDP 流
- 三条流都打到 `9999/udp`
- 相同的 `DemoHeader`
- 每口独立重组

## 6. 接收端最小统计项

接收端至少应按口分别统计：

- received packets
- received bytes
- reassembled blocks
- missing fragments
- duplicate fragments
- invalid header count
- reassembly timeout count
- per-port throughput

如果接收端后续还要做汇总，再在“每口统计完成后”做 aggregate。

## 7. 与 sender 当前终端摘要的关系

当前 sender 终端里会显示：

- 已发数据量（MiB）
- 接收数据量（MiB）：`待接收端反馈`
- 丢包率（%）：`待接收端反馈`
- 平均速率（Gbps）
- 状态

这意味着：

- sender 当前并没有真实拿到接收端统计
- sender 终端里出现 `待接收端反馈` 是正常行为
- 如果希望 sender 终端显示真实“接收数据量 / 丢包率”，必须由接收端提供回传或共享统计

## 8. 当前 sender 基线对接收端意味着什么

当前 sender 已有这些发送段事实：

- 单口 `sender0` root 真实 `5s` 样本：
  - `avg_gbps=6.17163`
  - `tx_errors=0`
  - `backpressure=0`
- 三口最稳样本：
  - `5s`: 每口约 `5.59 ~ 5.68 Gbps`
  - `60s`: 每口约 `5.34 ~ 5.48 Gbps`
  - `180s`: 每口约 `5.46 ~ 5.59 Gbps`

并且这些样本中：

- `tx_errors=0`
- `backpressure=0`

这说明：

- sender 当前重点不是“跑不起来”
- 接收端现在更应该优先保证自己能独立观测每口吞吐、丢片和重组结果
- 若联调时出现异常，应优先按“每口独立链路”排查，而不是先怀疑 sender 协议格式变化

## 9. 当前建议

当前阶段最推荐的接收端工作顺序是：

1. 按三口独立 socket 接入
2. 按 `DemoHeader` 做每口独立重组
3. 把每口统计做全
4. 与 sender 做长时间联调
5. 若后续需要，再把真实接收统计回传给 sender 摘要层
