# 接收端适配说明

> 这份文档只说明“当前 sender 实际发送出来的是什么”和“接收端需要如何适配接收”。
> 它不替代接收端自身实现文档，也不把未验证的 sender 未来方案写成既成事实。

## 1. 当前 sender 事实

当前 sender 已经稳定下来的事实是：

- 发送端采用三口并发：
  - `sender0`
  - `sender1`
  - `sender2`
- 运行模型是：
  - 单进程
  - 三线程
  - 每口一个完整发送上下文
- 每个口当前仍使用单队列发送：
  - `queue-id=0`
- 当前候选实现是 sender 侧的 `block-batch fast path`
- 当前 sender 已在真实链路上拿到稳定样本，具备 `5.5+ Gbps / port` 的发送能力

接收端需要适配的，是这个 sender 实际发出来的数据，而不是 sender 内部是如何实现 batch 的。

## 2. 三条独立流

接收端应把三口流量视为三条独立 UDP 数据流：

| 发送口 | sender 源地址 | receiver 目标地址 | sender 源端口 | receiver 目标端口 |
|------|------|------|------|------|
| `sender0` | `172.20.11.11` | `172.20.11.100` | `4010` | `5010` |
| `sender1` | `172.20.12.12` | `172.20.12.100` | `4011` | `5011` |
| `sender2` | `172.20.13.13` | `172.20.13.100` | `4012` | `5012` |

当前建议：

- 一口一个 socket
- 一口一个接收线程或一口一个接收队列
- 不要先把三口流量混成一条再处理

## 3. 线上的包格式

每个 UDP 数据报的格式是：

1. Ethernet
2. IPv4
3. UDP
4. `DemoHeader`
5. IQ payload fragment

其中 `DemoHeader` 定义见 [`include/demo/protocol.h`](/home/klein/WorkSpace/sender/include/demo/protocol.h)。

当前接收端最需要使用的字段是：

- `magic`
- `version`
- `flags`
- `stream_id`
- `block_id`
- `block_bytes`
- `frag_idx`
- `frag_count`
- `frag_payload_bytes`

## 4. 接收端应如何重组

接收端应按“每口独立重组”处理。

建议规则：

- 以 `(port, block_id)` 作为重组单元
- 使用 `frag_idx` 决定 fragment 在 block 内的位置
- 使用 `frag_count` 判断一个 block 是否完整
- 使用 `frag_payload_bytes` 处理尾片的真实长度
- `flags` 中的 first/last 只作为辅助校验，不应替代 `frag_idx/frag_count`

也就是说：

- `sender0` 的 block 只在 `sender0` 自己的接收流里重组
- `sender1` 的 block 只在 `sender1` 自己的接收流里重组
- `sender2` 的 block 只在 `sender2` 自己的接收流里重组

## 5. 不需要适配的 sender 内部细节

接收端不需要关心 sender 的这些内部实现细节：

- sender 是否用了 `block-batch fast path`
- sender 是否在内部复用了 packet 缓冲区
- sender 内部如何决定一次 batch 包含几个 block
- sender 内部何时 poll / wakeup

这些都属于 sender 的实现策略，不属于线上协议语义。

对接收端来说，真正稳定的事实只有：

- 三条独立 UDP 流
- 相同的 `DemoHeader` 协议
- 每口独立重组

## 6. 接收端最小统计项

接收端至少应按口分别统计：

- received packets
- received bytes
- reassembled blocks
- missing fragments
- duplicate fragments
- invalid header count
- reassembly timeout count
- per-port throughput

如果接收端后续还要汇总全局指标，再在“每口统计完成之后”做 aggregate。

## 7. 当前最稳的适配边界

当前推荐的接收端适配边界是：

- 先把三口都稳定接住
- 先把每口重组做对
- 先把每口统计做全
- 不要一上来做跨口合流
- 不要一上来做复杂全局排序

只有当业务层明确要求“三口最终汇成一个上层数据流”时，才建议在“每口重组完成之后”再做上层汇聚。

## 8. 当前 sender 基线对接收端意味着什么

当前 sender 候选实现已经拿到这些发送段样本：

- `5s` 样本：每口约 `5.59 ~ 5.68 Gbps`
- `60s` 样本：每口约 `5.34 ~ 5.48 Gbps`
- `180s` 样本：每口约 `5.46 ~ 5.59 Gbps`

并且样本中：

- `tx_errors=0`
- `backpressure=0`

这意味着：

- 接收端现在不需要优先去解释 sender 侧错误
- 接收端更需要保证自己能独立观测三口吞吐、丢片和重组结果
- 若联调时出现掉包或重组异常，应优先按“每口独立链路”排查，而不是先怀疑 sender 协议格式变化

## 9. 当前建议

当前阶段最推荐的接收端工作顺序是：

1. 按三口独立 socket 接入
2. 按 `DemoHeader` 做每口独立重组
3. 把每口统计做全
4. 再做 sender/receiver 联调
5. 若业务需要，再讨论三口上层汇聚
