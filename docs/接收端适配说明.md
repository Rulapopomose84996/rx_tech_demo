# 接收端适配说明

> 本仓库当前接收主线已收敛为 **AF_XDP**。

## 1. 当前发送端事实

发送端仍是三口并发、每口单队列、统一 `DemoHeader` 协议。接收端需要适配的是 sender 实际发出的以太网/IPv4/UDP/`DemoHeader`/IQ fragment 数据，而不是 sender 内部 batch 细节。

## 2. 当前接收主线

当前接收实现的唯一主线是：

- `AF_XDP backend`
- `rxbench_xdp` 主入口
- 前台长时间手动停止模式
- `DemoHeader` 解析 + 每口独立重组 + per-port 指标

当前不再维护 `socket` 作为联调主线。

## 3. 目标链路

| 阵面 | sender 源地址 | receiver 目标地址 | sender 源端口 | receiver 目标端口 |
|------|------|------|------|------|
| `sender0` | `172.20.11.11` | `172.20.11.100` | `4010` | `9999` |
| `sender1` | `172.20.12.12` | `172.20.12.100` | `4011` | `9999` |
| `sender2` | `172.20.13.13` | `172.20.13.100` | `4012` | `9999` |

## 4. 协议与重组语义

- 每个 UDP 报文都带 `DemoHeader`
- 关键字段：`magic/version/flags/stream_id/block_id/block_bytes/frag_idx/frag_count/frag_payload_bytes`
- 重组规则：按 `(port, block_id)` 独立重组
- `flags.first/last` 只作辅助校验，不替代 `frag_idx/frag_count`

## 5. AF_XDP 运行约束

- 当前平台目标模式：`driver + copy`
- 实际绑定 queue 必须与 sender 流量 RSS 落点一致
- 前台运行，不使用 `tmux` 作为默认联调方式
- 终端每 10 秒打印一次状态快照

## 6. 最小验证目标

1. `rxbench_xdp` 能在目标口成功 attach / bind
2. 能收到真实 sender UDP 流量
3. 能完成 `DemoHeader` 解析与 block 重组
4. 能输出每口吞吐、重组、缺片、重片、非法头与超时统计
