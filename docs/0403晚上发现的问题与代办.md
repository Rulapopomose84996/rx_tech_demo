# 2024-04-03晚上发现的问题与待办

## 🔴 关键设计决策

### 1. 调试/统计功能与核心业务分离

**问题背景**：
- 状态报告、日志输出等辅助任务会抢占 CPU 资源
- 在 5Gbps 高负载场景下，即使是纳秒级的开销也可能导致丢包
- 生产环境应该只保留核心业务逻辑，禁止任何调试代码参与资源竞争

**设计方案**：
```cpp
// 新增编译时配置选项
#ifdef RXTECH_DEBUG_MODE
    // 调试模式：启用所有统计、日志、诊断输出
    #define ENABLE_STATUS_PANEL      1
    #define ENABLE_DIAGNOSTIC_DUMP   1
    #define ENABLE_METRICS_DETAIL    1
    #define ENABLE_RAW_FRAME_RECORD  1
#else
    // 生产模式：完全禁用调试功能
    #define ENABLE_STATUS_PANEL      0
    #define ENABLE_DIAGNOSTIC_DUMP   0
    #define ENABLE_METRICS_DETAIL    0
    #define ENABLE_RAW_FRAME_RECORD  0
#endif
```

**影响范围**：
1. `runtime_status_reporter.cpp` - 状态面板输出
2. `metrics.cpp` - 详细指标统计
3. `raw_frame_recorder.cpp` - 原始帧录制
4. `packet_pipeline.cpp` - 错误包诊断打印（`invalid_dumped < 5U`）
5. `owner_loop.cpp` - 主循环中的同步调用

**实施步骤**：
- [ ] 在 CMake 中添加 `RXTECH_DEBUG_MODE` 编译选项
- [ ] 使用条件编译包裹所有调试相关代码
- [ ] 配置文件增加 `debug_mode = true/false` 参数
- [ ] 生产环境构建时禁用调试模式（`-DRXTECH_DEBUG_MODE=OFF`）

**性能收益预期**：
- 减少 CPU 缓存污染（调试代码不占用 cache line）
- 消除状态输出的系统调用开销（`std::cout` / `fprintf`）
- 降低内存带宽占用（不写入调试缓冲区）

#### 1. 调整 DPDK 内存池大小（巨型帧场景）

**计算依据**：
- 单包大小：8000 字节（巨型帧，9000B MTU）
- 单 CPI 大小：23MB
- 单 CPI 包数：23MB / 8000B ≈ 3,014 包
- 5Gbps 线速：625MB/s → 78,125 包/秒
- 处理窗口：36.8ms（单 CPI 持续时间）

[protocol]
udp_packet_size = 8000      # 从 2048 改为 8000（巨型帧）

#### 6. CPI 跳变双缓冲设计

**你的担忧**：延迟判决不利于实时性和高性能

**结论**：双缓冲更适合高性能场景

**实施方案**：
```cpp
struct CpiStateCoordinator {
    CpiContext* buffers[2];  // 奇偶双缓冲

    CpiProcessResult process_data_packet(...)
    {
        uint16_t buf_index = packet.cpi % 2;  // 奇偶判断
        CpiContext* ctx = buffers[buf_index];

        if (ctx == nullptr || ctx->cpi_id != packet.cpi) {
            // 新 CPI，初始化缓冲
            ctx = allocate_context(packet.cpi);
            buffers[buf_index] = ctx;
        }

        write_to_context(ctx, packet);

        // 检查是否完成
        if (is_complete(ctx)) {
            finalize_and_output(ctx);
            buffers[buf_index] = nullptr;  // 释放
        }
    }
};
```

**极端场景处理**：CPI 从 1 跳到 3
```cpp
// 检测跳变
if (abs(new_cpi - old_cpi) > 1) {
    finalize(buffers[old_cpi % 2]);  // 强制清空旧的
}
```

**实施建议**：
- [ ] 采用双缓冲设计（奇偶索引）
- [ ] 添加跳变检测（处理 CPI 1→3 跳跃）
- [ ] 性能对比（双缓冲 vs 延迟判决）

### 技术债务

#### 8. CPI 数据处理零拷贝验证

**待确认问题**：
- [ ] 验证 Consumer 线程拿到的是指针而非深拷贝
- [ ] 确认 mbuf 和 CPI Context 的生命周期管理
- [ ] 验证 recycle_ring 同步机制是否正确

**正确流程**：
```cpp
// Producer 推送
output_ring_->push(output);  // 传指针

// Consumer 处理
consumer_thread() {
    pop(out);
    write_to_file(out.data_ptr, out.size);  // 正在写入
    release_token.token_id = out.output_id;
    recycle_ring_->push(release_token);     // 通知可释放
}

// Producer 回收
drain_recycle() {
    ReleaseToken token;
    while (recycle_ring_->pop(token)) {
        ctx_pool_.release(token.ctx_index);  // 这时才真正释放
    }
}
```

**关键点**：
- mbuf 在 `release_burst()` 时立即归还到内存池
- CPI Context 要等 Consumer 处理完才能释放（通过 recycle_ring 同步）
