# 前端感知通信协议 V1.0

---

## 0 协议设计思想与宏观架构

### 0.1 系统角色

本协议服务于**非对称主从架构**：

- **SPS（Signal Processing System）**：信号处理系统，唯一的控制发起方（上位机/服务器）
- **DACS（Data Acquisition & Control System）**：数据采集与控制系统，前端 FPGA 设备，仅响应控制并主动推送数据

物理链路为 **10GbE 以太网（UDP/IPv4）**，每个 DACS 阵面通过独立万兆光口直连 SPS。

### 0.2 五平面分离与本文范围

协议采用平面分离设计，每个平面独立承载不同语义：

| 平面 | 报文类型 | 方向 | 核心语义 | 可靠性策略 |
|------|---------|------|---------|-----------|
| **控制平面** | `0x01` 控制指令 + `0x02` 确认应答 | SPS → DACS | 波束安排周期下发（一个周期内所有波束参数集合） | CRC32C + 确认应答闭环 + 超时重传 |
| **数据平面** | `0x03` 回波数据 | DACS → SPS | 原始 I/Q 回波上传 | 以太网 FCS + 丢包补零（冻结为“按分片补零”，见 4.1） |
| **遥测平面** | `0x04` 心跳状态 | DACS → SPS | 健康状态上报 | 尽力传输，允许丢包 |
| **时基边界平面** | `0x05` PPS Notify | DACS → SPS | 软件秒边界事件上送 | 尽力传输 + 去重/乱序检测 + 失边界降级 |
| **维护平面** | `0xFF` 远程维护 | SPS ↔ DACS | 固件升级/远程维护 | CRC32C + 会话令牌事务闭环 |

本文档专注定义 **控制平面 / 数据平面 / 遥测平面** 的公共字段与载荷，并纳入 **时基边界平面（`PacketType=0x05`）** 的通用入口约束；`0x05` 的独立载荷定义见 [时基边界事件协议（PPS Notify）V1.0.md](./时基边界事件协议（PPS Notify）V1.0.md)，维护平面（`PacketType=0xFF`）见 [RMA维护通信协议v1.0.md](./RMA维护通信协议v1.0.md)。

### 0.2.1 控制闭环重传与幂等（冻结）

为确保控制平面在 UDP 之上具备可预期的工程行为，控制闭环的**超时、重传上限、退避、重复命令幂等**统一冻结如下。

**重传参数（冻结）**：

- `RTO_MS`：首次重传超时（从 SPS 首次发送该 Control 起计）
- `MAX_RETRY`：最大重传次数（不含首次发送）
- `BACKOFF`：指数退避：第 `k` 次重传等待时间 = `RTO_MS × 2^k`（`k` 从 0 开始）

**默认值（冻结）**：

- `RTO_MS = 2500 ms`
- `MAX_RETRY = 3`

**重传报文一致性（冻结）**：

- SPS 对同一条 Control 的重传必须保持 **Common Header 与 Payload 完全一致**，尤其：
   - Common Header.`SequenceNumber` **必须不变**（重传不分配新序号）
   - Control Payload 必须完全一致（含 CRC32C）

**DupCmd 幂等（冻结）**：

- DACS 必须对重复 Control 做幂等去重：对“同键重复”的 Control **不得重复执行**，只允许**复用/回放**之前计算出的执行结果。
- **Control 幂等键（冻结）**：`(SourceID, ControlEpoch, SequenceNumber)`。
- 同键重复的 Control：
   - 若首次已被接收并完成参数/CRC 校验：不得重复改写排程，只需保证最终 `ScheduleApplyAck` 的结果与首次一致。
   - 若首次尚未完成校验：实现可丢弃重复包或以“最后一次到达”为准，但不得导致同键出现不同的周期执行结果。

### 0.3 核心设计原则

1. **全小端序编码**：所有多字节字段采用小端序（Little-Endian），包括整数、浮点数、时间戳等
2. **整数量化物理量**：协议层禁止浮点数，所有物理量使用整数编码并定义量化单位（见"物理量量化表"）
3. **统一32字节通用报文头**：所有报文类型共享同一通用报文头，通过"报文类型"字段分发到不同处理流程
4. **波束安排表下发**：控制平面以"波束安排表"为单位下发，每个控制指令包含一秒周期内的所有波束配置
5. **任务周期应答**：确认应答以"逻辑周期号（FrameCounter）"为单位反馈执行结果，而非逐个波束应答
6. **保留字段策略（MBZ 放宽）**：发送端应填 0；接收端在**主版本一致**时应允许保留字段非 0（视为小扩展/兼容扩展），按计数器记录并告警，但不应仅因保留字段非 0 直接丢包；如需上报可使用 `ScheduleApplyAck.ErrorCode=0x28`，且允许 `AckResult=OK`
7. **校验策略**：移除通用头部 CRC16；控制/应答/心跳/时基边界等带 CRC32C 的载荷仍以 CRC32C 作为主要完整性校验
8. **任务标识跨平面绑定**：控制指令下发的"逻辑周期号（FrameCounter）"与回波数据上报的"周期计数"形成任务上下文关联
9. **四字节自然对齐**：所有数据结构遵循4字节自然对齐，优化DMA传输性能
10. **时间戳统一编码**：全协议仅保留一种时间戳：`uint64` Unix Epoch 毫秒（ms），小端序传输

**量化关系通用公式**：

```txt
实际物理量 = 编码值 × 量化单位
编码值 = round(实际物理量 / 量化单位)
```

**示例**：
- 天线方位指向 = 10000 × 0.0025° = 25°
- 手动增益控制 = 40 × 0.5 dB = 20 dB
- 核心温度 = 725 × 0.1 ℃ = 72.5 ℃

### 0.5 波束安排周期与秒边界语义

控制平面采用**波束安排周期**作为传输单位，通常为 1 秒。每个周期内，SPS 下发该周期内的所有波束参数集合给 DACS，而非实时下发单个波束。为实现多阵面的射频参数同步切换，秒边界语义冻结为“**前端执行侧物理秒边界 + 服务器侧软件秒边界事件**”双层模型：

- **前端物理秒边界**：各 DACS 在前端执行侧感知 UTC 秒边界（由 PTP/GNSS 授时）并进行波束切换，精度目标为 `±T_APPLY_WINDOW_MS`（默认值见 1.3.2）
- **服务器侧软件秒边界事件**：SPS 在没有硬件秒边界输入条件下，通过 `0x05 PPS Notify` 接收前端边界事实，并由时基管理组件维护权威周期状态
- **CPI序列号同步**：通过控制指令中的"CPI计数基数"字段实现应用层同步，同一CPI计数基数对应的所有波束应在同一CPI周期内调度执行

### 0.5.1 索引体系与职责分层（冻结）

为避免实现侧混淆，跨平面索引体系冻结如下：

| 字段 | 所属平面 | 角色定位 | 说明 |
|------|---------|---------|------|
| `PpsSeq` | `0x05 PPS Notify` | 软件秒边界序号 | 表示服务器侧消费的边界事件序号，用于去重、乱序检测、失边界统计 |
| `FrameCounter` | `0x01/0x02/0x03` | 逻辑周期号 | 外部字段名保留 `FrameCounter`，但在 V1.0 中统一解释为逻辑周期号，不仅限于“搜索帧” |
| `Timestamp` | 通用头 | 时间边界/发送时间语义 | 统一编码、按 PacketType 分语义解释，不承担唯一周期号职责 |
| `CpiCountBase` | `0x01 Control` | 秒内 CPI 推进基数 | 描述当前逻辑周期内 CPI 推进起点，不等价于周期号 |
| `BeamID` | `0x01/0x03` | 秒内波束序号 | 标识同一逻辑周期内的波束对象 |
| `CpiCount` | `0x03 Data` | 波束级 CPI 标识 | 由 `CpiCountBase + BeamID` 推导，用于数据与控制的波束级绑定 |

### 0.6 数据流向与控制命令流

明确定义控制命令流的方向：**信号处理系统（SPS）→ 数据采集控制系统（DACS）**。SPS作为控制发起方，下发控制指令至DACS；DACS接收并执行配置，通过确认应答反馈执行结果。与此同时，数据平面流向相反：DACS采集数据后通过回波数据上传至SPS，保证控制与数据的分层隔离与流向明确。

### 0.8 UDP 端口分配

| 报文类型 | 方向 | 源端口 | 目标端口 |
|---------|------|-------|---------|
| 控制指令 (0x01) | SPS → DACS | 随机(>1024) | 8888 |
| 确认应答 (0x02) | DACS → SPS | 8888 | 回复至控制指令源端口 |
| 回波数据 (0x03) | DACS → SPS | 9999 | 9999 |
| 心跳状态 (0x04) | DACS → SPS | 9999 | 9999 |
| PPS Notify (0x05) | DACS → SPS | 8890 | 8890 |

#### 0.8.1 同端口承载 Data/Heartbeat 的 QoS（冻结）

当 `0x03(Data)` 与 `0x04(Heartbeat)` 复用 UDP/9999 时，为保证拥塞/洪峰下的可用性与可观测性，QoS 冻结如下：

1. **DSCP 标记（冻结）**
   - Heartbeat：发送端必须设置 `DSCP = CS6`。
   - Data：发送端必须设置 `DSCP = CS0`。
   - 若网络侧清除/重写 DSCP，则仍必须满足第 2/3 条的软件侧保底机制。
2. **发送端保活带宽预留（冻结）**
   - 发送端必须对 Heartbeat 采用令牌桶/节流策略，确保在 Data 洪峰下仍能按配置心跳周期发送。
   - 默认心跳频率仍为 1 Hz（见 5 章），但该 QoS 机制不依赖固定频率。
3. **接收端优先队列（冻结）**
   - 接收端在同一 socket 上按 `PacketType` 分类：Heartbeat 进入高优先级队列并优先处理；Data 进入普通队列。

### 0.9 协议版本

本手册对应 **协议版本 = `0x10`**（V1.0）。协议版本编码为 `uint8`，高4位=主版本，低4位=次版本。

## 1 通用报文头 — 全部报文类型共用

### 1.1 包结构

```
┌──────────────────────────────────────────┐
│  通用报文头 (固定 32 字节)                 │
├──────────────────────────────────────────┤
│  后续载荷 (按报文类型不同而不同)            │
└──────────────────────────────────────────┘
```

所有UDP报文的前32字节均为通用报文头。"载荷长度"字段描述紧随其后的载荷字节数，即 `UDP载荷总长 = 32 + 载荷长度`。

### 1.2 字段表

| 偏移 | 字段名称 | 类型 | 字节数 | 说明 |
|------|---------|------|-------|------|
| **0** | **魔数（Magic）** | `uint32` | 4 | **含义**：帧首同步字（固定值）。<br/>**取值**：`0x55AA55AA`（固定）。<br/>**设计目的**：快速定位帧首；`0x55AA`交替位模式利于硬件对齐。<br/>**表示**：`uint32`，小端序（线上字节：`[AA 55 AA 55]`）。<br/>**处理**：不匹配→静默丢弃（计数见失败处理表）。 |
| **4** | **序列号（SequenceNumber）** | `uint32` | 4 | **含义**：逻辑流内递增序列号。<br/>**作用**：去重/幂等、丢包检测、（控制平面）重传去重键组成部分。<br/>**Ack 配对说明（冻结）**：`PacketType=0x02` 的配对键见 3 章 `AckedSequenceNumber`，Common Header.`SequenceNumber` 不再用于 Ack 配对。<br/>**递增作用域**：按`(源标识, 报文类型, 方向)`三元组划分逻辑流（同一逻辑流内对“新报文”递增；重传报文保持序号不变）。<br/>**回绕比较**：必须使用半区间算法（Δ=无符号减法；`0<Δ<2^31`为新包；`Δ==0`重复；`Δ≥2^31`旧包）。 |
| **8** | **时间戳（Timestamp）** | `uint64` | 8 | **含义**：统一时间戳（Unix Epoch，毫秒）。<br/>**按报文类型解释**：<br/>• `0x01(控制指令)`：生效时刻（期望生效的 UTC 时刻）<br/>• `0x02(确认应答)`：应答生成时刻<br/>• `0x03(回波数据)`：采样起始时刻（sample0 时刻；建议与 Specific Header 的数据时间戳一致）<br/>• `0x04(心跳状态)`：发送时刻<br/>• `0x05(PPS Notify)`：报文发送时刻（边界事实本身见 PPS Payload）<br/>**取值**：`ms_since_epoch`，即自 1970-01-01 00:00:00 UTC 起的毫秒数。<br/>**取值零处理**：允许为 `0`（表示未锁定授时/未实现时统/立即生效语义由报文类型决定）。 |
| **16** | **载荷长度（PayloadLen）** | `uint16` | 2 | **含义**：后续载荷字节长度（不含32字节报文头）。<br/>**约束**：`载荷长度 ≤ 65475`（`65507-32`）。<br/>**一致性检查**：必须满足`UDP载荷字节数 == 32 + 载荷长度`，否则静默丢弃。<br/>**安全约束**：必须在 Magic/ProtoVer/PacketType/长度一致性等基础检查通过后才允许按该长度访问载荷（防越界）。 |
| **18** | **报文类型（PacketType）** | `uint8` | 1 | **含义**：载荷类型/解析分发键。<br/>**取值**：<br/>• `0x01` 控制指令<br/>• `0x02` 确认应答<br/>• `0x03` 回波数据<br/>• `0x04` 心跳状态<br/>• `0x05` PPS Notify（独立载荷定义见 PPS 协议文档）<br/>**处理**：未知类型→静默丢弃并计数。 |
| **19** | **协议版本（ProtocolVersion）** | `uint8` | 1 | **含义**：协议版本（高4位主版本，低4位次版本）。<br/>**基线**：V1.0 = `0x10`。<br/>**处理**：主版本不匹配→静默丢弃（禁止猜测解析）。<br/>**兼容**：当`协议版本 > 0x10`时，保留字段非零可能表示扩展（需结合版本策略）。 |
| **20** | **源标识（SourceID）** | `uint8` | 1 | **含义**：发送端逻辑标识。<br/>**取值**：<br/>• `0x01` = SPS<br/>• `0x11-0x13` = DACS阵面<br/>• `0x10` 禁止作为源标识（仅用于目标标识广播）<br/>• `0xFF` 禁止使用<br/>**设计目的**：逻辑寻址与去重键组成部分（与IP解耦）。 |
| **21** | **目标标识（DestID）** | `uint8` | 1 | **含义**：接收端逻辑标识。<br/>**取值**：<br/>• `0x10` = 逻辑广播（所有DACS）<br/>• 其他取值同源标识（点对点）<br/>• `0xFF` 禁止使用<br/>**处理**：不匹配→静默丢弃（建议NIC/FPGA层早过滤）。 |
| **22** | **控制纪元（ControlEpoch / LinkEpoch）** | `uint16` | 2 | **含义（冻结）**：控制/链路纪元标识（SPS 启动/重启/链路重建递增；原“SessionCode”术语废止）。<br/>**作用（冻结）**：会话隔离；用于控制幂等去重键边界。<br/>**禁止互用声明（冻结）**：本字段与维护会话标识无关且禁止互用（维护面详见独立 RMA 协议文档）。<br/>**处理（冻结）**：检测到 ControlEpoch 变化→清空旧纪元去重缓存，并将“当前 ControlEpoch”更新为新值；对**旧 ControlEpoch** 的 Control（即 `ControlEpoch != 当前 ControlEpoch`）必须**拒绝执行**，并回 `ScheduleApplyAck`：`AckResult=NOT_APPLIED, ErrorCode=0x03, ErrorDetail=0xFFFF`（表示纪元不一致）。<br/>**取值**：`uint16`自然回绕。 |
| **24** | **扩展标志（ExtFlags）** | `uint8` | 1 | **含义**：通用头扩展标志（保留）。<br/>**基线要求**：V1.0（0x10）发送端应填 `0x00`。<br/>**兼容策略**：接收端在主版本一致时，若该字段非 0，应继续解析已知字段并计数告警（视为小扩展）；主版本不一致则丢弃。 |
| **25** | **保留1（Reserved1）** | `uint8` | 1 | **含义**：对齐保留（Reserved）。<br/>**发送端**：应填 `0x00`。<br/>**接收端**：主版本一致时允许非 0（计数告警，继续解析）。 |
| **26** | **保留2（Reserved2）** | `uint16` | 2 | **含义**：保留（Reserved）。用于未来扩展（例如：头部特性协商/能力标识）。<br/>**发送端**：应填 `0x0000`。<br/>**接收端**：主版本一致时允许非 0（计数告警，继续解析）。 |
| **28** | **保留3（Reserved3）** | `uint8[4]` | 4 | **含义**：对齐与扩展保留（Reserved）。<br/>**发送端**：应填全 `0x00`。<br/>**接收端**：主版本一致时允许非 0（计数告警，继续解析）。 |

**总长度 = 32 字节**

### 1.2.1 Timestamp 语义矩阵（冻结）

> 说明：通用头 `Timestamp` 只有一种编码（Unix Epoch ms），但**不同 PacketType 语义不同**。实现端必须按下表解释。

| PacketType | 报文名称 | Common Header.Timestamp 语义 | 允许为 0 | 0 的语义 |
|---|---|---|---|---|
| `0x01` | Control | **期望生效时刻**（UTC 秒边界对齐；见 1.3.3） | 允许 | `0` 表示“立即生效” |
| `0x02` | ScheduleApplyAck | **应答生成时间** | 允许 | `0` 表示“时间不可用/未锁定/未实现”，但仍可回包 |
| `0x03` | Data | **采样起始时刻**（sample0；与 `DataTimestamp` 一致性见 4.2） | 允许 | `0` 表示“时间不可用/未锁定/未实现” |
| `0x04` | Heartbeat | **发送时刻** | 允许 | `0` 表示“时间不可用/未锁定/未实现” |
| `0x05` | PPS Notify | **报文发送时刻** | 允许 | `0` 表示“时间不可用/未锁定/未实现” |

### 1.2.2 CRC 策略矩阵（冻结）

> 说明：V1.0 **移除通用头 CRC16**，但部分 PacketType 的载荷内含 CRC32C。为避免“覆盖范围/是否回包”误解，统一如下。

| PacketType | 报文名称 | 是否含 CRC32C 字段 | CRC32C 位置 | CRC32C 覆盖范围（相对本报文载荷起始 Offset） | 校验失败处理 |
|---|---|---|---|---|---|
| `0x01` | Control | 是 | 载荷末尾 4B | `0 .. (12 + N×32 - 1)`（表头+波束项，不含 CRC 字段本身） | **不得执行**；**必须回** `0x02`：`NOT_APPLIED + ErrorCode=0x01` |
| `0x02` | ScheduleApplyAck | 是 | Payload Offset `20` | `0 .. 19`（不含 CRC 字段与尾部填充） | SPS 端丢弃该 Ack（是否重传由实现决定） |
| `0x03` | Data | 否 | — | — | 依赖以太网 FCS；丢包/坏包由接收端重组超时处理 |
| `0x04` | Heartbeat | 是 | Payload Offset `44` | `0 .. 43` | 失败→静默丢弃（不得回包） |
| `0x05` | PPS Notify | 是 | Payload Offset `36` | `0 .. 35` | 失败→静默丢弃（不得回包） |

## 1.3 时间同步专题：DACS 与 SPS（北斗/GNSS → PTP → 本地晶振）

> **目标**：把“自动降级/恢复”做成**可判定、可观测、可回归**的机制，使 SPS 与 DACS 在联调与长期运行中能明确回答：
> 1) 当前系统时间来自哪里？（北斗/GNSS / PTP / 本地晶振）
> 2) 当前时间质量处于哪一档？（锁定/保持/自由运行）
> 3) 何时、因何触发降级/恢复？以及是否影响“定时生效/时间戳”语义。

### 1.3.1 目的与边界

**必须保证**：

1. **控制定时生效的一致性**：当 DACS 处于“可授时”状态时，支持按 Common Header.Timestamp 在秒边界附近生效（见 Control 的“定时生效控制”）。
2. **数据时间戳的可解释性**：Data/Heartbeat/CommonHeader.Timestamp 必须能解释为“该报文使用的时间坐标系下的 Unix Epoch ms”。
3. **软件秒边界语义一致性**：当服务器侧没有硬件秒边界输入时，SPS 必须通过 `0x05 PPS Notify` 建立“软件秒边界事件 -> 时基管理组件 -> 权威周期状态”的统一链路。
4. **可诊断性**：SPS 必须能通过心跳 / PPS Notify / 回执，判断 DACS 当前授时来源与质量，以及是否发生过降级/恢复。

**不保证**：

1. 在“本地晶振”档位下，不保证与 SPS 的绝对时间一致性；仅保证时间单调递增（尽力）与误差可观测。
2. 不保证跨重启的时间连续性；重启后可能回到 `Timestamp=0` 或重新锁定过程。
3. 不保证 GNSS 受遮挡/干扰场景下仍能维持锁定；该场景应触发自动降级。

### 1.3.2 时间来源与质量分级

**时间来源（Time Coordinate Source）三档**：

- **Level-1：北斗/GNSS**（优先）
- **Level-2：PTP**（降级，网络授时；通常由 SPS/交换机提供授时体系）
- **Level-3：本地晶振**（再次降级，自由运行/保持）

**时间质量（Time Quality）三档**（与来源正交，来源不等于质量）：

- **Q0：Locked**：已锁定并参与同步闭环（可按秒边界定时生效）
- **Q1：Holdover**：外部授时短暂丢失但处于保持（允许继续运行，但需标记降级；**V1.0 默认禁止定时生效**，见 1.3.5）
- **Q2：Freerun/Unknown**：自由运行或未知（必须限制/禁止定时生效；时间戳仅用于相对序）

**秒边界允许窗口（参数 + 默认值，冻结）**：`T_APPLY_WINDOW_MS`。

- **默认值（冻结）**：`T_APPLY_WINDOW_MS = 5 ms`
- **取值范围（冻结）**：`1 ms ≤ T_APPLY_WINDOW_MS ≤ 20 ms`
- **用途（冻结）**：
   - `APPLIED_LATE` 判定阈值
   - “秒边界附近生效”的允许窗口

**调参验收指标（冻结）**：

- 在时间质量 `Q0(Locked)` 且网络/负载正常时，`APPLIED_LATE` 占比应可被压低至可接受水平（建议：≤ 0.1%）
- `AppliedTimestamp` 相对目标秒边界的抖动（绝对值）在 P99.9 上应 ≤ `T_APPLY_WINDOW_MS`

> **说明**：
> - 具体“质量档位判定”依赖实现（PTP 状态机、GNSS 状态、偏差与漂移等）。本协议要求**判定规则对外可观测**（见 1.3.6）。

### 1.3.3 正常工作流程（锁定后秒边界生效 / 软件秒边界事件 / 数据时间戳）

1. **上电 → 授时锁定**：DACS 优先尝试北斗/GNSS 锁定；成功后进入 Q0。
2. **前端秒边界生效**：当 SPS 下发 Control 且 Common Header.Timestamp 非 0 时，DACS 在前端执行侧目标秒边界（允许窗口内）应用配置并执行。
3. **软件秒边界事件上送**：DACS 将本次边界事实通过 `0x05 PPS Notify` 上送 SPS；SPS 侧由时基管理组件更新权威周期状态，任务调度中心基于该状态推进逻辑秒周期。
4. **数据打点**：
   - Common Header.Timestamp：各报文自身“发送/采样/生效/应答生成”的时间（见通用报文头定义）。
   - PPS Notify：Common Header.Timestamp 表示通知包发送时刻，边界事实以 PPS Payload 内 `PpsSeq/UtcSeconds/EdgeLocalTimestampNs` 为准。
   - Data：采样起始时刻建议与 Data Specific Header 内的时间信息一致（如存在）。

### 1.3.4 自动降级 / 恢复规则（触发条件、滞回、最短保持）

自动降级/恢复以**状态机**实现（可回归），并采用“触发持续时间 + 滞回 + 最短保持”避免抖动。

**状态定义**：

- `SRC_GNSS`：时间来源=北斗/GNSS
- `SRC_PTP`：时间来源=PTP
- `SRC_XO`：时间来源=本地晶振

**触发条件（示例形式，门限可配置）**：

- `SRC_GNSS → SRC_PTP`：GNSS 失锁持续 ≥ `T_GNSS_LOSS`（或 GNSS 状态进入 Lost/Searching 并持续）
- `SRC_PTP → SRC_XO`：PTP 非 Locked/Holdover 且持续 ≥ `T_PTP_LOSS`
- `SRC_PTP → SRC_GNSS`：GNSS 连续 Locked ≥ `T_GNSS_STABLE`
- `SRC_XO → SRC_PTP`：PTP 连续 Locked ≥ `T_PTP_STABLE`

**滞回与最短保持**：

- 每次切换后，至少保持当前来源 `T_MIN_HOLD`，期间忽略反向恢复条件。
- 对偏差阈值判定（如使用 `TimeOffset`），采用双门限：进入降级阈值 `TH_IN`、恢复阈值 `TH_OUT`（`TH_OUT < TH_IN`）。

> 关键要求：以上 `T_*` 与阈值必须在固件/驱动侧作为可配置参数，且在联调中可通过日志与心跳字段回归验证。

### 1.3.5 与控制/ACK/PPS Notify 的关系（定时生效允许条件、拒绝/降级行为、偏晚判定）

**定时生效允许条件（建议做成硬约束）**：

- 当时间质量处于 **Q0（Locked）**：允许执行 Common Header.Timestamp 指定的定时生效。
- 当时间质量处于 **Q1（Holdover）**：可配置“允许/禁止定时生效”，**V1.0 默认=禁止**。若禁止，按“拒绝/降级行为”处理。
- 当时间质量处于 **Q2（Freerun/Unknown）**：必须禁止定时生效。

**拒绝/降级行为（必须可判定）**：

- 若收到定时生效 Control（Timestamp 非 0），但当前不满足允许条件：
  - ScheduleApplyAck.应答结果 = `NOT_APPLIED`
  - ScheduleApplyAck.错误码 = `0x04`（时统未锁定，定时生效被拒绝/降级）
  - ScheduleApplyAck.生效时间戳 = `0`

**服务器侧周期推进口径（冻结）**：

- SPS 侧不得把“服务器本地时间到秒”直接当作权威换秒条件。
- SPS 侧应以 `0x05 PPS Notify` 形成的软件秒边界事件作为正式边界输入，由时基管理组件维护权威周期状态。
- 任务调度中心不得“认包换秒”，而应消费时基管理组件发布的权威周期状态。

**偏晚判定（必须可观测）**：

- 若 Control 被执行，但实际生效相对目标秒边界的偏差超过允许窗口 `T_APPLY_WINDOW_MS`：
  - ScheduleApplyAck.应答结果 = `APPLIED_LATE`
  - ScheduleApplyAck.错误详情建议填“超出秒边界的毫秒数”（与 3.2 表一致）

### 1.3.6 可观测性与验收（必须上报字段、日志点、联调检查步骤）

**必须上报/回显（用于观测与回归）**：

1. Heartbeat：
   - `SystemStatus_Alive`（含 PTP/GNSS 锁定位）
   - `SystemStatus_State`（含 PTP/GNSS 状态枚举、可见星数等）
   - `TimeCoordSource`（本次心跳/本机时间坐标来源：北斗/GNSS / PTP / 本地晶振）
   - `TimeOffset`（ns）
   - `ErrorCounters`（授时相关错误计数可并入 `CRC_Errors` 或后续扩展）
2. PPS Notify：
   - `PpsSeq / UtcSeconds / EdgeLocalTimestampNs`
   - `TimeSource / TimeQuality / EdgeFlags`
   - `SuggestedGlobalEpoch`
3. ScheduleApplyAck：
   - `应答结果 / 错误码 / 错误详情 / 生效时间戳`

**必须打点日志（建议至少包含以下事件）**：

- `TIME_SRC_SWITCH(old→new, reason, hold_timer, offsets)`：每次来源切换记录
- `TIME_QUALITY_CHANGE(Qx→Qy, reason)`：质量档变化记录
- `PPS_RX_OK(seq, utc_sec, src, quality)`：软件秒边界事件接收成功
- `PPS_REJECT(reason)`：软件秒边界事件拒绝/丢弃
- `CONTROL_TIMED_REJECT(frame, reason)`：定时生效被拒绝/降级记录

**联调验收步骤（可回归脚本化）**：

1. 正常：GNSS 锁定 → Heartbeat 显示 `TimeCoordSource=北斗/GNSS`，且 `TimeOffset` 在可接受范围；`PPS Notify` 正常连续上送；定时生效 OK。
2. 触发降级1：屏蔽/断开 GNSS → 在 `T_GNSS_LOSS` 后切到 PTP；心跳与 `PPS Notify` 可共同观测到来源变化；定时生效策略按 Q0/Q1 配置验证。
3. 触发降级2：再断开/破坏 PTP → 在 `T_PTP_LOSS` 后切到本地晶振；对 Timestamp 非 0 的 Control 必须出现 `NOT_APPLIED + 0x04`；`PPS Notify` 应可观测到质量下降。
4. 恢复：先恢复 PTP，再恢复 GNSS → 按 `T_*_STABLE` 与 `T_MIN_HOLD` 验证不抖动；切换事件可在日志、心跳与 `PPS Notify` 中回归。

## 2 Control Payload（控制载荷）— PacketType = 0x01

### 2.1 包结构

```
┌──────────────────────────────────────────────────────────┐
│  UDP Payload (32 + PayloadLen)                           │
├──────────────────────────────────────────────────────────┤
│  [0..31]    Common Header (32B)                          │
│             PacketType = 0x01                            │
│             PayloadLen = (表头长度 12) + (波束项数 × 32) + 4 │
│                        = 16 + N × 32                      │
├──────────────────────────────────────────────────────────┤
│  [32..]     Control Payload (变长)                        │
│             表头 (12B)                                   │
│             波束项 1 (32B)                               │
│             波束项 2 (32B)                               │
│             ...                                         │
│             波束项 N (32B)                               │
│             CRC32C (4B)                                 │
└──────────────────────────────────────────────────────────┘
```

Control 由 SPS 发往 DACS，用于承载一个波束安排周期（通常 1 秒）内的所有波束参数集合。采用"表头 + 多个波束项"的变长结构，表头承载周期级元数据，每个波束项（32B）对应一个波束在该周期内的工作参数。支持一次性下发 N 个波束（N 由表头中的"秒周期安排波束总数"指示）。

### 2.2 表头字段表

> Offset 基准：相对于 Control Payload 起始位置（UDP Payload Offset 32）

| Offset | 字段名称 | 类型 | 字节数 | 量化单位 | 说明 |
|--------|---------|------|-------|---------|------|
| **0** | **周期计数（FrameCounter）** | `uint32` | 4 | 1 | **含义**：逻辑周期号（字段名保留 `FrameCounter` 以兼容历史实现）。<br/>**取值**：从 0 递增，允许 `uint32` 自然回绕。<br/>**回绕比较**：必须使用半区间算法（Δ=无符号减法；`0<Δ<2^31`为新值；`Δ==0`相等；`Δ≥2^31`为旧值）。<br/>**用途**：<br/>• 与 Data 报文中的周期计数对应，实现控制与数据的周期级同步<br/>• 各 DACS 在同一逻辑周期号于前端执行侧秒边界（服务器侧对应 `0x05 PPS Notify` 软件秒边界事件）进行波束切换并采集数据<br/>• SPS 据此追踪波束安排周期的执行状态 |
| **4** | **帧波束总数（FrameBeamTotal）** | `uint16` | 2 | 1 | **含义**：搜索全空域需要安排的波束总数。<br/>**范围**：1 ~ 65535。<br/>**约束**：必须 > 0。<br/>**用途**：<br/>• 指示完成一次完整搜索扫描所需的波束总量<br/>• 与"秒周期安排波束总数"配合，支持跨周期分批下发 |
| **6** | **秒周期安排波束总数（BeamsPerSecond）** | `uint16` | 2 | 1 | **含义**：本秒周期内实际安排的波束个数（n）。<br/>**范围**：1 ~ 512。<br/>**约束**：<br/>• 必须 > 0<br/>• 不得超过 DACS 硬件可支持的单周期波束处理上限<br/>• 必须 ≤ 帧波束总数<br/>**用途**：<br/>• SPS 与 DACS 数据对齐的核心依据<br/>• DACS 据此预分配缓冲区与调度资源<br/>• 决定后续波束项的数量（n 个波束项）|
| **8** | **CPI 计数基数（CpiCountBase）** | `uint32` | 4 | 1 | **含义**：本周期波束的 CPI 计数基准值（m）。<br/>**取值**：从 0 开始，周期递增，允许 `uint32` 自然回绕。<br/>**回绕比较**：同 FrameCounter（半区间算法）。<br/>**用途**：<br/>• 每个波束的实际 CPI 计数 = CPI 计数基数 + 波束编号<br/>• 与 Data 报文中的 CPI 计数对应，实现波束级数据关联<br/>• 支持慢时间维度的数据组织与处理<br/>**注意**：波束编号定义在波束项中（字段 5） |

**表头总长 = 12 字节**

### 2.3 波束项字段表（每项 32 字节）

> Offset 基准：相对于表头末尾（UDP Payload Offset 32+12）；每项偏移递增 32

波束项承载该周期内单个波束的完整工作参数，包括天线指向、信号配置、采样参数等。每个波束项对应一个独立的波束配置，支持波束间参数的灵活变化。

| Offset | 字段名称 | 类型 | 字节数 | 量化单位 | 说明 |
|--------|---------|------|-------|---------|------|

### 2.4 CRC32C 校验字段

> Offset 基准：相对于 Control Payload 起始位置（UDP Payload Offset 32+12+N×32）

| Offset | 字段名称 | 类型 | 字节数 | 说明 |
|--------|---------|------|-------|------|
| **+0** | **CRC32C（CRC32C）** | `uint32` | 4 | **含义**：Control 载荷完整性校验（CRC-32C Castagnoli）。<br/>**覆盖范围**：Payload Offset 0 ~ (12+N×32-1)（表头 + 所有波束项）。<br/>**校验算法**：CRC-32C (Castagnoli)，多项式 0x1EDC6F41。<br/>**失败处理**：<br/>• 不得执行该 Control<br/>• **Control 属于控制平面闭环报文，DACS 必须回 ScheduleApplyAck（PacketType=0x02）**：应答结果=`NOT_APPLIED`，错误码=`0x01`（Control CRC32C 校验失败）<br/>• 回包与限速/抑制（冻结）：见 3.3.5 |

### 2.5 PayloadLen 计算

```
PayloadLen = 表头 + 波束项总长 + CRC32C
           = 12 + (秒周期安排波束总数 × 32) + 4
           = 12 + (N × 32) + 4
           = 16 + N × 32
```

**例子**：
- 1 个波束：PayloadLen = 16 + 32 = 48 字节
- 10 个波束：PayloadLen = 16 + 320 = 336 字节
- 100 个波束：PayloadLen = 16 + 3200 = 3216 字节
- 512 个波束（建议最大）：PayloadLen = 16 + 16384 = 16400 字节

**约束**：
- PayloadLen ≤ 65475（UDP 最大载荷 65507 - Common Header 32）
- 理论最大支持单周期波束数：`(65475 - 16) / 32 ≈ 2042` 个
- 实际限制：受 DACS 硬件处理能力约束（建议 ≤ 512）

### 2.6 字段更新特性

新架构采用**波束项全参数配置**模式，取消了"强不变/可变"的二分设计。所有射频与天线参数均在波束项中独立配置，支持波束间的灵活参数差异。

**设计优势**：

| 特性 | 说明 |
|------|------|
| **波束级独立性** | 每个波束项携带完整参数（天线指向、信号配置、采样设置等），支持波束间参数变化 |
| **周期级一致性** | 表头仅承载周期元数据（帧计数、波束总数、CPI 基数），不约束射频参数 |
| **混合模式支持** | 同一周期内可包含搜索波束、跟踪波束、测试波束（通过"波束工作状态"字段区分）|
| **动态适配** | 支持跟踪模式下的周期级参数更新（角度、增益等），无需重新初始化 RF 链路 |

**典型应用场景**：

- **纯搜索模式**：所有波束参数一致（工作频率、信号类型、带宽等），仅天线指向变化
- **搜索+跟踪混合**：搜索波束使用宽带宽、低增益；跟踪波束使用窄带宽、高增益
- **多目标跟踪**：不同波束针对不同目标，独立配置距离门、多普勒补偿、积累点数等

### 2.6.1 运行规则冻结（搜索 / 跟踪 / 混合）

为避免“协议能承载但系统规则未冻结”，V1.0 在运行层补充冻结如下：

1. **搜索模式**：默认采用预生成固定扫描表周期轮转，不要求逐周期闭环重规划。
2. **跟踪模式**：允许 SPS 基于上一逻辑周期处理结果生成下一逻辑周期表，但仍以“周期级定版”下发。
3. **混合模式**：跟踪波束优先级高于搜索补盲波束；当资源冲突时，搜索波束应让位于跟踪波束。
4. **热更新限制**：V1.0 不支持秒内任意热更新，统一采用“周期级定版、秒边界生效、秒内按表推进”。

### 2.7 Control 收发示例

**DACS 处理流程**：

1. **接收与初步校验**
   - 接收 Control 包（UDP 端口 8888）
   - 验证 Magic Word (`0x55AA55AA`)
   - 检查 DestID 是否匹配（或为广播地址 `0x10`）

   > 广播补充（冻结）：当 `DestID=0x10` 时，所有通过上述基础校验的 DACS **都必须**分别回 `ScheduleApplyAck`（其 `SourceID` 不同）；SPS 以 `(ControlEpoch, FrameCounter, SourceID)` 聚合判断各阵面生效结果。

2. **载荷完整性检查**
   - 读取 PayloadLen，验证一致性：`PayloadLen == 16 + 秒周期安排波束总数 × 32`
   - 验证 CRC32C（覆盖表头 + 所有波束项）

3. **参数解析与验证**
   - 遍历所有波束项，提取并校验参数范围
   - 检查波束编号是否重复
   - 验证位域字段（波束工作状态、天线展宽选择、周期组合模式等）的取值合法性
   - 验证长短码周期时序约束（长短组合模式下）

4. **波束调度准备**
   - 按波束编号排序存储波束配置
   - 根据 CPI 计数基数计算每个波束的 CPI 计数
   - 根据周期组合模式、波形类型、采样点数预分配数据缓冲区

5. **定时生效控制**
   - 检查 Common Header 的 Timestamp（生效时刻）
   - 若为全零：立即应用配置
   - 若非零：在指定时刻（前端执行侧秒边界 `±T_APPLY_WINDOW_MS`）切换配置

6. **周期内波束执行**
   - 按波束编号顺序依次执行采集/发射任务
   - 每个波束按其配置参数独立工作（周期模式、波形类型、增益等）
   - 长短组合模式下，按"短码上升沿→长码上升沿"时序发射
   - 采集数据打上对应的 CPI 计数与波束编号标签

7. **ScheduleApplyAck 反馈**
   - **必须**回复 ScheduleApplyAck（周期级应用结果回执），包含：
     * 帧计数回显
     * Applied 状态（成功/失败）
     * ErrorCode（如有错误）
     * 硬件状态快照

## 3 ScheduleApplyAck Payload（周期应用结果回执载荷）— PacketType = 0x02

ScheduleApplyAck 报文由 **DACS → SPS** 发送，用于对**上一秒下发的波束安排表（Control, PacketType=0x01）**的**应用与执行结果**进行回报。
ScheduleApplyAck 不再针对单条控制指令，而是以**“秒周期波束安排表”**为对象，形成稳定的秒级控制闭环。

---

### 3.1 包结构

```
┌──────────────────────────────────────────────┐
│  UDP Payload                                 │
├──────────────────────────────────────────────┤
│  通用报文头 Common Header (32B)              │
│    PacketType = 0x02                         │
│    Timestamp  = ScheduleApplyAck 生成时刻   │
│    SequenceNumber = Ack 自身序列号（DACS 发送端递增） │
│    ControlEpoch = 对应 Control 回显          │
├─────────────────────────────────────────────┤
│  ScheduleApplyAck Payload (固定 24B)        │
│    周期级执行结果回执                         │
├─────────────────────────────────────────────┤
│  [可选] Padding (0~N B, 全 0, 不参与校验)    │
└──────────────────────────────────────────────┘
```

**说明：**

* ScheduleApplyAck Payload **固定长度 24 字节**，满足 4 字节自然对齐
* `PayloadLen = 24`（或 28/32/64，若尾部填充）
* 若需要 DMA/缓存对齐，可在 ScheduleApplyAck Payload 末尾追加 **全 0 填充**，但**不参与 CRC32C 计算**
* **Ack 配对键（冻结）**：在 ScheduleApplyAck Payload 内新增 `AckedSequenceNumber`（回显被确认的 Control 序列号）作为配对键；Common Header.`SequenceNumber` 按 Ack 发送端递增，不再回显 Control 序列号

---

### 3.2 字段表（ScheduleApplyAck Payload）

> Offset 基准：相对于 ScheduleApplyAck Payload 起始位置（即 UDP Payload Offset = 32）

| Offset | 字段名称          | 类型       | 字节数 | 说明                                                                                                                                                                                                                                                                                                                                      |
| ------ | ------------- | -------- | --- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **0**  | **周期计数回显（AckFrameCounter）**     | `uint32` | 4   | **含义**：回显被确认的“上一逻辑周期波束安排表”的周期计数。<br/>**来源**：Control Payload 表头中的 `FrameCounter`。<br/>**语义**：唯一标识本 ScheduleApplyAck 所回报的周期级任务。                                                                                                                                                                                                                                     |
| **4**  | **被确认序列号（AckedSequenceNumber）** | `uint32` | 4 | **含义（冻结）**：回显被确认的 Control 报文 Common Header.`SequenceNumber`。<br/>**用途**：SPS 侧对 Ack 与 Control 的唯一配对键（配对时建议再结合 `SourceID/ControlEpoch` 做会话隔离）。 |
| **8**  | **应答结果（AckResult）**      | `uint8`  | 1   | **含义**：上一秒周期表的总体执行结果（周期级）。<br/>**取值定义**：<br/>• `0x00` = OK（按期生效，周期内全部波束执行完成）<br/>• `0x01` = PARTIAL（周期表生效，但部分波束未执行或被跳过）<br/>• `0x02` = NOT_APPLIED（周期表未生效，未执行）<br/>• `0x03` = APPLIED_LATE（生效但相对秒边界偏差超过允许窗口 `T_APPLY_WINDOW_MS`）<br/>• `0x04` = OVERRUN（周期内执行溢出，跨秒影响后续周期）                                                                                     |

| **10**  | **错误详情（ErrorDetail）**      | `uint16` | 2   | **含义**：错误码的补充说明，语义随 ErrorCode 变化。<br/>**约定示例**：<br/>• `0x03` 参数失败 → 失败的首个波束编号<br/>• `0x21` 到达过晚 → 超过“最晚提前量边界”的毫秒数（`late_ms`）<br/>• `0x23` 部分失败 → 首个失败波束编号<br/>• `0x28` 非致命告警 → 非零保留字段计数（可选；无则填 0）<br/>• 其余情况 → 填 0                                                                                                                                                                               |
| **12**  | **生效时间戳（AppliedTimestamp）** | `uint64` | 8 | **含义**：上一秒周期表的**实际生效时刻**（Unix Epoch，ms）。<br/>**约束**：<br/>• `OK / PARTIAL / APPLIED_LATE / OVERRUN` 必须填写<br/>• `NOT_APPLIED` 填 `0` |
| **20** | **载荷校验（PayloadCRC32C）**      | `uint32` | 4   | **含义**：ScheduleApplyAck Payload 校验值。<br/>**算法**：CRC32C（Castagnoli，多项式 0x1EDC6F41）。<br/>**覆盖范围**：Offset 0 ~ 19（不含本字段与尾部填充）。                                                                                                                                                                                                                           |

**ScheduleApplyAck Payload 固定长度 = 24 字节**

#### 3.2.1 ErrorCode 来源矩阵（冻结）

| ErrorCode | 名称 | 典型触发点（来源） | AckResult 约束 | ErrorDetail 语义（冻结） |
|---|---|---|---|---|
| `0x01` | Control CRC32C 失败 | Control 载荷 CRC32C 校验失败 | 必须 `NOT_APPLIED` | 固定填 `0` |
| `0x03` | 参数校验失败 | 字段范围/组合约束不满足（含 BeamID 重复、PeriodComboMode 约束冲突、ControlEpoch 不一致等） | 必须 `NOT_APPLIED` | 优先填“首个失败 BeamID”；若为非波束项原因则填 `0xFFFF` |
| `0x04` | 时统未锁定/定时拒绝 | `Timestamp!=0` 且当前时间质量不允许定时生效（Q1 默认禁用、Q2 必禁用） | 必须 `NOT_APPLIED` | 固定填 `0` |
| `0x20` | 周期表缺失 | 目标周期未收到任何“有效 Control”（通过基础校验+CRC+参数） | 必须 `NOT_APPLIED` | 固定填 `0` |
| `0x21` | 到达过晚 | Control 到达并完成校验时刻晚于目标秒边界 `-LeadTimeMs`（错过最晚提前量） | 必须 `NOT_APPLIED` | 超出毫秒数（`late_ms`） |
| `0x22` | 执行溢出/超时 | 周期内执行跨秒影响后续周期 | 必须 `OVERRUN` 或 `PARTIAL` | 首个受影响/失败 BeamID（可选） |
| `0x23` | 部分波束失败 | 周期生效但部分波束被跳过/失败 | 必须 `PARTIAL` | 首个失败 BeamID |
| `0x28` | 非致命告警 | Reserved/扩展字段非 0（MBZ 放宽兼容） | 允许 `OK / PARTIAL / APPLIED_LATE / OVERRUN` | 可选：非零保留字段计数 |

---

### 3.3 特别说明（语义与时序冻结）

#### 3.3.1 周期级应用结果回执原则（核心语义）

* ScheduleApplyAck **只回报“上一秒下发的波束安排表”的应用结果**
* 不对单个波束、不对单条参数、不对瞬时配置单独回执
* 一个 ScheduleApplyAck ⇔ 一个帧计数 ⇔ 一个 1 秒周期

该设计保证：

* 控制平面节奏稳定（1 Hz）
* FPGA/驱动侧无需维护细粒度回执状态
* SPS 可基于帧计数形成确定性的控制闭环

---

#### 3.3.2 必须满足的控制—应答时序

为保证 ScheduleApplyAck 语义成立，控制与执行需满足以下约束：

1. **提前下发原则**
   SPS 必须在周期 `T` 内，下发 **周期 `T+1`** 的波束安排表

   **最晚提前量（冻结）**：`LeadTimeMs`。

   - 定义：对目标秒边界 `B`（即 Control Common Header.`Timestamp` 指示的期望生效时刻），SPS 必须确保 Control **到达并完成校验**的时刻 `t_valid` 满足：`t_valid ≤ B - LeadTimeMs`。
   - 默认值（冻结）：`LeadTimeMs = 200 ms`

   **提前接收持有窗口（冻结）**：`HoldWindowMs`。

   - 定义：当 `t_valid` 早于目标秒边界 `B` 时，DACS 必须支持将该周期表**持有**至 `B` 并按秒边界生效；该能力在 `B - HoldWindowMs ≤ t_valid ≤ B` 的到达区间内必须成立。
   - 默认值（冻结）：`HoldWindowMs = 800 ms`
2. **执行—应答错位一秒**

   * 周期 `T`：执行周期表 `T`
   * 周期 `T+1`：发送 ScheduleApplyAck，回执周期表 `T`
3. **禁止“到点才下发”**
   若 Control 到达并完成校验时刻晚于目标秒边界 `-LeadTimeMs`，则视为 `ErrorCode = 0x21`（到达过晚）

---

#### 3.3.3 ScheduleApplyAck 与通用报文头的一致性要求

* **Common Header.SequenceNumber（冻结）**：ScheduleApplyAck 的序列号按 **DACS 发送端递增**（不回显 Control 序列号）
* **ControlEpoch（冻结）**：必须回显对应 Control 的控制纪元（用于会话隔离）
* **AckedSequenceNumber（冻结）**：必须回显被确认的 Control 报文 Common Header.`SequenceNumber`，作为唯一配对键
* **Timestamp（Common Header）**：

   * 对 ScheduleApplyAck，表示“应答生成时间”
   * ScheduleApplyAck Payload 不再携带“应答生成时间戳”，以 Common Header.Timestamp 为准

---

#### 3.3.4 未收到有效 Control 的默认行为（冻结）

当某个周期 `T` 未收到任何“有效 Control”（通过：通用头基础校验 + PayloadLen 一致性 + Control CRC32C + 参数校验）时：

1. DACS **不得执行**该周期的波束安排（不得进入发射/采集调度）。
2. DACS 在周期 `T+1` 必须回 `ScheduleApplyAck` 回执周期 `T`：`AckResult=NOT_APPLIED, ErrorCode=0x20, ErrorDetail=0`。
3. 数据平面（`0x03`）在该周期内的处理策略：默认**不发送**对应波束数据；如实现必须发送，则 RAW 区域必须填 0，且 `HealthSummary` 应标记“控制缺失/未执行”（具体位定义可由实现约定）。

---

#### 3.3.5 不可执行类错误：统一回 Ack + 全局限速/抑制（冻结）

为消除“有的错误回包、有的错误静默”的不一致性，凡 **Control（0x01）已通过基础可识别检查**（至少满足：Magic/ProtocolVersion/PacketType/长度一致性/DestID 匹配，且可读取 `SourceID/ControlEpoch/SequenceNumber`）但因错误而**不可执行**的情形，处理统一冻结如下：

1. **必须回 Ack（冻结）**
   - DACS 必须回 `ScheduleApplyAck (0x02)`，并设置：`AckResult=NOT_APPLIED`，同时携带明确 `ErrorCode/ErrorDetail`（例如：CRC 失败=0x01、参数/纪元不一致=0x03、时统拒绝=0x04、到达过晚=0x21 等）。
   - 该规则覆盖所有“不可执行类错误”，不允许因错误类型不同而改为静默丢弃（除非触发第 2 条限速抑制）。
2. **全局限速/抑制（冻结）**
   - 为避免异常风暴导致反向拥塞，DACS 对上述“不可执行类错误 Ack”必须实施全局令牌桶限速。
   - **限速键（冻结）**：按 `(SourceID, 源 IP, ControlEpoch)` 组合。
   - **令牌桶参数（冻结）**：`rate = 10 Ack/s`，`burst = 20 Ack`。
   - 当触发限速抑制时：允许抑制该次 Ack 发送，但必须计数（建议：`AckSuppressed_Count`）以便联调诊断。

#### 3.3.6 多阵面 Ack 聚合规则（冻结）

当同一条 Control 以广播方式发往多个 DACS 阵面时，SPS 侧必须按 `(ControlEpoch, AckFrameCounter, AckedSequenceNumber, SourceID)` 聚合 `ScheduleApplyAck`，并以阵面为粒度判定结果：

1. **单阵面成功**：仅表示对应 `SourceID` 阵面本周期应用成功，不得外推为“全阵面成功”。
2. **全阵面成功**：只有当本周期目标阵面的 Ack 全部返回且均为 `OK / PARTIAL / APPLIED_LATE / OVERRUN` 中的允许结果时，SPS 才可判定“本周期全阵面已闭环”。
3. **部分缺失**：若在聚合超时窗口内部分阵面未返回 Ack，SPS 必须将缺失阵面标记为“未确认”，不得自动按成功补齐。
4. **阵面隔离**：任一阵面的 `NOT_APPLIED` 或缺失，不得覆盖其他已确认阵面的独立结果，但应抬升本周期系统级告警等级。

## 4 Data Specific Header（数据专用头）— PacketType = 0x03

### 4.1 包结构

数据平面（`0x03`）不携带应用层 CRC32C（见 `1.2.2 CRC 策略矩阵（冻结）`）。

**版本边界说明（冻结）**：

- 本文档定义的 `0x03 Data` 为 V1.0 正式数据面格式。
- 历史上存在的“固定 5120B UDP 包 / 16 包每 PRT / 末包包尾”草案，仅作为历史背景，不再作为 V1.0 正式实现基线。
- 若现场仍保留历史格式实现，必须在项目级联调说明中单独声明，不得与本文档定义的 V1.0 数据面格式混淆并存。

**非尾包（分片索引 = i，且 i < TotalFrags-1）结构**：

```
┌──────────────────────────────────────────────────────────┐
│  UDP Payload (Total: 32 + 40 + RAW_FragLen_i)           │
├──────────────────────────────────────────────────────────┤
│  [0..31]     Common Header (32B)                        │
│              PacketType = 0x03                           │
│              PayloadLen = 40 + RAW_FragLen_i             │
├──────────────────────────────────────────────────────────┤
│  [32..71]    Specific Header (40B, 4-Byte 对齐)          │
├──────────────────────────────────────────────────────────┤
│  [72..]      RAW Fragment Payload (变长, 4B 对齐填充)     │
└──────────────────────────────────────────────────────────┘
```

**尾包（分片索引 = TotalFrags-1）结构（冻结：尾包携带 Snapshot）**：

```
┌───────────────────────────────────────────────────────┐
│  UDP Payload (Total: 32 + 40 + 40 + RAW_FragLen_tail) │
├───────────────────────────────────────────────────────┤
│  [0..31]    Common Header (32B)                       │
│             PacketType = 0x03                         │
│             PayloadLen = 40 + 40 + RAW_FragLen_tail    │
├───────────────────────────────────────────────────────┤
│  [32..71]   Specific Header (40B, 4-Byte 对齐)         │
├───────────────────────────────────────────────────────┤
│  [72..111]  Execution Snapshot (40B, 4-Byte 对齐)      │
├───────────────────────────────────────────────────────┤
│  [112..]    RAW Fragment Payload (变长, 4B 对齐填充)    │
└───────────────────────────────────────────────────────┘
```

Data 由 DACS 发往 SPS，承载原始 I/Q 回波数据。采用 4-Byte Natural Alignment 以适配高速 DMA。当数据量超过 MTU 时进行应用层分片：

- **每个分片均携带：Common Header + Specific Header + RAW 片段**
- **Snapshot 携带方式（冻结）**：Execution Snapshot 由**尾包（FragIndex = TotalFrags-1）携带**；当 `TotalFrags=1` 时该唯一分片同时为尾包，因此仍携带 Snapshot。
- **尾包判定**：`FragIndex == TotalFrags - 1`

**分片时 PayloadLen 计算（冻结）**：

- 非尾包：`PayloadLen = 40 + RAW_FragLen_i`
- 尾包：`PayloadLen = 40 + 40 + RAW_FragLen_tail`

**重组键**：接收端按 `(ControlEpoch, SourceID, 周期计数(FrameCounter), 波束编号, CPI计数, 脉冲序号, ChannelMask, DataType)` 标识同一数据流；同一重组键内各分片这些字段必须一致。

**丢包补零粒度（冻结：按分片补零）**：

- 接收端在重组键维度维护 `FragIndex=0..TotalFrags-1` 的分片到达位图。
- 对缺失分片，补零粒度冻结为“按分片补零”，即：对每一个缺失的 `FragIndex=k`，插入一段全 0 字节，长度为“该分片应有的 RAW 有效载荷长度”（见下文“缺失分片长度可确定性”）。

**缺失分片长度可确定性（冻结）**：

为确保接收端在缺片时仍能确定应补零的字节数，本协议冻结 Data 分片载荷长度规则如下：

- **非尾包固定采样点数（冻结）**：对同一重组键内所有**非尾包**，Specific Header.`SampleCount` 必须固定为工程配置常量 `SampleCountFixed`（该常量在系统集成时冻结；同一系统内不得变更）。据此可确定“非尾包 RAW 有效载荷字节数”=
   `SampleCountFixed × 复数大小 × popcount(ChannelMask)`。
- **尾包实际长度字段（冻结）**：Specific Header 的 `TailFragPayloadBytes`（见 4.2，Offset=38）用于描述尾包 RAW **有效载荷**字节数（不含 padding）。该字段必须在同一重组键的**所有分片中保持一致**，以便尾包丢失时仍可确定尾包应补零长度。

注：padding 仅用于封装对齐；补零与重组以“RAW 有效载荷字节数”为准。

**重组规则（冻结）**：

- **不要求到达顺序**：分片允许乱序到达。
- **完整性条件**：在 `T_reasm` 超时前，若同一重组键内收到的 `FragIndex` 索引集合完整且不重复（覆盖 `0..TotalFrags-1`），则输出完整帧。
- **超时输出（冻结）**：当 `T_reasm` 到期仍未收齐 `0..TotalFrags-1` 时：立即按 `FragIndex` 顺序输出一帧，缺失分片按“按分片补零”补齐；同时在接收端重组元信息中标记 `IncompleteFrame=1` 并累加缺片计数。
- **重复处理**：若同一重组键内出现重复 `FragIndex`，后到分片应丢弃并计数。
- **迟到分片（冻结）**：重组超时并输出后，后续到达的同重组键分片一律丢弃并计数。
- **重组步骤**：按 `FragIndex` 从小到大顺序拼接各分片 RAW 有效载荷；对缺失分片按规则补零。

#### 4.1.1 重组资源与超时（冻结）

为防止重组缓存失控、以及使“丢包补零”行为可回归，接收端（SPS）对每个重组键必须冻结如下参数与策略：

- `MAX_REASM_BYTES_PER_KEY`：单重组键最大缓存字节数（含已收分片数据与元数据）。默认值（冻结）：`16 MiB`。
- `MAX_TOTALFRAGS`：允许的最大 `TotalFrags`。默认值（冻结）：`256`。超出则丢弃该帧并计数。
- `T_reasm`：重组超时（从收到该重组键的第一个分片起计）。默认值（冻结）：`100 ms`。

超时后的输出策略已在 4.1 主规则中冻结为“补零后输出（IncompleteFrame=1）”；同时规定“迟到分片一律丢弃并计数”。

### 4.4 RAW Payload 格式

> Offset 基准：UDP Payload Offset 72（非尾包）或 UDP Payload Offset 112（尾包）

**数据类型与复数样点大小**：

| DataType | I 分量 | Q 分量 | 复数大小 |
|----------|-------|-------|---------|
| `0x01` (Int16) | `int16` | `int16` | 4 Bytes |

**通道交织规则**：按 ChannelMask 的 Bit 顺序（从低到高）排列。例如 `ChannelMask=0x000F`（4 通道）时，每个采样点的排列为：`[Σ_I, Σ_Q, ΔAz_I, ΔAz_Q, ΔEl_I, ΔEl_Q, Aux_I, Aux_Q]`。

**RAW Payload 大小计算**：

```
通道数 = popcount(ChannelMask)               # 统计 ChannelMask 中置位的比特数
每采样点字节数 = 复数大小 × 通道数           # 单个采样点占用的字节数
RAW_Payload_大小 = 采样点数 × 每采样点字节数  # 原始数据载荷的总字节数
```

**RAW_Size / RAW_FragLen（冻结）**：

- `RAW_Size` / `RAW_FragLen_i` 指 **RAW 区域线上实际字节长度**，包含对齐填充（padding）。
- 对齐填充仅用于满足“每个 Data 报文 RAW 区域 4-Byte 对齐”的封装要求，填充值必须为 `0x00`。

对任意一个 Data 分片，其 RAW 区域长度计算如下：

```
RAW_FragPayloadBytes_i = SampleCount × 复数大小 × popcount(ChannelMask)
RAW_FragPadding_i = (4 - (RAW_FragPayloadBytes_i mod 4)) mod 4
RAW_FragLen_i = RAW_FragPayloadBytes_i + RAW_FragPadding_i
```

**固定分片载荷长度（冻结）**：为配合“按分片补零”，对同一重组键内所有非尾包，发送端必须满足：

- `SampleCount == SampleCountFixed`（工程配置常量；同一系统内固定）
- 尾包的 `RAW_FragPayloadBytes_tail == TailFragPayloadBytes`（见 4.2 Offset=38；不含 padding）

接收端据 `SampleCountFixed` 与 `TailFragPayloadBytes` 可在缺片时确定补零长度。

注：
- 当 `RAW_FragPayloadBytes_i` 本身已 4 字节对齐时，`RAW_FragPadding_i = 0`。
- 若不分片（`TotalFrags = 1`），则 `RAW_Size = RAW_FragLen_0`。

**计算示例**：
- 复数数据类型 `0x01` (Int16)：复数大小 = 4 字节
- 通道掩码 `0x0003`（2 个通道）：通道数 = 2
- 采样点数 = 1024
- 则 RAW_Payload_大小 = 1024 × 4 × 2 = 8192 字节

**对齐约束**：若 RAW Payload 字节数 % 4 ≠ 0，尾部补充零字节至 4 字节边界；该 padding 计入本分片 `RAW_FragLen_i`（即 `RAW_Size`）。所有多字节字段（含 Float32）均采用小端序（Little-Endian）编码。

## 5 Heartbeat Payload（心跳载荷）— PacketType = 0x04

### 5.1 包结构

```
┌────────────────────────────────────────────────────┐
│  UDP Payload (固定 80 Bytes = 32 + 48)              │
├────────────────────────────────────────────────────┤
│  [0..31]   Common Header (32B)                     │
│            PacketType = 0x04, PayloadLen = 48      │
├────────────────────────────────────────────────────┤
│  [32..79]  Heartbeat Payload (48B)                 │
└────────────────────────────────────────────────────┘
```

Heartbeat 由 DACS 单向发往 SPS，周期性上报设备健康状态。推荐 1 Hz，可配置 0.1-10 Hz。不参与控制闭环，丢失不触发告警或重启。专注于设备健康监控和基本运行指标。携带 CRC32C 确保运维数据准确性。

### 5.2 设计原则

心跳部分重新聚焦于**真正的设备健康检查**，移除无用字段，节省带宽：

| 项目 | 说明 |
|------|------|
| **取缔字段** | `LastTaskID`（任务管理已在控制指令中通过 CPI 计数基数实现）|
| **移除扩展** | `Reserved_Extend`（不预留扩展空间，节省 24 字节带宽） |
| **拆分 SystemStatus** | 从 64-bit 拆分为两个 32-bit 字段，分离关键状态与基本位图 |
| **聚焦内容** | 仅保留：时间同步、温度、工作模式、关键状态、错误计数 |

### 5.3 字段表

> Offset 基准：相对于 Heartbeat Payload 起始位置（UDP Payload Offset 32）

| Offset | Field | Type | Bytes | 说明 |

## 6 校验算法参数速查

### 6.1 HCS（已移除）

V1.0 起通用报文头不再包含 CRC16/HCS 字段。

### 6.2 CRC32C — CRC-32C (Castagnoli)

| 参数 | 值 |
|------|-----|
| 多项式 | `0x1EDC6F41`（反射：`0x82F63B78`） |
| 初始值 | `0xFFFFFFFF` |
| 输入反射 | true |
| 输出反射 | true |
| 最终异或 | `0xFFFFFFFF` |
| 线上字节序 | Little-Endian |

**各报文 CRC32C 覆盖范围**：

| 报文 | 覆盖起止（Payload 内 Offset） | 校验字段位置 |
|------|----------------------------|------------|
| Control (0x01) | `0 .. (12 + N×32 - 1)` | `12 + N×32 .. 12 + N×32 + 3` |
| ScheduleApplyAck (0x02) | `0 .. 15` | `16 .. 19` |
| Heartbeat (0x04) | `0 .. 43` | `44 .. 47` |
| PPS Notify (0x05) | `0 .. 35` | `36 .. 39` |

> 说明：若某些报文为了 DMA/缓存对齐在 CRC32C 后追加全 0 Padding，则 Padding **不参与 CRC32C**，但计入 `PayloadLen`。

### 6.3 FCS — CRC-32 (IEEE 802.3)

由 NIC 硬件处理，协议层无感知。多项式 `0x04C11DB7`，FCS 失败时 NIC 静默丢弃帧。
