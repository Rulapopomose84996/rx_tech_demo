# 发送端默认实现说明

> 这份文档描述当前主工作区默认 sender 的真实运行口径。
> 它面向联调、接收端适配与 AI 上下文，不承担历史实验归档职责。

## 1. 当前默认版本

当前主工作区默认 sender 是：

- 单进程
- 三线程
- 三口并发：
  - `sender0`
  - `sender1`
  - `sender2`
- 每口一个完整发送上下文
- 每口单队列发送：
  - `queue-id=0`
- sender 热路径当前采用：
  - `block-batch fast path`

这个默认版本的定位是：

- 作为当前最稳定的发送端联调基线
- 已具备真实链路、真实 AF_XDP、zero-copy 的可运行证据
- 已足够支撑接收端适配与 sender/receiver 联调

它当前不是“每口已经稳定达到 `6 Gbps`”的宣称版本。

## 2. 当前默认链路映射

当前三条默认链路是：

| 发送口 | sender 地址 | receiver 地址 | sender 源端口 | receiver 目标端口 |
|------|------|------|------|------|
| `sender0` | `172.20.11.11` | `172.20.11.100` | `9999` | `9999` |
| `sender1` | `172.20.12.12` | `172.20.12.100` | `9999` | `9999` |
| `sender2` | `172.20.13.13` | `172.20.13.100` | `9999` | `9999` |

这意味着当前接收端 AI 或接收端实现最应该假设的是：

- 三条独立 UDP 流
- 三条流都发往 `9999/udp`
- 三条流按口独立接收、独立统计、独立重组

## 3. 当前默认运行入口

发送端当前入口仍然是：

- 单口：[`scripts/run_demo.sh`](/home/klein/WorkSpace/sender/scripts/run_demo.sh)
- 三口：[`scripts/run_demo_multi.sh`](/home/klein/WorkSpace/sender/scripts/run_demo_multi.sh)

### 单口长期联调

适合只开放一个发射口给接收端联调：

```bash
sudo env \
  DEMO_IFNAME=sender0 \
  DEMO_XDP_BACKEND=real \
  DEMO_RUN_UNTIL_STOPPED=1 \
  ./scripts/run_demo.sh
```

### 三口长期联调

适合三口一起联调：

```bash
sudo env \
  DEMO_XDP_BACKEND=real \
  ./scripts/run_demo_multi.sh
```

## 4. 当前终端与日志行为

当前 sender 输出分成两层：

### 终端层

终端只显示人读友好的摘要，不再滚详细调试信息：

- 首次输出：
  - `[本次发射状态]`
  - 或多口的 `[本次发射状态][端口 N]`
- 运行中输出：
  - `[实时摘要][sender0][HH:MM:SS]`
  - 多口时每口各自摘要
- 结束时输出：
  - `[结束摘要][sender0]`
  - 多口时额外输出 `[结束汇总]`

终端摘要当前只包含：

- 已发数据量（MiB）
- 接收数据量（MiB）：`待接收端反馈`
- 丢包率（%）：`待接收端反馈`
- 平均速率（Gbps）
- 状态

### 文件层

详细日志完整落盘到项目根目录 [`logs`](/home/klein/WorkSpace/sender/logs)：

- 单口：
  - `logs/send_sender0_YYYYMMDD_HHMMSS.log`
- 多口：
  - `logs/send_multi_YYYYMMDD_HHMMSS.log`

详细日志中保留：

- 预检查
- attach 细节
- 原始 `demo_sender` 状态行
- 心跳 / final / aggregate 原始行
- 异常与警告

## 5. 当前默认参数边界

接收端通常不需要关心 sender 的内部批处理参数，但当前默认实现确实基于：

- `max_blocks_per_batch = 4`
- `max_packets_per_batch = 128`
- `max_consecutive_submit_failures = 4`

这些属于 sender 内部实现，不属于线上协议语义。

## 6. 当前已验证发送段事实

当前 sender 已经验证成立的事实是：

- 可进入 `backend=af_xdp`
- 可进入 `transport=real`
- 可进入 `mode=zero-copy`
- 单口 `sender0` 已有新的 root 真实 `5s` 样本：
  - `avg_gbps=6.17163`
  - `tx_errors=0`
  - `backpressure=0`
- 三口当前最稳 sender 候选样本约为：
  - `5s`: 每口约 `5.59 ~ 5.68 Gbps`
  - `60s`: 每口约 `5.34 ~ 5.48 Gbps`
  - `180s`: 每口约 `5.46 ~ 5.59 Gbps`

这些样本里：

- `tx_errors=0`
- `backpressure=0`

## 7. 接收端 AI 最需要记住的事实

如果这份文档被用作接收端 AI 上下文，最重要的是这几条：

1. 当前 sender 是三条独立 UDP 流，不是一条合流。
2. 当前三条流的目标地址分别是：
   - `172.20.11.100:9999`
   - `172.20.12.100:9999`
   - `172.20.13.100:9999`
3. 当前三条流都使用相同的 `DemoHeader`。
4. 当前接收端应按“每口独立接收、每口独立重组”实现。
5. sender 终端摘要里的“接收数据量/丢包率”目前不是真实测量值，显示为 `待接收端反馈` 是正常行为。

## 8. 当前边界

当前默认 sender 的边界是：

- 它是当前最稳的发送端联调基线
- 它已经足以支撑接收端开发和 sender/receiver 联调
- 它当前仍未把三口每口都稳定推到 `6 Gbps`
- 若后续继续冲每口 `6 Gbps`，应视为下一轮 sender 深度优化，而不是当前默认版本的定义内容
