# 发送端默认实现说明

> 这份文档说明当前主工作区推荐采用的 sender 默认实现。
> 它描述的是“当前稳定可用的默认发送端版本”，不是所有历史实验的合集。

## 1. 当前默认版本

当前主工作区推荐采用的 sender 默认实现是：

- 单进程
- 三线程
- `sender0` / `sender1` / `sender2`
- 每口一个完整发送上下文
- 每口单队列发送：
  - `queue-id=0`
- sender 热路径采用：
  - `block-batch fast path`

这个版本的默认目标不是声称“已经达到每口 `6 Gbps`”，而是：

- 在当前硬件和单队列约束下
- 提供一个已经过真实链路验证的
- 稳定 `5.5+ Gbps / port` sender 候选实现

## 2. 为什么它是默认实现

当前默认版本之所以成立，是因为它同时满足：

- 三口真实链路已打通
- 三口都能进入 `transport=real` / `mode=zero-copy`
- `tx_errors=0`
- `backpressure=0`
- 已获得短时和长时样本

并且，相比已经验证过的其它 sender 优化尝试：

- 简单改绑核：无明显收益
- 简单加大 burst：无明显收益，部分情况下出现负作用
- 预构包模板：真实性能下降
- 首版减少 submit/wakeup 频率：真实性能下降

当前只有 `block-batch fast path` 这条更重重构线，稳定把每口提升到了 `5.5+ Gbps`。

## 3. 默认参数

当前默认 sender 实现的核心参数是：

- `max_blocks_per_batch = 4`
- `max_packets_per_batch = 128`
- `max_consecutive_submit_failures = 4`

这些参数当前是内部默认值，不要求接收端关心。

发送端运行入口仍然是：

- 单口：[`scripts/run_demo.sh`](/home/klein/WorkSpace/sender/scripts/run_demo.sh)
- 三口：[`scripts/run_demo_multi.sh`](/home/klein/WorkSpace/sender/scripts/run_demo_multi.sh)

## 5. 当前已验证样本

### 5s 样本

- `sender0 = 5.67268 Gbps`
- `sender1 = 5.58853 Gbps`
- `sender2 = 5.6818 Gbps`
- aggregate `16.943 Gbps`

### 60s 样本

- `sender0 = 5.47698 Gbps`
- `sender1 = 5.34124 Gbps`
- `sender2 = 5.44987 Gbps`
- aggregate `16.2681 Gbps`

### 180s 样本

- `sender0 = 5.59097 Gbps`
- `sender1 = 5.46372 Gbps`
- `sender2 = 5.55392 Gbps`
- aggregate `16.6086 Gbps`

这些样本中：

- `tx_errors = 0`
- `backpressure = 0`

## 6. 这份默认实现对接收端意味着什么

接收端不需要关心 sender 内部是如何做 block-batch 的。

接收端只需要知道：

- 这是三条独立 UDP 流
- 每口使用同一种 `DemoHeader`
- 每口独立重组
- 当前 sender 已稳定在 `5.5+ Gbps / port`

接收端适配细节请看 [`docs/接收端适配说明.md`](接收端适配说明.md)。

## 7. 当前边界

当前默认实现的边界是：

- 它是 sender 阶段当前最稳的默认版本
- 它已经足以支撑接收端适配和联调
- 它还没有把每口推到 `6 Gbps`
- 若后续继续冲 `6 Gbps`，应视为下一轮 sender 深度优化，而不是当前默认版本的定义内容
