# 当前接收端代码实现与执行逻辑详解

## 1. 文档目的

本文档只描述当前仓库已经落地的接收端实现，不把历史 AF_XDP benchmark 叙述、未实现规划或理想终态写成既成事实。

当前主线位于 `src/receiver`，是以 DPDK 为入口的一条 Phase 3 接收链路：

- 真实网卡收包
- UDP payload 提取
- 协议解析与校验
- 语义化统计
- capture 落盘
- CPI 热路径骨架接入

## 2. 当前源码结构

| 目录 | 作用 |
|---|---|
| `src/receiver/app` | 可执行入口、CLI、公共启动逻辑 |
| `src/receiver/runtime` | 配置、运行器、停止控制、输出路径准备 |
| `src/receiver/ingress/dpdk` | DPDK 收包与 ARP 应答 |
| `src/receiver/protocol` | UDP payload 提取、解析、校验、序列解释 |
| `src/receiver/core` | owner loop，串起主热路径 |
| `src/receiver/sidecar` | 指标、汇总、RunSummary |
| `src/receiver/storage` | CPI 上下文、slot 写入、进度推进 |
| `src/receiver/admit` | CPI 准入决策 |
| `src/receiver/output` | 输出对象定义 |
| `src/receiver/finalize` | CPI finalize 骨架 |
| `src/legacy` | 历史/兼容 AF_XDP 路径 |

当前主可执行文件是：

- `rx_receiver_dpdk`
- `rxbench_dpdk`

它们都从 `app/main_dpdk.cpp` 进入统一的 `run_app("dpdk", argc, argv)`。

## 3. 启动与配置流程

当前启动主流程在 `src/receiver/app/common/app_main_common.cpp`。

实际顺序是：

1. 解析 CLI。
2. 只接受 `--config`、`--dry-run`、`--help`。
3. 加载默认配置。
4. 若提供 `--config`，加载 section 化配置文件。
5. 强制把 `backend_name` 设为当前入口对应后端，这里是 `dpdk`。
6. 对 capture、log、protocol 关键字段做校验。
7. `--dry-run` 时打印展开后的最终配置。
8. 非 dry-run 时创建 backend、metrics，并进入 `ReceiveRunner`。

当前示例配置文件采用 section 风格，重点 section 为：

- `[capture]`
- `[network]`
- `[dpdk]`
- `[runtime]`
- `[protocol]`
- `[log]`
- `[feedback]`

协议默认参数仍是：

- `udp_packet_size = 2048`
- `channels_per_prt = 3`
- `packets_per_channel = 9`

## 4. ReceiveRunner 做什么

`src/receiver/runtime/src/receive_runner.cpp` 负责运行期框架动作：

- 安装 `SIGINT` / `SIGTERM` 停止处理
- 初始化 backend
- 准备 capture 输出目录
- 打开 `capture_packets.bin` 和 `capture_index.csv`
- 调用 `OwnerLoop::run()`
- 收集 `CaptureArtifacts`
- 补齐 `RunSummary`
- 关闭 backend

capture 启用时，接收端会实时写出：

- `capture_packets.bin`
- `capture_index.csv`

当前 capture index 表头是：

```text
cpi,channel,prt,packet_index,packet_kind,payload_len,valid
```

这已经是 Phase A 简化后的输出，不再记录旧的端口/offset/face 级别细节。

## 5. Owner Loop 的真实热路径

真正的主循环在 `src/receiver/core/src/owner_loop.cpp`。

当前热路径可以概括成：

```text
DpdkIngress.recv_burst
  -> UdpPayloadAssembler.push
  -> matches_packet_filter
  -> PacketParser.parse
  -> PacketValidator.validate
  -> ProtocolSequenceInterpreter.interpret
  -> CpiAdmission
  -> SlotWriter
  -> ProgressTracker
  -> CpiFinalizer
  -> MetricsCollector / capture / summary
```

这里需要明确几点：

- 当前链路仍是“边收边处理”，不是先把所有数据攒满后再统一导出。
- capture 是实时写入的，不依赖暂停动作。
- Phase D 的 `storage/admit/output/finalize` 已经接入，但仍属于骨架型实现，不应描述成完整业务收尾系统。

## 6. DPDK ingress 当前职责

`src/receiver/ingress/dpdk/src/dpdk_backend.cpp` 当前负责：

- DPDK EAL 初始化
- 端口和 queue 配置
- mbuf pool 创建
- 批量收包
- 统计原始收包信息
- 按需应答 ARP 请求

它向上层交付统一 `PacketDesc`，后续协议处理不在 ingress 层完成。

## 7. UDP payload 组装层

`src/receiver/protocol/src/udp_payload_assembler.cpp` 负责：

- 解析以太网头、IPv4 头、UDP 头
- 提取 UDP payload
- 在需要时做 IPv4 分片重组
- 输出 `UdpPayloadFrame`

当前上送给协议层的 `UdpPayloadFrame` 已经不再携带旧的 DPDK 元数据字段，协议边界只保留与后续过滤和解析直接相关的信息。

## 8. 当前协议模型

### 8.1 ProtocolSpec

`src/receiver/protocol/rxtech/protocol_spec.h` 当前集中定义协议形状：

- 控制表 magic：`0x55AAFF00`
- 数据包 magic：`0x55AAFF03`
- 尾包 tail：`0x55AAFF30`
- 头部长度：16 字节
- UDP 包长：2048 字节
- 数据负载长度：2032 字节
- 每 PRT 3 个通道
- 每通道 9 个包
- 前 8 包每包 508 个 IQ
- 最后 1 包 476 个 IQ，剩余字节必须补零

### 8.2 PacketParser

`src/receiver/protocol/src/sample_packet_parser.cpp` 当前解析 16 字节小端协议头，得到：

- `PacketKind`
- `cpi`
- `channel`
- `prt`
- `packet_index`
- `tail`
- payload 指针和长度

这已经不是旧文档里的 `DemoHeader` / `TPDX` 模型。

### 8.3 PacketValidator

`src/receiver/protocol/src/sample_packet_validator.cpp` 当前做的校验包括：

- 控制表包 payload 长度是否匹配
- `channel` 是否落在 `0..channels_per_prt-1`
- `packet_index` 是否落在 `1..packets_per_channel`
- 数据 payload 是否等于 `packet_data_size`
- `tail` 是否为 `0` 或 `magic_tail`

### 8.4 ProtocolSequenceInterpreter

`src/receiver/protocol/src/protocol_sequence_interpreter.cpp` 当前不再按“到达顺序推导 PRT / channel / packet index”，而是直接基于已解析字段和 `ProtocolSpec` 解释协议语义。

对于最后一个包，它会严格检查：

- `iq_count = 476`
- 后续 padding 必须全部为 0

## 9. Phase D 骨架模块当前状态

### 9.1 Storage

- `CpiContext`：承载一个 CPI 的当前上下文。
- `CpiContextPool`：提供上下文池。
- `SlotWriter`：按协议位置写入 slot。
- `ProgressTracker`：推进进度并设置 trigger bits。

### 9.2 Admission

- `CpiAdmission` 负责判断当前包是否允许进入活动 CPI 上下文。

### 9.3 Finalize

- `CpiFinalizer` 当前支持在 `TriggerFullReady` 和 `TriggerCpiSwitch` 条件下尝试 finalize。
- 这仍是热路径骨架，不应对外宣称已经完成最终产物写出闭环。

## 10. Metrics 与总结

当前 `MetricsCollector` 和 `RunSummary` 重点记录：

- `raw_rx_packets` / `raw_rx_bytes`
- `filtered_packets`
- `rx_packets`
- `parsed_packets`
- `control_table_packets`
- `data_packets`
- `dropped_packets`
- `captured_packets`
- `cpi_count` / `prt_count` / `channel_count`
- `complete_prt_count`
- `final_tail_packets`

程序结束后会输出：

- 一行机器可读摘要
- 中文 `human_summary`

若 `run_until_stopped = true`，还会按 `status_interval_seconds` 输出 `[status]` 快照。

## 11. 外部 sender 与真实联调

当前仓库里应区分两类 sender：

- `tools/raw_eth_sender.py`
  - 只适合验证原始收包或过滤路径。
  - 不保证通过当前协议解析。
- `tools/rxtech_protocol_sender.py`
  - 会发送控制表包和符合当前 `3 x 9` 默认协议形状的数据包。
  - 适合做真实接收解析联调。

如果接收端使用 `configs/dpdk_single_face.conf`，则外部 sender 至少要保证：

- `src_ip = 172.20.11.222`
- `dst_ip = 172.20.11.100`
- `dst_port = 9999`
- `channels_per_prt = 3`
- `packets_per_channel = 9`

否则流量可能在过滤层或协议层被丢弃。

## 12. 当前验证边界

- Windows 侧只用于读代码、改代码、改文档。
- 权威构建、测试、运行验证必须在 Linux 服务器完成。
- 本轮 Phase 3 收尾内容已经在 `ssh kds` 的隔离目录完成复验：构建通过，unit tests 11/11，integration tests 1/1，协议 sender 脚本语法检查通过。

## 13. 结论

当前接收端的真实现状可以概括为：

> `rx_tech_demo` 当前已经形成一条以 DPDK 为主入口、以 UDP payload 组装和协议解析为核心、以 capture 落盘和语义化统计为输出、并已接入 Phase 3 CPI 热路径骨架的 Linux-only 接收端主线。

它已经不是旧文档中描述的 AF_XDP benchmark 主线，也还不是最终完整业务接收模块。对它的描述应始终保持这两个边界。
