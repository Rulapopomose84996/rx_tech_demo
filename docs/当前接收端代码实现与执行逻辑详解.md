### Phase 3 新增核心组件说明

#### 1. PacketPipeline (`src/receiver/protocol/packet_pipeline.h`)
- **职责**：封装完整的包处理流水线，统一调度 Parser、Validator、Interpreter、Assembler
- **输入**：PacketDesc（原始数据包描述）
- **输出**：ProcessedPacket（包含 udp_frame、parsed、interpreted 的完整视图）
- **优势**：将分散的处理逻辑聚合为单一流水线，便于测试和替换

#### 2. DataOrderTracker (`src/receiver/protocol/data_order_tracker.h`)
- **职责**：监控数据包的顺序完整性，检测乱序、丢包等异常
- **功能**：
  - 记录期望的下一个 CPI/PRT/channel/packet_index
  - 比对实际接收包与期望值
  - 统计检查过的包数量、失配次数
  - 在 RunSummary 中填充顺序验证信息
- **优势**：独立于热路径的顺序验证逻辑，不阻塞核心处理流程

#### 3. CpiStateCoordinator (`src/receiver/core/internal/cpi_state_coordinator.h`)
- **职责**：CPI 状态管理中枢，协调 admission、context pool、writer、progress、finalizer
- **功能**：
  - 处理数据包的 CPI 准入决策
  - 管理活动 CPI 上下文的开启和释放
  - 触发 finalize 判定
  - 返回 CpiProcessResult 告知调用者是否接受该包
- **优势**：将复杂的状态管理逻辑从 OwnerLoop 剥离，主循环只负责调用

#### 4. CaptureSink (`src/receiver/output/internal/capture_sink.h`)
- **职责**：格式化输出处理后的数据包到 capture 文件
- **功能**：
  - 写入 capture_packets.bin（二进制载荷）
  - 写入 capture_index.csv（语义索引）
  - 批量刷新优化 I/O 性能
- **优势**：输出逻辑独立，便于扩展多种输出格式

#### 5. RuntimeStatusReporter (`src/receiver/sidecar/internal/runtime_status_reporter.h`)
- **职责**：周期性状态报告和最终总结生成
- **功能**：
  - 按 status_interval_seconds 周期重绘中文状态面板
  - 构建 RunSummary（整合 Metrics、DataOrderTracker、OwnerLoopRuntimeState）
  - 发送反馈快照（feedback snapshot）
  - 渲染最终人类可读的总结报告
- **优势**：监控报告逻辑完全独立于热路径，符合 Sidecar 模式

## 3. 启动与配置流程

当前启动主流程在：

- `src/receiver/app/main_dpdk.cpp`
- `src/receiver/app/run_app.cpp`
- `src/receiver/app/cli_args.cpp`

实际顺序是：

1. 解析 CLI
2. 加载默认配置
3. 若提供 `--config`，加载 section 化配置文件
4. 强制把 `backend_name` 设为当前入口对应后端，这里是 `dpdk`
5. 校验 capture / raw_record / protocol / log 关键字段
6. `--dry-run` 时打印展开后的最终配置
7. 非 dry-run 时创建 backend、metrics，并进入 `ReceiveRunner`

当前示例配置的重点 section 为：

- `[capture]`
- `[raw_record]`
- `[network]`
- `[dpdk]`
- `[runtime]`
- `[protocol]`
- `[log]`
- `[feedback]`

## 4. ReceiveRunner 做什么

`src/receiver/runtime/receive_runner.cpp` 负责运行期框架动作：

- 安装 `SIGINT` / `SIGTERM` 停止处理
- 初始化 backend
- 准备 capture 输出目录
- 打开 `capture_packets.bin` 和 `capture_index.csv`
- 启动 `RawFrameRecorder`
- 调用 `OwnerLoop::run()`（**注意：OwnerLoop 现在只负责创建组件和协调流程，不包含具体业务逻辑**）
- 收集 `CaptureArtifacts`
- 补齐 `RunSummary`（**由 RuntimeStatusReporter 完成**）
- 关闭 recorder 和 backend

当前 capture index 表头是：

```
cpi,channel,prt,packet_index,packet_kind,payload_len,valid
```

## 5. Owner Loop 的真实热路径

### Phase 3 重构后的主循环架构

真正的主循环在 `src/receiver/core/owner_loop.cpp`。经过 Phase 3 重构，代码量从 1000+ 行缩减至约 120 行。

**主循环职责**：
- 创建和初始化所有模块化组件
- 执行收包循环
- 委托具体业务给子组件处理
- 调用 RuntimeStatusReporter 进行周期性状态报告
- 构建最终总结并渲染

### 当前热路径流程

```
OwnerLoop::run() 创建组件
  -> PacketPipeline (流水线)
  -> CpiStateCoordinator (CPI 状态协调)
  -> DataOrderTracker (顺序追踪)
  -> CaptureSink (数据输出)
  -> RuntimeStatusReporter (状态报告)

收包循环开始:
DpdkIngress.recv_burst
  -> for each packet:
       -> PacketPipeline.process_packet()
            -> UdpPayloadAssembler.push (提取 UDP payload)
            -> matches_packet_filter (过滤)
            -> PacketParser.parse (解析)
            -> PacketValidator.validate (校验)
            -> ProtocolSequenceInterpreter.interpret (序列解释)
       -> on_packet 回调:
            -> DataOrderTracker.observe (顺序监控)
            -> CpiStateCoordinator.process_data_packet (CPI 准入)
                 -> CpiAdmission (准入决策)
                 -> SlotWriter (写入 slot)
                 -> ProgressTracker (推进进度)
                 -> CpiFinalizer (finalize 判定)
            -> CaptureSink.write (格式化输出)
       -> MetricsCollector.on_burst (指标收集)
  -> RuntimeStatusReporter.emit_periodic (周期性状态报告)
收包循环结束

-> CaptureSink.flush (刷新缓冲区)
-> RuntimeStatusReporter.build_final_summary (构建最终总结)
-> RuntimeStatusReporter.render_final (渲染最终报告)
-> CpiStateCoordinator.release_active (释放活动上下文)
```

### 组件协作关系

```
OwnerLoop (协调器)
  ├─ PacketPipeline (协议处理)
  │   ├─ UdpPayloadAssembler
  │   ├─ PacketParser
  │   ├─ PacketValidator
  │   └─ ProtocolSequenceInterpreter
  ├─ CpiStateCoordinator (状态管理)
  │   ├─ CpiAdmission
  │   ├─ SlotWriter
  │   ├─ ProgressTracker
  │   └─ CpiFinalizer
  ├─ DataOrderTracker (顺序验证)
  ├─ CaptureSink (输出格式化)
  └─ RuntimeStatusReporter (监控报告)
      ├─ StatusPanelWriter
      └─ MetricsCollector
```

## 8. 当前协议模型

### 8.1 ProtocolSpec

`include/rxtech/protocol_spec.h` 当前集中定义协议形状：

- 控制表 magic：`0x55AAFF00`
- 数据包 magic：`0x55AAFF03`
- 尾包 tail：`0x55AAFF30`
- 头部长度：16 字节
- UDP 包长：2048 字节
- 数据负载长度：2032 字节
- 每 PRT 3 个通道
- 每通道 9 个包
- 前 8 包每包 508 个 IQ
- 最后 1 包 476 个 IQ，后续补零必须全为 0

### 8.2 Parser / Validator / Interpreter

当前协议相关实现分别位于：

- `src/receiver/protocol/sample_packet_parser.cpp`
- `src/receiver/protocol/sample_packet_validator.cpp`
- `src/receiver/protocol/protocol_sequence_interpreter.cpp`

这三层共同完成：

- 解析 16 字节协议头
- 校验 channel / packet_index / tail / payload 长度
- 按 `ProtocolSpec` 给出最终序列语义

## 10. Metrics 与总结

### 10.1 MetricsCollector 指标收集

当前 `MetricsCollector` 重点记录：

- `raw_rx_packets` / `raw_rx_bytes`
- `filtered_packets`
- `rx_packets`
- `parsed_packets`
- `control_table_packets`
- `data_packets`
- `dropped_packets`
- `captured_packets`
- `cpi_count` / `prt_count` / `channel_count`
- `active_cpi` / `active_prt` / `active_prt_channels`
- `complete_prt_count`
- `final_tail_packets`
- raw_record 相关统计

### 10.2 DataOrderTracker 顺序验证指标

新增的顺序追踪器提供：

- `checked_packets`：已检查的包数量
- `matches_expected`：是否匹配期望顺序
- `first_mismatch`：首次失配的位置描述
- `channel_batched_`：是否按通道批量到达

### 10.3 OwnerLoopRuntimeState 运行时状态

新增的运行时状态管理：

- `run_status` / `run_error`：运行状态和错误信息
- `protocol_packets` / `data_packets` / `captured_packets`：各阶段包计数
- `filtered_packets`：过滤掉的包数量
- CPI/PRT/Channel 活动状态追踪

### 10.4 RuntimeStatusReporter 状态报告

**周期性状态报告** (`emit_periodic`)：
- 按 `status_interval_seconds` 周期重绘中文状态面板
- 整合 Metrics、DataOrderTracker、OwnerLoopRuntimeState 生成 RunSummary
- 发送反馈快照（每 `feedback_interval_` 秒）

**最终总结生成** (`build_final_summary`)：
- 计算运行时长
- 汇总所有指标
- 填充人类可读的 summary 字符串
- 包含数据顺序验证结果

**渲染最终报告** (`render_final`)：
- 输出中文 `human_summary`
- 显示详细的统计表格
- 标记任何检测到的异常（如乱序）

### 10.5 状态面板显示逻辑

程序结束后会输出：

- 中文 `human_summary`
- 若 `run_until_stopped = true`，按 `status_interval_seconds` 重绘状态面板
- 未见业务协议流量时，状态面板只显示统一的链路层单面板
- 一旦见到业务协议流量，状态面板切换为协议层 + 结果层扩展视图
- 结果层中的 `channel_count` 是全局去重结果；若要判断某个当前 PRT 是否仍在接收中，应看 `active_prt_channels` 对应的逐通道覆盖
- **新增**：显示数据顺序验证状态（是否失配、首次失配位置）

## 12. Phase 3 架构重构总结

### 12.1 重构目标

针对承担过多职责的臃肿主循环（超过 1000 行），采用**按职责分层**的策略进行组件拆分：

1. **核心协调器精简**：主循环文件应仅保留流程控制逻辑，委托具体业务给子组件，目标是将主函数代码量缩减至约 200 行以内 ✅ **已完成（120 行）**
2. **数据处理流水线**：独立封装解析、校验、解释等热路径逻辑 ✅ **已完成（PacketPipeline）**
3. **状态管理**：将上下文池、准入控制、进度追踪等状态维护逻辑独立 ✅ **已完成（CpiStateCoordinator）**
4. **辅助逻辑隔离**：将数据落地（写文件）、顺序追踪、监控报告等逻辑拆分为独立模块 ✅ **已完成（CaptureSink、DataOrderTracker、RuntimeStatusReporter）**

### 12.2 重构收益

- ✅ **单一职责原则**：每个组件职责清晰，无交叉耦合
- ✅ **可测试性提升**：独立模块便于单元测试和 Mock
- ✅ **可理解性提升**：代码结构清晰，新成员可快速上手
- ✅ **模块可替换性**：可单独替换某个组件而不影响其他部分
- ✅ **性能优化**：热路径逻辑精简，避免不必要的分支判断

### 12.3 代码量对比

| 模块 | 重构前 | 重构后 | 变化 |
|------|--------|--------|------|
| `owner_loop.cpp` | ~1000 行 | ~120 行 | -88% |
| 协议处理 | 分散在多处 | PacketPipeline 统一调度 | 聚合 |
| 状态管理 | 嵌入主循环 | CpiStateCoordinator 独立 | 解耦 |
| 监控报告 | 混杂在热路径 | RuntimeStatusReporter Sidecar 模式 | 分离 |
| 输出格式化 | 直接写入 | CaptureSink 封装 | 抽象 |
