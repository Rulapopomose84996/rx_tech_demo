# 当前接收端代码实现与执行逻辑详解

## 1. 文档目的

本文档描述接收端实现细节，该系统基于 DPDK 数据平面构建了一条稳定的 Linux-only 高性能雷达数据接收链路。系统具备模块化架构、CPI 异步输出流水线和动态 PRT 支持等核心能力。

**系统特性**：

- **DPDK 收包**: 基于 DPDK 后端进行批量收包和 ARP 应答，绕过内核协议栈以提升性能
- **协议解析**: 支持 IPv4/UDP payload 提取、IP 分片重组、协议头校验及序列解释
- **动态 PRT**: 从控制包动态获取每 CPI 的 PRT 数量，支持 provisional→control 收敛绑定
- **数据落盘**: 将有效包写入二进制文件 (`capture_packets.bin`) 并生成语义索引 (`capture_index.csv`)
- **原始帧记录**: 可选异步记录原始接收帧到机械盘 (`raw_frames/*.rawbin`)
- **实时监控**: 持续接收模式下按周期重绘中文状态面板，展示链路层、协议层及结果层统计
- **配置管理**: 支持 section 化配置文件和 CLI 参数覆盖
- **异步输出流水线**: 基于 lock-free SPSC ring、原子化 CPI 上下文池及后台消费线程构建
- **数据顺序验证**: DataOrderTracker 监控包顺序完整性，检测乱序和丢包

### 2.3 核心组件说明

#### PacketPipeline (`src/receiver/protocol/packet_pipeline.{h,cpp}`)
- **职责**：封装完整的包处理流水线，统一调度 UdpPayloadAssembler、Parser、Validator、Interpreter
- **输入**：PacketDesc（原始数据包描述）
- **输出**：ProcessedPacket（包含 udp_frame、parsed、interpreted 的完整视图）

#### DataOrderTracker (`src/receiver/protocol/data_order_tracker.{h,cpp}`)
- **职责**：监控数据包的顺序完整性，检测乱序、丢包等异常
- **功能**：记录期望的下一个 CPI/PRT/channel/packet_index，比对实际接收包，统计失配次数

#### CpiStateCoordinator (`src/receiver/core/internal/cpi_state_coordinator.h`)
- **职责**：CPI 状态管理中枢，协调 admission、context pool、writer、progress、finalizer
- **核心机制**：持有 CpiContextPool（原子化 4-slot 池），通过 SpscRing 连接输出流水线
- **双窗口支持**：维护 `previous_ctx_` / `previous_ctx_index_`，CPI 切换时保留上一个上下文以容忍迟到包
- **动态 PRT 绑定**：支持三态绑定源（`BindSource::fixed` / `provisional` / `control`），控制包到达后可从 provisional 收敛到 control
- **功能**：
  - 处理数据包的 CPI 准入决策（委托 CpiAdmission）
  - 管理活动 CPI 上下文的 pool acquire/release
  - 双窗口：CPI 切换时将 active 挪到 previous，允许迟到包写入 previous
  - `process_control_packet()` 解析控制包携带的 n_prt，按动态 PRT 规则更新 `current_snapshot_`
  - `open_active()` 首包为数据包时赋 provisional 绑定（上限 `max_n_prt`），控制包到达后可收敛
  - finalize 后推送 CpiOutput 到 output_ring
  - drain_recycle() 从 recycle_ring 回收 ReleaseToken 并释放池位
  - `finalize_active_for_shutdown()` 在收包循环结束后顺序 finalize previous 和 active 上下文（TriggerStop）

#### CaptureSink (`src/receiver/output/internal/capture_sink.h`)
- **职责**：格式化输出处理后的数据包到 capture 文件
- **功能**：写入 capture_packets.bin（二进制载荷）和 capture_index.csv（语义索引），批量刷新优化 I/O

#### RuntimeStatusReporter (`src/receiver/sidecar/internal/runtime_status_reporter.h`)
- **职责**：周期性状态报告和最终总结生成
- **功能**：按 status_interval_seconds 周期重绘中文状态面板，构建 RunSummary，渲染最终报告

#### CpiConsumer (`src/receiver/output/cpi_consumer.cpp`)
- **后台消费线程**：从 output_ring 弹出 CpiOutput，调用用户 handler 回调，推送 ReleaseToken 到 recycle_ring
- **生命周期**：由 OwnerLoop 在主循环前启动、循环结束后 join

#### SpscRing (`include/rxtech/spsc_ring.h`)
- **lock-free 单生产者/单消费者环形缓冲区模板**：原子操作 + 缓存行对齐（64 字节），memory_order_acquire/release 同步
- **容量语义**：min_capacity → 内部上取 power-of-two → 可用 slot = (power_of_two - 1)

#### CpiContextPool (`include/rxtech/cpi_context_pool.h`)
- **4-slot 上下文池**：使用 `std::atomic<bool>` CAS 操作实现 lock-free acquire/release
- **关键约束**：CpiContext 体量约 16 MB，必须堆分配（栈分配会导致服务器端栈溢出）

## 5. Owner Loop 的真实热路径

### 主循环架构

真正的主循环在 `src/receiver/core/owner_loop.cpp`（约 155 行）。

**主循环职责**：
- 创建和初始化所有模块化组件
- 创建 CPI 输出流水线的两个 SpscRing（output_ring + recycle_ring）
- 启动 CpiConsumer 后台线程
- 执行收包循环，委托具体业务给子组件
- 调用 RuntimeStatusReporter 进行周期性状态报告
- 循环结束后 join 消费线程、构建最终总结

### 当前热路径流程

```
OwnerLoop::run() 初始化阶段
  -> 创建 PacketPipeline、CpiStateCoordinator、DataOrderTracker、CaptureSink
  -> 创建 SpscRing<CpiOutput> output_ring (capacity=8)
  -> 创建 SpscRing<ReleaseToken> recycle_ring (capacity=8)
  -> CpiStateCoordinator.attach_rings(output_ring, recycle_ring)
  -> 启动 CpiConsumer 后台线程 (consumer_thread)
  -> 创建 RuntimeStatusReporter

收包循环:
  DpdkIngress.recv_burst
    -> for each packet:
         -> RawFrameRecorder.submit (如启用)
         -> PacketPipeline.process_packet()
              -> UdpPayloadAssembler.push (提取 UDP payload)
              -> matches_packet_filter (过滤)
              -> PacketParser.parse (解析)
              -> PacketValidator.validate (校验)
              -> ProtocolSequenceInterpreter.interpret (序列解释)
         -> on_packet 回调:
              -> control_table: CpiStateCoordinator.process_control_packet
                   -> 动态 PRT：解析 n_prt，更新 current_snapshot_
                   -> 若 active 已 provisional 绑定 → 尝试收敛到 control
              -> data_packet:
                   -> DataOrderTracker.observe (顺序监控)
                   -> CpiStateCoordinator.process_data_packet
                        -> CpiAdmission.judge (准入决策)
                        -> open_active: CpiContextPool.acquire (原子池分配)
                             -> dynamic_prt: 赋予 provisional 绑定 (n_prt=max_n_prt)
                        -> TRIGGER_CPI_SWITCH: active 挪入 previous，开新 active
                        -> WRITE_ACTIVE: SlotWriter.write (写入 slot)
                        -> ProgressTracker.advance (推进进度)
                        -> finalize_active (如触发):
                             -> CpiFinalizer.try_finalize → CpiOutput
                             -> output_ring.push(CpiOutput) [lock-free]
                        -> 非 WRITE_ACTIVE: 尝试写入 previous_ctx_ (双窗口中次包容忍)
                             -> SlotWriter.write(*previous_ctx_, parsed)
                             -> ProgressTracker.advance(*previous_ctx_, ...)
              -> CaptureSink.write (格式化输出)
         -> MetricsCollector.on_burst (指标收集)
    -> backend.release_burst

    -> CpiStateCoordinator.check_timeout (T-004 超时)
    -> CpiStateCoordinator.drain_recycle [从 recycle_ring 回收池位]
    -> RuntimeStatusReporter.emit_periodic (周期性状态报告)

收包循环结束:
  -> CaptureSink.flush
  -> CpiStateCoordinator.finalize_active_for_shutdown (*context.metrics)
       -> finalize_previous(TriggerStop)  [先 finalize 上一个 CPI]
       -> finalize_active(TriggerStop)    [再 finalize 当前 CPI]
  -> consumer_stop = true; consumer_thread.join()
  -> CpiStateCoordinator.drain_recycle (排空剩余回收令牌)
  -> RuntimeStatusReporter.build_final_summary
  -> RuntimeStatusReporter.render_final (渲染最终报告)
```

### CPI 输出流水线

```
Producer (主线程 / OwnerLoop)          Consumer (后台线程 / CpiConsumer)
─────────────────────────────          ─────────────────────────────────

CpiStateCoordinator
  .finalize_active()
    ├─ CpiFinalizer.try_finalize()
    │   → 生成 CpiOutput
    └─ output_ring.push(CpiOutput)  ─→  output_ring.pop()
                                           ├─ handler(CpiOutput) [用户回调]
                                           └─ recycle_ring.push(ReleaseToken)
drain_recycle()  ←────────────────────
  ├─ recycle_ring.pop()
  └─ CpiContextPool.release(index)
```

**关键设计**：
- output_ring 和 recycle_ring 均为 SpscRing，lock-free 无锁通信
- CpiContextPool 使用 CAS 原子操作分配/释放 4 个 ~16 MB 的上下文 slot
- 池位在 push 到 output_ring 后保持占用，直到消费线程返回 ReleaseToken
- output_ring 满时 backpressure 策略：丢弃并立即释放池位

### 组件协作关系

```
OwnerLoop (协调器, ~155 行)
  ├─ PacketPipeline (协议处理)
  │   ├─ UdpPayloadAssembler
  │   ├─ PacketParser
  │   ├─ PacketValidator
  │   └─ ProtocolSequenceInterpreter
  ├─ CpiStateCoordinator (状态管理 + 输出流水线)
  │   ├─ CpiContextPool (原子化 4-slot 池)
  │   ├─ CpiAdmission + RecentClosedRing
  │   ├─ SlotWriter

  │   ├─ SpscRing<CpiOutput> output_ring ──→ CpiConsumer (后台线程)
```

## 8. 当前协议模型

### 8.1 ProtocolSpec

`include/rxtech/protocol_spec.h` 当前集中定义协议形状：

- 控制表 magic：`0x55AAFF00`
- 数据包 magic：`0x55AAFF03`
- 尾包 tail：`0x55AAFF30`
- 头部长度：16 字节
- UDP 包长：2048 字节
- 数据负载长度：2032 字节
- 每 PRT 3 个通道
- 每通道 9 个包
- 前 8 包每包 508 个 IQ
- 最后 1 包 476 个 IQ，后续补零必须全为 0
- **`dynamic_prt_enabled`**（默认 `true`）：开启后从控制包动态获取每 CPI 的 PRT 数量
- **`expected_n_prt`**（默认 `0`）：静态模式下固定 PRT 数量；动态模式下作为回退值
- **`max_n_prt`**（默认 `100`）：动态模式下 n_prt 的合法上限，超出则回退到 `expected_n_prt`

对应的 `RxConfig` 配置字段：

- `protocol_dynamic_prt_enabled`（默认 `true`）
- `protocol_max_n_prt`（默认 `100`）

### 8.2 Parser / Validator / Interpreter

当前协议相关实现分别位于：

- `src/receiver/protocol/sample_packet_parser.cpp`
- `src/receiver/protocol/sample_packet_validator.cpp`
- `src/receiver/protocol/protocol_sequence_interpreter.cpp`

这三层共同完成：

- 解析 16 字节协议头
- 校验 channel / packet_index / tail / payload 长度
- 按 `ProtocolSpec` 给出最终序列语义

## 9. CPI 热路径骨架

### 9.1 Storage

- `CpiContext`（定长连续缓冲区，有效负载约 4.5 MB）
  - `CpiHotHeader`：64 字节对齐状态机（CpiState、trigger_bits、IQ 计数、时间戟，包含 `observed_n_prt`）
  - `BoundWaveSnapshotLite`：波形绑定元数据，包含 `bind_source`（BindSource: fixed / provisional / control）和 `conflict` 字段
  - `prt_summary` 数组：逐 PRT 完成跟踪
  - `slot_valid_bytes`：逐 slot 有效性
  - `payload`：2048 字节步幅的 IQ 数据 slot（最大 `kCpiMaxTotalSlots = kCpiPrtMax(64) × kCpiMaxSlotsPerPrt(36) = 2304` 个 slot）
- `BindSource` 枚举（`cpi_context.h`）：
  - `fixed`：使用静态 `expected_n_prt`
  - `provisional`：数据包先到，用 `max_n_prt` 临时占位
  - `control`：控制包到达后确认绑定
- `CpiContextPool`（4-slot 池，原子 CAS acquire/release）
- `SlotWriter`（写入 slot、检测重复、V-006 PRT 范围校验）
- `ProgressTracker`（推进 PRT/channel 完成标志、触发判定：T-003 FullReady / T-005 WaveEnd / TailObserved）

### 9.2 Admit

- `CpiAdmission`：根据当前包的 CPI ID 与活动上下文、RecentClosedRing 判定准入
  - `WRITE_ACTIVE`：属于活动 CPI
  - `TRIGGER_CPI_SWITCH`：下一个 CPI（CPI+1）—— active 挪入 previous，开新 active
  - `LATE_TO_CLOSED`：属于已 finalize 的 CPI——尝试写入 previous_ctx_（双窗口容忍）
  - `DROP`：无效或乱序（丢弃）
- **双窗口机制**：CPI 切换时 previous_ctx_ 保留上一个 CPI 上下文，非 WRITE_ACTIVE 的包若 cpi_id 匹配 previous 则允许写入

### 9.3 Finalize

- `CpiFinalizer::try_finalize(ctx, trigger)` 评估触发类型：
  - `TriggerFullReady` → `COMPLETE_OK`（所有 slot 已填满）
  - `TriggerCpiSwitch | TriggerWaveEnd` → `INCOMPLETE_BUT_COMMITTABLE` 或 `ABNORMAL_CUTOFF_COMMIT`
  - `TriggerTimeout | TriggerStop` → 返回 nullopt（静默丢弃）
- `finalize_active_for_shutdown(metrics)`：在收包循环结束后顺序调用
  - 先 `finalize_previous(TriggerStop)` 处理双窗口中上一个 CPI
  - 再 `finalize_active(TriggerStop)` 处理当前活动 CPI
  - 确保关机前 CPI 输出路径完整
- 输出 `CpiOutput`（cpi_id、decision、received_slot_count、timing、CpiReadOnlyView、pool_index）
- 上下文状态转移为 `SEALED`

### 9.4 Output Pipeline

- `SpscRing<CpiOutput>` output_ring：主线程 push，消费线程 pop
- `CpiConsumer`：后台线程 busy-wait 弹出 CpiOutput，调用用户 handler 回调，推送 ReleaseToken 到 recycle_ring
- `SpscRing<ReleaseToken>` recycle_ring：消费线程 push，主线程 drain
- `ReleaseToken`（output_id + ctx_pool_index）：告知主线程可安全释放池位
- 主线程在每轮收包循环末尾调用 `drain_recycle()` 执行池位回收
- `CpiConsumer.processed_count_` 使用 `std::atomic<uint64_t>`，避免 `-O2` 下的数据竞争

## 10. Metrics 与总结

### 10.1 MetricsCollector 指标收集

当前 `MetricsCollector` 重点记录：

- `raw_rx_packets` / `raw_rx_bytes`
- `filtered_packets`
- `rx_packets`
- `parsed_packets`
- `control_table_packets`
- `data_packets`
- `dropped_packets`
- `captured_packets`
- `cpi_count` / `prt_count` / `channel_count`
- `active_cpi` / `active_prt` / `active_prt_channels`
- `complete_prt_count`
- `final_tail_packets`
- raw_record 相关统计

### 10.2 DataOrderTracker 顺序验证指标

顺序追踪器提供：

- `checked_packets`：已检查的包数量
- `matches_expected`：是否匹配期望顺序
- `first_mismatch`：首次失配的位置描述
- `channel_batched_`：是否按通道批量到达

### 10.3 OwnerLoopRuntimeState 运行时状态

运行时状态管理：

- `run_status` / `run_error`：运行状态和错误信息
- `protocol_packets` / `data_packets` / `captured_packets`：各阶段包计数
- `filtered_packets`：过滤掉的包数量
- CPI/PRT/Channel 活动状态追踪

### 10.4 RuntimeStatusReporter 状态报告

**周期性状态报告** (`emit_periodic`)：
- 按 `status_interval_seconds` 周期重绘中文状态面板
- 整合 Metrics、DataOrderTracker、OwnerLoopRuntimeState 生成 RunSummary

**最终总结生成** (`build_final_summary`)：
- 计算运行时长
- 汇总所有指标
- 填充人类可读的 summary 字符串
- 包含数据顺序验证结果

**渲染最终报告** (`render_final`)：
- 输出中文 `human_summary`
- 显示详细的统计表格
- 标记任何检测到的异常（如乱序）

### 10.5 状态面板显示逻辑

程序结束后会输出：

- 中文 `human_summary`
- 若 `run_until_stopped = true`，按 `status_interval_seconds` 重绘状态面板
- 未见业务协议流量时，状态面板只显示统一的链路层单面板
- 一旦见到业务协议流量，状态面板切换为协议层 + 结果层扩展视图
- 结果层中的 `channel_count` 是全局去重结果；若要判断某个当前 PRT 是否仍在接收中，应看 `active_prt_channels` 对应的逐通道覆盖
- 显示数据顺序验证状态（是否失配、首次失配位置）

## 12. 结论

当前接收端的真实现状可以概括为：

> `rx_tech_demo` 当前已经形成一条以 DPDK 为唯一入口、以 UDP payload 组装和协议解析为核心、以 capture/raw_record 落盘和语义化统计为输出、已完成**模块化架构** + **CPI 异步输出流水线** + **动态 PRT 支持**的 Linux-only 接收端主线。

### 已落地能力

1. **模块化架构**：主循环精简至 ~155 行，5 个职责清晰的独立模块（PacketPipeline、CpiStateCoordinator、DataOrderTracker、CaptureSink、RuntimeStatusReporter）
2. **完整协议解析链**：UdpPayloadAssembler → PacketParser → PacketValidator → ProtocolSequenceInterpreter
3. **CPI 生命周期管理**：准入 (Admission) → 写入 (SlotWriter) → 进度 (ProgressTracker) → finalize(CpiFinalizer)
4. **异步输出流水线**：lock-free SpscRing + 原子 CpiContextPool(4-slot) + CpiConsumer 后台消费线程 + ReleaseToken 回收机制
5. **动态 PRT 支持**：双窗口 previous_ctx_、provisional→control 收敛绑定、`finalize_active_for_shutdown()` 干净关机路径
6. **运行时监控**：中文状态面板、数据顺序验证、最终总结报告
7. **原始帧录制**：后台写入线程、文件轮转、ringbuffer 缓冲

### 当前边界

- 它已经不是 AF_XDP benchmark 主线
- 还不是最终完整业务接收模块
- 外部发送端的真实闭环网络验证依赖服务器环境和发送工具就位
- 文档描述必须保持这两个边界
