# 当前接收端代码实现与执行逻辑详解

## 1. 文档目的

本文档只描述当前仓库已经落地的接收端实现，不把历史 AF_XDP 路径、旧目录结构或未实现规划写成既成事实。

当前唯一主线位于 `src/receiver`，并以 DPDK 为入口。

## 2. 当前源码结构

| 目录 | 作用 |
|---|---|
| `include/rxtech` | 当前唯一公共头入口 |
| `src/receiver/app` | 可执行入口、CLI 解析、启动装配 |
| `src/receiver/runtime` | 配置、运行器、停止控制、raw_record |
| `src/receiver/ingress/dpdk` | DPDK 收包与 ARP 应答 |
| `src/receiver/protocol` | UDP payload 提取、解析、校验、序列解释 |
| `src/receiver/core` | owner loop、状态面板、汇总辅助 |
| `src/receiver/sidecar` | 指标与 `RunSummary` |
| `src/receiver/storage` | CPI 上下文、slot 写入、进度推进 |
| `src/receiver/admit` | CPI 准入决策 |
| `src/receiver/finalize` | CPI finalize 和 finalized output 结构 |

当前主可执行文件是：

- `rx_receiver_dpdk`
- `rxbench_dpdk`

它们从 `app/main_dpdk.cpp` 进入统一的 `run_app("dpdk", argc, argv)`。

## 3. 启动与配置流程

当前启动主流程在：

- `src/receiver/app/main_dpdk.cpp`
- `src/receiver/app/run_app.cpp`
- `src/receiver/app/cli_args.cpp`

实际顺序是：

1. 解析 CLI
2. 加载默认配置
3. 若提供 `--config`，加载 section 化配置文件
4. 强制把 `backend_name` 设为当前入口对应后端，这里是 `dpdk`
5. 校验 capture / raw_record / protocol / log 关键字段
6. `--dry-run` 时打印展开后的最终配置
7. 非 dry-run 时创建 backend、metrics，并进入 `ReceiveRunner`

当前示例配置的重点 section 为：

- `[capture]`
- `[raw_record]`
- `[network]`
- `[dpdk]`
- `[runtime]`
- `[protocol]`
- `[log]`
- `[feedback]`

## 4. ReceiveRunner 做什么

`src/receiver/runtime/receive_runner.cpp` 负责运行期框架动作：

- 安装 `SIGINT` / `SIGTERM` 停止处理
- 初始化 backend
- 准备 capture 输出目录
- 打开 `capture_packets.bin` 和 `capture_index.csv`
- 启动 `RawFrameRecorder`
- 调用 `OwnerLoop::run()`
- 收集 `CaptureArtifacts`
- 补齐 `RunSummary`
- 关闭 recorder 和 backend

当前 capture index 表头是：

```text
cpi,channel,prt,packet_index,packet_kind,payload_len,valid
```

## 5. Owner Loop 的真实热路径

真正的主循环在 `src/receiver/core/owner_loop.cpp`。

当前热路径可以概括成：

```text
DpdkIngress.recv_burst
  -> UdpPayloadAssembler.push
  -> matches_packet_filter
  -> PacketParser.parse
  -> PacketValidator.validate
  -> ProtocolSequenceInterpreter.interpret
  -> CpiAdmission
  -> SlotWriter
  -> ProgressTracker
  -> CpiFinalizer
  -> MetricsCollector / capture / summary
```

其中辅助职责已拆分到：

- `src/receiver/core/status_panel.cpp`
- `src/receiver/core/owner_loop_summary.cpp`
- `src/receiver/core/internal/*`

## 6. DPDK ingress 当前职责

`src/receiver/ingress/dpdk/dpdk_backend.cpp` 当前负责：

- DPDK EAL 初始化
- 端口和 queue 配置
- mbuf pool 创建
- 批量收包
- 原始收包统计
- 按需应答 ARP 请求

它向上层交付统一 `PacketDesc`，协议处理不在 ingress 层完成。

## 7. UDP payload 组装层

`src/receiver/protocol/udp_payload_assembler.cpp` 负责：

- 解析以太网头、IPv4 头、UDP 头
- 提取 UDP payload
- 在需要时做 IPv4 分片重组
- 输出 `UdpPayloadFrame`

## 8. 当前协议模型

### 8.1 ProtocolSpec

`include/rxtech/protocol_spec.h` 当前集中定义协议形状：

- 控制表 magic：`0x55AAFF00`
- 数据包 magic：`0x55AAFF03`
- 尾包 tail：`0x55AAFF30`
- 头部长度：16 字节
- UDP 包长：2048 字节
- 数据负载长度：2032 字节
- 每 PRT 3 个通道
- 每通道 9 个包
- 前 8 包每包 508 个 IQ
- 最后 1 包 476 个 IQ，后续补零必须全为 0

### 8.2 Parser / Validator / Interpreter

当前协议相关实现分别位于：

- `src/receiver/protocol/sample_packet_parser.cpp`
- `src/receiver/protocol/sample_packet_validator.cpp`
- `src/receiver/protocol/protocol_sequence_interpreter.cpp`

这三层共同完成：

- 解析 16 字节协议头
- 校验 channel / packet_index / tail / payload 长度
- 按 `ProtocolSpec` 给出最终序列语义

## 9. CPI 热路径骨架

### 9.1 Storage

- `CpiContext`
- `CpiContextPool`
- `SlotWriter`
- `ProgressTracker`

### 9.2 Admit

- `CpiAdmission` 负责判断当前包是否允许进入活动 CPI 上下文

### 9.3 Finalize

- `CpiFinalizer` 当前负责 finalize 判定和 finalized output 结构返回
- 原 `output` 结构已并入 `include/rxtech/cpi_finalizer.h`

## 10. Metrics 与总结

当前 `MetricsCollector` 和 `RunSummary` 重点记录：

- `raw_rx_packets` / `raw_rx_bytes`
- `filtered_packets`
- `rx_packets`
- `parsed_packets`
- `control_table_packets`
- `data_packets`
- `dropped_packets`
- `captured_packets`
- `cpi_count` / `prt_count` / `channel_count`
- `complete_prt_count`
- `final_tail_packets`
- raw_record 相关统计

程序结束后会输出：

- 中文 `human_summary`
- 若 `run_until_stopped = true`，按 `status_interval_seconds` 重绘状态面板

## 12. 结论

当前接收端的真实现状可以概括为：

> `rx_tech_demo` 当前已经形成一条以 DPDK 为唯一入口、以 UDP payload 组装和协议解析为核心、以 capture/raw_record 落盘和语义化统计为输出、并已接入 CPI 热路径骨架的 Linux-only 接收端主线。

它已经不是 AF_XDP benchmark 主线，也还不是最终完整业务接收模块。文档描述必须保持这两个边界。
