# 当前接收端代码实现与执行逻辑详解

## 1. 文档范围

本文档只描述当前工作区里真实存在、且已经接入 `src/receiver` 主线的实现。

当前接收端的运行主线是：

- 统一 `IRxBackend` 接口
- 统一 `UdpDatagramDesc` 交付模型
- 统一 `UdpDatagramPipeline` 协议处理入口
- `OwnerLoop` 协调的 CPI 生命周期与摘要输出

当前代码里与接收主线直接相关的后端分成三类：

- `DpdkIngress`
  - 当前真实网卡主路径
  - 也是当前 CLI 默认后端 `rx_receiver_dpdk`
- `LinuxSocketIngress`
  - 当前正式接入主线的 Linux UDP datagram 后端
  - 对应 CLI 入口 `rx_receiver_socket`
- `FileReplayBackend`
  - 仅在 `BUILD_REPLAY=ON` 时构建
  - 主要用于测试和回放，不是当前 CLI 正式入口

需要特别区分的一点是：

- 当前运行时主循环直接使用 `UdpDatagramPipeline`
- `PacketPipeline` 仍然保留在代码中，但它只是 legacy frame adapter
- `PacketPipeline` 负责把原始以太帧经 `UdpPayloadAssembler` 转成 datagram，再转交 `UdpDatagramPipeline`
- 当前 `OwnerLoop` 并不走 `PacketPipeline`

## 3. 启动入口与配置生效规则

### 3.1 CLI 入口

当前两个 CLI 入口非常薄：

- `src/receiver/app/main_dpdk.cpp`
- `src/receiver/app/main_socket.cpp`

它们分别调用：

- `run_app("dpdk", argc, argv)`
- `run_app("socket", argc, argv)`

### 3.2 CLI 参数

`src/receiver/app/cli_args.cpp` 当前支持的参数只有：

- `--config FILE`
- `--dry-run`
- `--run-until-stopped`
- `--duration SECONDS`
- `--status-interval SEC`
- `-h` / `--help`

当前没有别的 CLI 覆盖项。

### 3.3 配置生效顺序

`run_app.cpp` 里的当前顺序是：

1. 解析 CLI
2. 从 `load_default_config()` 取得默认配置
3. 若提供 `--config`，用 `load_config_file()` 覆盖默认值
4. 强制把 `process.backend_name` 设为当前入口后端，即 `dpdk` 或 `socket`
5. 再应用 CLI 覆盖：
   - `--run-until-stopped`
   - `--duration`
   - `--status-interval`
6. 调用 `prepare_run_artifact_paths()` 生成 `run_label` 并改写运行产物路径
7. 调用 `validate_config()`
8. 若是 `--dry-run`，打印最终展开配置后直接退出
9. 非 dry-run 时配置结构化日志、创建 backend、创建 `MetricsCollector`
10. 进入 `ReceiveRunner::run()`

### 3.4 当前已解析的配置段

`rx_config.cpp` 当前真正支持解析并写入 `RxConfig` 的配置段主要是：

- `[capture]`
- `[raw_record]`
- `[network]`
- `[socket]`
- `[dpdk]`
- `[runtime]`
- `[protocol]`
- `[log]`
- `[metrics]`

当前 `RxConfig::OperationsConfig` 里虽然还有这些字段：

- `feedback_enabled`
- `feedback_host`
- `feedback_bind_host`
- `feedback_port`
- `feedback_interval_seconds`
- `metrics_detail_enabled`

但当前接收主线没有解析和消费它们，它们只是结构体中的预留字段，不属于当前实际生效链路。

### 3.5 run artifact 路径改写规则

`prepare_run_artifact_paths()` 的当前行为不是“在原目录下再套一层固定子目录”，而是：

- 生成 `process.run_label = 时间戳 + "_" + 后缀`
- 若 capture 输出目录是 `results/socket_loopback`
  - 最终运行目录会变成 `results/<run_label>`
  - 不是 `results/socket_loopback/<run_label>`
- 若启用了 raw record
  - `raw_record_output_dir` 会被改写成 `<raw_record_output_dir>/<run_label>`
- 若普通日志走文件
  - `log_file_path` 会被改写成 `<原日志目录>/<run_label>/<原文件名>`

### 3.6 socket 相关当前校验规则

`validate_config()` 当前对 `backend=socket` 的约束是：

- `socket_bind_ip` 为空时，回退到 `receiver_ipv4`
- `socket_bind_port` 为 `0` 时，回退到 `allowed_dest_port`
- 生效后的 bind IP 不能为空
- 生效后的 bind port 必须大于 `0` 且小于等于 `65535`
- `socket_bind_ip` 若非空且不是 `0.0.0.0`，则不能和 `receiver_ipv4` 冲突
- `socket_bind_port` 若非零，则不能和 `allowed_dest_port` 冲突
- `socket_rcvbuf_bytes` 必须大于 `0`

### 3.7 结构化日志与指标导出

当前 dry-run 和实际运行都覆盖到这些字段：

- `structured_log_output`
- `structured_log_file_path`
- `structured_log_format`
- `log_rate_limit_seconds`
- `metrics_export_mode`
- `metrics_export_path`
- `metrics_export_interval_seconds`

其中：

- `structured_log_output` 仅支持 `disabled/stdout/stderr/file`
- `structured_log_format` 仅支持 `json/text`
- `metrics_export_mode` 仅支持 `none/prometheus_text/json_socket`

## 4. 运行期主链路

### 4.1 从 CLI 到 OwnerLoop 的当前路径

```text
main_dpdk / main_socket
  -> run_app()
     -> parse_cli_args()
     -> load_default_config() / load_config_file()
     -> prepare_run_artifact_paths()
     -> validate_config()
     -> configure_structured_logger()
     -> make_backend()
     -> ReceiveRunner::run()
        -> backend->init()
        -> 打开 capture 文件
        -> 启动 RawFrameRecorder（如果启用）
        -> OwnerLoop::run()
```

### 4.2 ReceiveRunner 当前职责

`src/receiver/runtime/receive_runner.cpp` 当前负责：

- 安装 `SIGINT` / `SIGTERM` 停止处理
- 调用 `backend->init(config)`
- 按配置打开：
  - `capture_packets.bin`
  - `capture_index.csv`
- 即使 `capture_enabled=false`
  - 也仍然会创建内存 `ostringstream`，让 `OwnerLoop` 保持统一写接口
- 若启用 raw record
  - 创建并启动 `RawFrameRecorder`
- 计算 deadline
- 调用 `OwnerLoop::run(context, artifacts, should_stop)`
- 结束后关闭流、填充 `RunSummary.capture.*`
- 最后调用 `backend->shutdown()`

### 4.3 OwnerLoop 当前真实热路径

`src/receiver/core/owner_loop.cpp` 当前初始化的组件是：

- `ProtocolSpec spec = protocol_spec_from_config(config)`
- `UdpDatagramPipeline packet_pipeline(config, spec)`
- `CpiStateCoordinator cpi_state_coordinator(spec)`
- `DataOrderTracker data_order_tracker(spec)`
- `SpscRing<CpiOutput> output_ring`
- `SpscRing<ReleaseToken> recycle_ring`
- `CpiConsumer consumer`
- `RuntimeStatusReporter status_reporter`

注意：

- 当前主循环直接构造 `UdpDatagramPipeline`
- 没有在 `OwnerLoop` 中构造 `PacketPipeline`

### 4.4 主循环执行顺序

当前循环可概括为：

```text
while (!should_stop())
  -> backend->recv_burst(burst, max_burst)
  -> 为本次 burst 建立 release guard，确保函数返回前调用 backend->release_burst()
  -> for each datagram:
       -> cpi_state_coordinator.drain_recycle()
       -> 如果 datagram 带 raw_frame_data/raw_frame_len，则送 RawFrameRecorder
       -> packet_pipeline.process_datagram(...)
            -> 过滤
            -> parser
            -> validator
            -> interpreter
            -> on_packet(lambda)
                 -> record_protocol_packet()
                 -> control_table: process_control_packet()
                 -> data_packet:
                      -> DataOrderTracker.observe()
                      -> process_data_packet()
                      -> 若 CPI 不接受，则直接 return
                      -> record_data_packet()
                 -> record_captured_packet()
                 -> 写 capture_packets.bin
                 -> 写 capture_index.csv
  -> metrics.on_burst(accepted_packets, accepted_bytes)
  -> check_timeout()
  -> drain_recycle()
  -> emit_periodic()
```

### 4.5 “候选业务包”和“已落盘包”不是同一层计数

当前代码里这几类计数要分开理解：

- `summary.protocol.rx_packets`
  - 来自 `MetricsCollector::on_burst()`
  - 表示通过地址/端口过滤后的候选 datagram 数
- `summary.protocol.parsed_packets`
  - 表示 parser + validator + interpreter 都通过的有效协议包
- `summary.capture.packet_count` / `artifacts.recorded_packets`
  - 只有进入 `on_packet` 且没有在 data path 中被 CPI admission 提前 `return` 的包才会落盘
  - 也就是：
    - 控制表包会落盘
    - 被 CPI 接受的数据包会落盘
    - 被 CPI admission 拒绝的数据包不会落盘

因此，当前 `capture_packets.bin` / `capture_index.csv` 记录的是：

- 控制表包
- 被 CPI 主线接受的数据包

而不是“所有 filter 通过的候选业务报文”。

### 4.6 shutdown 路径

主循环退出后当前顺序是：

1. flush `packet_stream` 与 `index_stream`
2. `finalize_active_for_shutdown()`
3. 若输出链路发生退化，根据 `output_drop_policy` 升级运行状态
4. 通知 `CpiConsumer` 停止并 `join`
5. `drain_recycle()`
6. 若存在 `RawFrameRecorder`，再调用一次 `stop()`
7. `build_final_summary()`
8. `render_final()`

## 5. ingress 与 datagram 交付模型

### 5.1 统一交付对象：UdpDatagramDesc

当前主线的 ingress 向上统一交付 `UdpDatagramDesc`，字段包括：

- UDP payload 指针和长度
- 原始帧指针和长度
- 源/目的 IPv4
- 源/目的端口
- steady clock 时间戳 `ts_ns`
- `queue_id`
- `cookie`
- `backend_kind`
- 可选 `global_sequence`
- `truncated`

这意味着当前 protocol 层不再关心底层是 DPDK、socket 还是 file replay，只消费 datagram 描述。

### 5.2 DpdkIngress 当前行为

`src/receiver/ingress/dpdk/dpdk_backend.cpp` 当前负责：

- 组装 EAL 参数
- `rte_eal_init`
- 解析 port id
- 创建 mempool
- 配置 RX/TX queue
- 启动端口
- 初始化本机 MAC 和接收端 IP
- 启动时清空 NIC 上旧报文
- 收包时定期检查链路状态
- 遇到 ARP 请求时构造并发送 ARP reply
- 把以太帧适配成 `UdpDatagramDesc`

`DpdkDatagramAdapter` 的当前特点：

- 只接收非分片 IPv4/UDP
- 若发现 IPv4 分片，直接记为 backend drop，不做重组
- `global_sequence` 取自 IPv4 `identification`
- `raw_frame_data/raw_frame_len` 保留底层 frame 视图
- `backend_kind = BackendKind::dpdk`

### 5.3 LinuxSocketIngress 当前行为

`src/receiver/ingress/socket/linux_socket_backend.cpp` 当前负责：

- `socket(AF_INET, SOCK_DGRAM, 0)`
- 设置 `SO_REUSEADDR`
- 设置 `SO_RCVBUF`
- 阻塞模式下可选设置 `SO_RCVTIMEO`
- 非阻塞模式下设置 `O_NONBLOCK`
- 设置 `IP_PKTINFO`
- 设置 `SO_RXQ_OVFL`
- `bind(bind_ip, bind_port)`
- 使用固定 slot + `recvmmsg()` 批量收 UDP datagram

它当前交付的 datagram 特征是：

- `payload_data` 指向 socket slot 缓冲
- `raw_frame_data == nullptr`
- `raw_frame_len == 0`
- `dst_ipv4_be` 优先来自 `IP_PKTINFO`，否则回退到配置中的目的地址
- `dst_port` 固定写成实际 bind 端口
- `ts_ns = steady_clock_now_ns()`
- `backend_kind = BackendKind::socket`
- `truncated` 由 `MSG_TRUNC` 反映

这也意味着：

- socket 路径当前不提供原始二层帧
- 因此即使启用了 `raw_record_enabled`，socket 主线也不会真正写出 raw frame 文件

### 5.4 FileReplayBackend 当前位置

`FileReplayBackend` 只在 `BUILD_REPLAY=ON` 时构建，当前用途是：

- 从回放目录读取 payload
- 组装成预构建 Ethernet frame
- 在 `recv_burst()` 中按配置节流或直接回放
- 向上交付 `BackendKind::file_replay`

它不是当前 CLI 里的正式 `backend_name` 选项，也不由 `run_app()` 创建。

## 6. 协议处理链

### 6.1 UdpDatagramPipeline 是当前唯一协议主入口

`src/receiver/protocol/udp_datagram_pipeline.cpp` 的当前职责是：

- 对 `UdpDatagramDesc` 做基本合法性检查
- 处理 `truncated` datagram
- 根据配置做源 IP、目的 IP、目的端口过滤
- 调用：
  - `PacketParser`
  - `PacketValidator`
  - `ProtocolSequenceInterpreter`
- 对 reject 做 metrics 统计
- 对有效包调用 `on_packet` 回调

当前过滤规则由这些配置项驱动：

- `allowed_source_ipv4`
- `receiver_ipv4`
- `allowed_dest_port`

### 6.2 PacketPipeline 的当前定位

`PacketPipeline` 现在只是 legacy frame adapter：

- 输入是 `PacketDesc`
- 先经 `UdpPayloadAssembler::push()` 从 Ethernet frame 提取一个或多个 `UdpPayloadFrame`
- 再逐个构造 `UdpDatagramDesc`
- 最终仍交给 `UdpDatagramPipeline`

当前代码里 `PacketPipeline` 的实际使用场景主要是：

- 单元测试
- 需要从原始 frame 进入协议链的辅助路径

它不是当前 `OwnerLoop` 主循环的一部分。

### 6.3 UdpPayloadAssembler 的当前边界

`UdpPayloadAssembler` 当前负责：

- 解析 Ethernet / IPv4 / UDP 头
- 提取 UDP payload
- 对 legacy frame 路径中的 IPv4 分片做重组

因此当前仓库里出现了一个重要边界：

- frame-native legacy 路径支持 IPv4 分片重组
- DPDK 实时主路径不做分片重组，直接在 ingress 层丢弃分片
- socket 路径天然直接拿到 UDP datagram，不经过它

### 6.4 Parser / Validator / Interpreter 的当前规则

当前协议规则由 `ProtocolSpec` 定义，默认值是：

- `magic_control = 0x55AAFF00`
- `magic_data = 0x55AAFF03`
- `magic_tail = 0x55AAFF30`
- `packet_header_size = 16`
- `udp_packet_size = 2048`
- `packet_data_size = 2032`
- `channels_per_prt = 3`
- `packets_per_channel = 9`
- `iq_per_full_packet = 508`
- `iq_per_last_packet = 476`
- `dynamic_prt_enabled = true`
- `expected_n_prt = 0`
- `max_n_prt = 100`

但需要注意两个当前实现细节：

- `protocol_spec_from_config()` 会把 `channels_per_prt` 限制在 `1..4`，超出则回退到 `3`
- `protocol_spec_from_config()` 会把 `packets_per_channel` 限制在 `1..9`，超出则回退到 `9`

`PacketValidator` 当前检查：

- 控制表 payload 长度必须精确等于 `control_table_size - packet_header_size`
- 数据包 `channel` 范围
- 数据包 `packet_index` 范围
- 数据包 payload 长度必须精确等于 `packet_data_size`
- `tail` 只能是 `0` 或 `magic_tail`
- `tail` 只能出现在最后一个 packet 上

`ProtocolSequenceInterpreter` 当前负责：

- 计算 `packet_position_in_prt`
- 对最后一个 packet 校验：
  - IQ 数应为 `476`
  - 剩余 padding 必须全 `0`
- 对非最后一个 packet 使用 `508` 个 IQ

### 6.5 全局序列号检查

当前 `UdpDatagramPipeline` 会在 `datagram.has_global_sequence=true` 时调用：

- `metrics.on_global_packet_sequence(datagram.global_sequence)`

当前不同后端行为不同：

- DPDK datagram adapter 会填 `global_sequence`
- socket backend 不会填
- file replay 目前也不会填

因此全局序列号 gap 检查当前主要对 DPDK 或显式带序列号的 datagram 生效。

## 7. CPI 生命周期与输出流水线

### 7.1 ControlSnapshot 与动态 PRT

`ControlSnapshot` 当前包含：

- `cpi_id`
- `n_prt`
- `channel_count`
- `packets_per_channel`
- `timeout_ns`
- `valid`
- `bind_source`
- `conflict`

`BindSource` 当前只有三种：

- `fixed`
- `provisional`
- `control`

`CpiStateCoordinator` 当前策略是：

- 若 `dynamic_prt_enabled=false`
  - 始终按 fixed 模式绑定
- 若 control 包先到
  - `open_active()` 直接用 staged control 绑定
- 若 data 包先到
  - `open_active()` 先用 provisional 绑定
- 后续 control 包到达时
  - 若 `candidate.n_prt >= observed_n_prt`，允许 provisional 收敛到 control
  - 否则仅标记 `conflict=true`

还有一个容易忽略的当前实现细节：

- `spec.max_n_prt` 默认是 `100`
- 但 `CpiStateCoordinator` 实际可支持的上限会再取 `min(spec.max_n_prt, kCpiPrtMax)`
- 而 `kCpiPrtMax = 64`

也就是说，当前 CPI 上下文的真实最大 PRT 上限是 `64`。

### 7.2 CpiContext 与上下文池

当前 CPI 存储布局在 `include/rxtech/cpi_context.h`：

- `kCpiPrtMax = 64`
- `kCpiMaxChannelCount = 4`
- `kCpiMaxPacketsPerChannel = 9`
- `kCpiMaxTotalSlots = 64 * 4 * 9 = 2304`
- `kCpiSlotStride = 2048`

单个 `CpiContext` 当前包含：

- 热头 `CpiHotHeader`
- `ControlSnapshot`
- `prt_summary[64]`
- `slot_valid_bytes[2304]`
- `payload[2304 * 2048]`

`CpiContextPool` 的当前深度不是 4，而是：

- `kCpiContextPoolDepth = 16`

并且它通过：

- `std::unique_ptr<CpiContext[]>`
- `std::array<std::atomic<bool>, 16>`

实现 CAS 风格的 acquire/release。

### 7.3 准入与双窗口

`CpiAdmission` 当前只做非常明确的 4 类判定：

- `WRITE_ACTIVE`
  - `packet.cpi == active_cpi`
- `TRIGGER_CPI_SWITCH`
  - `packet.cpi == active_cpi + 1`
- `LATE_TO_CLOSED`
  - 命中 `RecentClosedRing`
- `DROP`
  - 其余情况

`CpiStateCoordinator` 在 `TRIGGER_CPI_SWITCH` 时的真实操作是：

1. 先 finalize 老的 `previous_ctx_`
2. 把当前 `active_ctx_` 挪成 `previous_ctx_`
3. 再为新 CPI 打开新的 `active_ctx_`

后续如果 admission 结果不是 `WRITE_ACTIVE`，但 `parsed.cpi == previous_ctx_->header.cpi_id`，则允许把迟到包写到 `previous_ctx_`。

也就是说，当前“双窗口”只保留：

- 一个 active
- 一个 previous

### 7.4 SlotWriter 与 ProgressTracker

`SlotWriter` 当前负责：

- 校验 slot 坐标合法性
- 若 `expected_n_prt` 已知，做 PRT 上界检查
- 拒绝重复包
- 把 payload 拷入 CPI slot
- 更新：
  - `slot_valid_bytes`
  - `received_slot_count`
  - `duplicate_count`
  - `first_rx_tsc` / `last_rx_tsc`
  - `observed_n_prt`

`ProgressTracker` 当前负责：

- 当某个 channel 收齐 `packets_per_channel` 时，增加 `ready_channel_count`
- 当某个 PRT 的所有 channel 都 ready 时，增加 `ready_prt_count`
- 看到 tail 时设置：
  - `TriggerTailObserved`
- 当 tail 出现在最后一个 expected PRT 上时，再设置：
  - `TriggerWaveEnd`
- 当 `received_slot_count >= expected_slot_count` 时设置：
  - `TriggerFullReady`

### 7.5 Finalize 规则

`CpiFinalizer::try_finalize()` 当前规则如下：

- `TriggerFullReady`
  - `decision = COMPLETE_OK`
- `TriggerCpiSwitch` 或 `TriggerWaveEnd`
  - 若 `ready_prt_count > 0`
    - `decision = INCOMPLETE_BUT_COMMITTABLE`
  - 否则
    - `decision = ABNORMAL_CUTOFF_COMMIT`
- `TriggerTimeout` 或 `TriggerStop`
  - 若上下文完全没有可观察数据
    - 返回 `nullopt`
  - 只要已有可观察数据
    - `decision = ABNORMAL_CUTOFF_COMMIT`

这和旧版本“超时/停机一律不产出 output”不同。

### 7.6 输出流水线

当前 CPI finalize 后的流水线是：

```text
OwnerLoop / CpiStateCoordinator
  -> output_ring.push(CpiOutput)
  -> CpiConsumer.run()
       -> 可选 handler_(output)
       -> recycle_ring.push(ReleaseToken)
  -> owner 线程 drain_recycle()
       -> CpiContextPool.release(pool_index)
```

关键点如下：

- `output_ring` 和 `recycle_ring` 都是 `SpscRing`
- `SpscRing` 对外暴露的 usable capacity 是一个值
  - 内部会分配 `usable + 1` 并上取到 2 的幂
- 当前默认 ring 容量来自配置：
  - `output_ring_capacity = 32`
  - `recycle_ring_capacity = 32`

输出退化策略：

- 若 `output_ring.push()` 失败
  - 当前主线程不会阻塞
  - 会立即丢弃这次 `CpiOutput`
  - 并立刻释放对应 pool slot
  - 同时记一次 `on_output_backpressure()`
- `output_drop_policy=degrade`
  - 最终运行状态升级成 `degraded`
- `output_drop_policy=error`
  - 最终运行状态升级成 `error`

还需要强调一点：

- 当前 CLI 主线没有给 `OwnerLoop` 注册 `output_handler`
- 所以 `CpiConsumer` 线程在正式运行里主要负责：
  - 从 `output_ring` 消费
  - 归还 `ReleaseToken`
- 它不是当前正式业务输出落点
- 这条流水线现在更像：
  - finalize/recycle 机制
  - 外部扩展点
  - 测试覆盖对象

## 8. 可观测性、日志与产物

### 8.1 capture 文件

当前 capture 输出是两份文件：

- `capture_packets.bin`
- `capture_index.csv`

CSV 表头当前固定为：

```csv
cpi,channel,prt,packet_index,packet_kind,payload_len,valid
```

当前写入内容是 UDP payload 本身，不是原始二层帧。

### 8.2 原始帧录制

`RawFrameRecorder` 当前特性：

- 后台 writer 线程
- 固定数量 buffer 池
- 批量写入
- segment 轮转
- 总保留量裁剪
- 输出文件扩展名 `.rawbin`

只有当 datagram 提供 `raw_frame_data/raw_frame_len` 时，主循环才会提交给 recorder。

当前因此实际有效的后端是：

- DPDK
- file replay

而不是 socket。

### 8.3 MetricsCollector 当前负责的主要指标

当前汇总里真实可见的指标包括：

- 协议候选报文数/字节数
- 协议有效包数
- control/data 包数
- reject/drop 数
- pool exhaustion
- output backpressure
- late packet accepted/rejected
- backend 原始收包/空轮询/批次/内核丢弃
- 全局序列号 gap
- CPU 用户/系统/峰值占比

其中：

- CPU 指标在 Linux 上来自 `/proc/self/stat`
- `batch_p99`、`latency_p50_us`、`latency_p99_us` 等更细采样值，只有在 `RXTECH_ENABLE_DEBUG_DIAGNOSTICS` 下才会真正积累样本

### 8.4 DataOrderTracker 当前行为

`DataOrderTracker` 当前按以下顺序构造“期望下一包”：

- 同一 channel 内 `packet_index` 递增
- 一个 channel 结束后切到下一个 channel
- 所有 channel 结束后 `prt + 1`

并且当前已经显式处理 CPI rollover：

- 若上一包正好是某 CPI 的最后一个 `channel/packet_index`
- 下一包是 `cpi + 1, prt = 1, channel = 0, packet_index = 1`
- 则视为正常 rollover，不记 mismatch

这意味着旧版本文档里那条“跨 CPI 一定会报 mismatch”的描述已经不成立。

### 8.5 StatusPanel 与最终汇总

`StatusPanelWriter` 当前行为：

- 若输出目标是交互式终端
  - 使用 ANSI 控制序列做原位刷新
- 否则
  - 直接顺序追加输出

周期性状态面板当前分为：

- `[链路层统计]`
- `[协议层统计]`
- `[输出链路统计]`
- `[可观测性]`
- `[结果层统计]`

最终人类可读总结由 `build_run_human_summary()` 生成，当前还会包含：

- `structured_log_backend`
- `metrics_export` 状态
- `global_packet_sequence` 状态
- 当前 active PRT 覆盖

### 8.6 结构化日志与指标导出

结构化日志当前实现是：

- 优先使用 `spdlog`（若构建时可用）
- 否则退回内建 `builtin` 文本/JSON 输出

当前记录的关键事件有：

- `receiver_start`
- `receiver_stop`
- `receiver_exception`
- `metrics_export_failed`

指标导出当前支持两种模式：

- `prometheus_text`
  - 写文本文件
- `json_socket`
  - 通过 Unix datagram socket 发送 JSON

## 10. 当前实现结论

基于当前代码，可以把接收端主线准确概括为：

> `rx_tech_demo` 当前已经形成一条 Linux-only、datagram-first 的接收主线：`IRxBackend` 负责交付统一 `UdpDatagramDesc`，`UdpDatagramPipeline` 负责协议过滤和解析，`OwnerLoop` 负责 CPI 生命周期与摘要输出，CLI 正式入口目前是 `dpdk` 与 `socket` 两种后端。

当前已经可以明确确认的边界是：

- `UdpDatagramPipeline` 才是当前真实协议入口
- `PacketPipeline` 仍在代码里，但只是 legacy frame adapter
- `socket` 已经是正式 ingress 实现，但它不提供 raw frame
- `file_replay` 是编译期开关控制的可选测试/回放后端，不是当前 CLI 正式入口
- CPI 输出流水线已经存在并被主线调用，但 CLI 默认不挂业务 handler，当前主要承担 finalize/recycle 与扩展点职责
- 文档、测试和运行结论都应继续以 Linux 服务器上的实际验证为准
