核心任务2：给本项目补充一个 Linux Socket的后端接入方式，以支持基于 Socket 的数据接收。

参考文档：`docs\接收模块三层设计参考.md`

现实情况：当前针对本项目有两种接入方式，一种是直接在 src\ 下新增 receiver_socket 模块，完全平行于当前的DPDK，好处是可以完全根据`docs\接收模块三层设计参考.md`中的业务逻辑进行设计，坏处是代码臃肿，设计模式糟糕；一种是在receiver内开启一个Socket后端，平行于dpdk_backend,两者共享收包后的业务逻辑，这样代码更轻便，也符合最佳实践，但是会丧失当前`docs\接收模块三层设计参考.md`中的设计逻辑。

我的思考：
1. 我们能不能使用双层解耦的设计模式，即接收技术上可以灵活选择Linux Socket或者DPDK（使用策略模式），而业务逻辑上融合一下（取其精华 去其糟粕）。但是我想知道如何把这个分层设计的思想融合到现在这个项目里去呢？ 这是我关心的第一个地方。
2. 我关心的第二个地方，我们的工作如何开展，是先完成Linux Socket的策略模式接入还是先完成当前业务逻辑的完整化和分层化实现？这个工程推进和发展的顺序应该是什么样的呢？

**任务 1：先把“控制流快照”从现有接收主线里显式建模出来。**

你要完成的子任务：

1. 梳理当前 `control_table` 包在 `PacketPipeline -> OwnerLoop -> CpiStateCoordinator` 里的传递路径。
2. 新定义一个轻量控制快照对象，建议暂命名为 `ControlSnapshot`。
3. 明确这个对象第一版只承载哪些字段：

   * `cpi_id`
   * `n_prt`
   * `channel_count`
   * `packets_per_channel`
   * `timeout_ns`
   * 后续组装真正需要的最小业务字段
4. 把现有 `current_snapshot_ / BoundWaveSnapshotLite` 和这个新对象的关系理清：

   * 是直接替换
   * 还是先并存，再逐步迁移
5. 给出一版类关系和调用关系设计，回答这三个问题：

   * 控制包由谁解析成快照
   * 快照在何时绑定到 active CPI
   * `CpiOutput` 是否要携带这份快照往下游传

任务 2：

**把 `CpiOutput` 升级成“完整业务单元输出”，显式带上控制面信息。**

你要完成的子任务：

1. 盘点当前 `CpiOutput` 输出给下游的字段，确认哪些只有数据面、哪些缺控制面。
2. 设计 `CpiOutput` 的增强版结构，明确是否增加：

   * `ControlSnapshot` 或其只读视图
   * 绑定来源（`fixed / provisional / control`）
   * 当前 CPI 的完整性/封口原因
3. 明确 `CpiFinalizer` 在 finalize 时，应该从哪个对象取控制快照并绑定到输出。
4. 明确下游消费时，应该以什么对象为唯一输入：

   * 不能再隐含依赖“控制包另找”
   * 必须从 `CpiOutput` 一次拿全
5. 写一页设计说明，回答这三个问题：

   * 为什么 `CpiOutput` 必须成为完整业务边界
   * 控制快照是值拷贝、指针引用，还是只读 view
   * 这样改完后，对后续 Socket/DPDK 双后端有什么好处

这一步的目标是：

**让第三层真正输出“完整 CPI 业务对象”，而不只是“数据拼装结果”。**

这样任务 1 和任务 2 连起来，就形成了第一大方向的主线：
**先把控制面显式化，再把它绑定进最终输出边界。**
