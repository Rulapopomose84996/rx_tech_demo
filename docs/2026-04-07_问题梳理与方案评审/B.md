# B表通用格式数据表文档

## 1. 适用范围

B表用于定义系统统一传输包格式，作为 A 表一次性装订、D 表波位控制、C 表状态回传、C 表数据回传、应答结果及维护调试等各类数据表的统一外层封装格式。

B表只负责以下功能：

1. 判帧
2. 包长识别
3. 包类型分发
4. 分包组织与重组标识
5. 包序检查
6. 整帧校验

B表不负责具体业务字段定义。各类业务字段均由对应包体格式自行定义。

---

## 2. 设计原则

### 2.1 结构原则

B表采用如下统一结构：

- 固定包头
- 变长包体
- 固定包尾

### 2.2 编码原则

1. 全表采用小端编码。
2. 按四字节自然对齐设计。
3. 包头固定长度，便于硬件判帧与状态机实现。
4. 包体采用变长设计，适配控制类与大数据类业务。
5. 包尾固定长度，便于统一校验与验收。

上述原则与参考协议中“统一通用头、全小端序、四字节自然对齐”的做法一致，但本设计进一步向传统硬件数据表表达方式收敛。

---

## 3. 总体结构

B表统一格式如下：

```text
┌──────────────────────────────┐
│ 包头（固定 24 字节）         │
├──────────────────────────────┤
│ 包体（变长）                 │
├──────────────────────────────┤
│ 包尾（固定 8 字节）          │
└──────────────────────────────┘
```

其中：

- 包头长度固定为 24 字节
- 包体长度按包类型变化
- 包尾长度固定为 8 字节

整包长度计算公式如下：

**整包长度 = 包头长度 + 包体长度 + 包尾长度**

---

## 4. 包头定义

## 4.1 包头结构

包头固定长度为 24 字节。

### 4.2 包头字段表

| 偏移 | 字段名称 | 数据类型 | 字长 | 说明 |
|---|---|---|---:|---|
| 0  | 帧首校验字 | 无符号 32 位整数 | 4 字节 | 固定值 `55AAAA55`，用于判定一帧起始 |
| 4  | 包头长度 | 无符号 16 位整数 | 2 字节 | 固定值 `24`，表示包头总长度 |
| 6  | 协议版本 | 无符号 8 位整数 | 1 字节 | 当前格式版本号，首版冻结为 `0x10` |
| 7  | 包类型 | 无符号 8 位整数 | 1 字节 | 表示当前包体所属业务类型 |
| 8  | 整包长度 | 无符号 32 位整数 | 4 字节 | 当前整包总长度，单位为字节，含包头、包体、包尾 |
| 12 | 包序号 | 无符号 32 位整数 | 4 字节 | 业务包序号，用于顺序检查、丢包判断与分包归属 |
| 16 | 分包总数 | 无符号 16 位整数 | 2 字节 | 当前业务包被拆分后的总包数；不分包时固定填 `1` |
| 18 | 分包序号 | 无符号 16 位整数 | 2 字节 | 当前分包序号，从 `1` 开始计数；不分包时固定填 `1` |
| 20 | 预留字段 1 | 无符号 16 位整数 | 2 字节 | 预留扩展，当前版本固定填 `0` |
| 22 | 预留字段 2 | 无符号 16 位整数 | 2 字节 | 预留扩展，当前版本固定填 `0` |

### 4.3 包头字段说明

#### 4.3.1 帧首校验字

- 固定值：`55AAAA55`
- 用途：快速定位包起始位置
- 说明：该字段仅用于判帧，不参与整帧校验计算

#### 4.3.2 包头长度

- 固定值：`24`
- 用途：用于包头格式识别和后续扩展兼容
- 说明：当前版本虽然为固定值，但必须保留此字段

#### 4.3.3 协议版本

- 当前冻结值：`0x10`
- 含义：高 4 位表示主版本，低 4 位表示次版本
- 用途：用于版本识别与兼容管理

#### 4.3.4 包类型

- 用于区分不同业务包体格式
- 解析端必须依据该字段选择对应包体格式

#### 4.3.5 整包长度

- 表示当前物理传输包总长度
- 长度范围为：包头 + 包体 + 包尾
- 接收端必须依据该字段完成当前整包接收

#### 4.3.6 包序号

- 用于顺序检查与丢包判断
- 同一业务消息若发生分包，所有分包应使用同一个包序号
- 新的业务消息应分配新的包序号
- 允许自然回绕

#### 4.3.7 分包总数

- 表示某一业务消息被拆分后的总包数量
- 未拆分时填 `1`
- 适用于 C 表大数据回传等场景

#### 4.3.8 分包序号

- 表示当前分包在整组分包中的位置
- 编号从 `1` 开始
- 未拆分时填 `1`

#### 4.3.9 预留字段

- 当前版本固定填 `0`
- 接收端应允许读取但不参与业务解释
- 用于后续扩展使用

---

## 5. 包体定义

## 5.1 包体总体定义

包体采用变长设计，其长度由下式确定：

**包体长度 = 整包长度 - 包头长度 - 包尾长度**

其中：

- 包头长度固定为 24 字节
- 包尾长度固定为 8 字节

### 5.2 包体组织原则

包体内容不在 B 表中统一展开，而是由包类型决定。各类包体定义如下：

| 包类型 | 包体定义 |
|---|---|
| A表一次性装订包 | 按 A 表格式定义 |
| D表波位控制包 | 按 D 表格式定义 |
| C表状态回传包 | 按状态回传格式定义 |
| C表数据回传包 | 按数据回传格式定义 |
| 应答结果包 | 按应答格式定义 |
| 维护调试包 | 按维护调试格式定义 |

### 5.3 包体对齐要求

1. 包体内部字段建议按四字节自然对齐设计。
2. 若包体实际内容不足四字节对齐要求，应在包体末尾补零。
3. 包体补零部分视为包体内容的一部分，参与整帧校验计算。

之所以保留变长包体和分包机制，是因为当前 C 表已明确包含“IQ 数据长度”和“IQ 数据内容”，该类数据天然存在大数据量分包需求。

---

## 6. 包尾定义

## 6.1 包尾结构

包尾固定长度为 8 字节。

### 6.2 包尾字段表

| 偏移 | 字段名称 | 数据类型 | 字长 | 说明 |
|---|---|---|---:|---|
| 包尾起始 + 0 | 整帧校验码 | 无符号 32 位整数 | 4 字节 | 对指定范围内的数据进行 32 位循环冗余校验后得到 |
| 包尾起始 + 4 | 帧尾校验字 | 无符号 32 位整数 | 4 字节 | 固定值 `AA5555AA`，用于判定一帧结束 |

### 6.3 包尾字段说明

#### 6.3.1 整帧校验码

- 用于整帧数据正确性校验
- 由发送端计算并填写
- 由接收端复算并比对
- 校验失败时，该包应判为无效包

#### 6.3.2 帧尾校验字

- 固定值：`AA5555AA`
- 用途：用于辅助判定一帧结束位置
- 说明：该字段不参与整帧校验计算

---

## 7. 整帧校验规则

## 7.1 校验范围

整帧校验码的计算范围冻结如下：

- 起点：从“包头长度”字段开始
- 终点：到“包体结束”位置为止

### 不包含以下内容

1. 帧首校验字
2. 整帧校验码字段自身
3. 帧尾校验字

### 7.2 等价偏移描述

若以整包起始字节为偏移零点，则整帧校验范围为：

- 起始偏移：`4`
- 结束偏移：`整包长度 - 9`

### 7.3 校验内容组成

参与整帧校验的内容包括：

1. 包头中的以下字段
   - 包头长度
   - 协议版本
   - 包类型
   - 整包长度
   - 包序号
   - 分包总数
   - 分包序号
   - 预留字段 1
   - 预留字段 2
2. 包体全部字节内容
   - 包体业务字段
   - 包体末尾用于对齐的补零字节（如有）

### 7.4 校验算法

本版冻结整帧校验算法为 **CRC32/IEEE 802.3**，参数如下：

| 参数项 | 冻结值 |
|---|---|
| CRC 名称 | `CRC-32/IEEE` |
| 生成多项式 | `0x04C11DB7` |
| 初始值 | `0xFFFFFFFF` |
| 输入反射 | `true` |
| 输出反射 | `true` |
| 最终异或值 | `0xFFFFFFFF` |
| 校验位宽 | 32 bit |
| 校验范围 | 偏移 `4` 至 `整包长度 - 9` |
| 结果写入字节序 | 小端写入“整帧校验码”字段 |

补充说明：

1. CRC 的输入顺序按报文实际发送字节顺序处理，不做额外字序翻转。
2. 包体末尾为满足 4B 对齐而补齐的零字节，属于 CRC 计算范围。
3. 若后续协议版本需要切换为其他 CRC 族，必须通过协议版本升级明确说明，当前版本不允许混用。

## 8. 包类型编码

B表包类型冻结如下：

| 包类型值 | 名称 | 说明 |
|---|---|---|
| `0x01` | A表一次性装订包 | 用于一次性下装的资源表或模板表 |
| `0x02` | D表波位控制包 | 用于波位级控制下发 |
| `0x03` | C表状态回传包 | 用于自检状态、工作状态等信息回传 |
| `0x04` | C表数据回传包 | 用于数据控制信息与 IQ 数据回传 |
| `0x05` | 应答结果包 | 用于命令执行结果或装订结果反馈 |
| `0x06` | 维护调试包 | 用于调试、维护、诊断用途 |
| `0xFF` | 预留扩展包 | 用于后续扩展 |

### 8.1 包类型使用说明

1. A 表与 D 表属于下行控制类包。
2. C 表状态回传与 C 表数据回传属于上行回传类包。
3. 应答结果包用于形成控制闭环。
4. 维护调试包独立于常规业务流程。

---

## 9. 长度与取值约束

## 9.1 固定长度项

| 项目 | 长度 |
|---|---:|
| 包头长度 | 24 字节 |
| 包尾长度 | 8 字节 |

## 9.2 长度关系

整包长度必须满足：

**整包长度 = 24 + 包体长度 + 8**

即：

**整包长度 = 包体长度 + 32**

## 9.3 最小整包长度

当包体长度为 0 时：

**最小整包长度 = 24 + 0 + 8 = 32 字节**

## 9.4 分包字段约束

1. 不分包时：
   - 分包总数 = 1
   - 分包序号 = 1
2. 分包传输时：
   - 分包总数 ≥ 2
   - 分包序号范围为 `1` 至 `分包总数`
   - 同一业务消息的各分包应保持相同包序号
   - 同一业务消息的各分包包类型必须一致

## 9.5 预留字段约束

1. 当前版本发送端必须填 `0`。
2. 当前版本接收端必须忽略其业务含义。
3. 后续如有扩展，必须通过协议版本升级明确说明。

---

## 10. 建议冻结结论

本版 B 表冻结如下：

1. B 表作为系统统一传输包格式，不承载业务细节。
2. B 表采用“固定包头 + 变长包体 + 固定包尾”结构。
3. 包头固定 24 字节。
4. 包尾固定 8 字节。
5. 帧首校验字固定为 `55AAAA55`。
6. 帧尾校验字固定为 `AA5555AA`。
7. 全表采用小端编码。
8. 按四字节自然对齐设计。
9. 整帧校验范围为“包头长度字段至包体结束”。
10. 整帧校验算法冻结为 `CRC-32/IEEE`，参数见 7.4 节。
11. 分包字段作为通用能力保留，不分包时固定填 1。
12. 包类型采用统一编码管理。
13. 预留字段当前版本全部清零。
