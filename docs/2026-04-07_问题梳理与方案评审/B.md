# B表 通用封装格式文档

## 1. 适用范围

B 表用于定义系统统一传输包格式，作为各类业务表的**通用外层封装**。

B 表只负责以下内容：

1. 判帧。
2. 包头识别。
3. 整包长度识别。
4. 包类型分发。
5. 分包组织与重组标识。
6. 整帧校验。

---

## 2. 完整包结构

```text
┌──────────────────────────────┐
│ 包头（固定 24B）             │
├──────────────────────────────┤
│ 包体（变长）                 │
├──────────────────────────────┤
│ 包尾（固定 8B）              │
└──────────────────────────────┘
```

其中：

1. 包头长度固定为 `24B`
2. 包体长度按业务类型变化
3. 包尾长度固定为 `8B`

整包长度计算公式如下：

**整包长度 = 包头长度 + 包体长度 + 包尾长度**

即：

**整包长度 = 24 + 包体长度 + 8**

---

## 3. B 表专用包头

### 3.1 结构说明

B 表专用包头固定长度为 **24B**，用于完成判帧、版本识别、包类型分发、整包长度识别和分包归组。

### 3.2 字段表

| 偏移 | 字段名称 | 数据类型 | 字长 | 取值范围 | 量化口径 | 注释说明 |
| -- | -- | -- | --: | -- | -- | -- |
| 0 | 帧首校验字 | `uint32` | 4B | 固定值 `0x55AAAA55` | 无 | **用于快速定位整帧起始位置。**<br>发送端固定写入 `0x55AAAA55`。<br>接收端若读取值不匹配，应立即判定当前字节流不是合法 B 表起始，不进入后续字段解析。 |
| 4 | 包头长度 | `uint16` | 2B | 固定值 `24` | `1B/LSB` | **表示 B 表专用包头总长度。**<br>本版固定为 `24`。<br>接收端可先检查该字段是否为 `24`，再决定后续偏移是否按本版格式解释。 |
| 6 | 协议版本 | `uint8` | 1B | 当前冻结 `0x10` | 高 4 位主版本<br>低 4 位次版本 | **表示 B 表当前协议版本。**<br>本版固定为 `0x10`。<br>示例：`0x10` 表示主版本 `1`、次版本 `0`。<br>若主版本不匹配，接收端不得按本版字段表继续解释。 |
| 7 | 包类型 | `uint8` | 1B | `0x01..0x06`<br>`0xFF` | 无量纲编码 | **表示当前包体所属业务类型。**<br>接收端必须依据该字段选择对应业务包体格式。<br>未定义包类型当前版本不得按已知业务表解析。 |
| 8 | 整包长度 | `uint32` | 4B | `32..0xFFFFFFFF` | `1B/LSB` | **表示当前整帧总长度，单位为字节。**<br>整包长度包含包头、包体、包尾三部分。<br>接收端必须校验“实际接收长度 = 整包长度字段值”。 |
| 12 | 包序号 | `uint32` | 4B | `0..0xFFFFFFFF` | 无 | **表示当前业务包序号。**<br>同一业务消息若被拆分成多包，所有分包必须使用同一个包序号。<br>新的业务消息必须分配新的包序号。<br>允许自然回绕。 |
| 16 | 分包总数 | `uint16` | 2B | `1..65535` | 无 | **表示当前业务消息被拆分后的总包数。**<br>不分包时固定填 `1`。<br>分包传输时必须大于等于 `2`。<br>同一业务组内所有分包必须保持相同的分包总数。 |
| 18 | 分包序号 | `uint16` | 2B | `1..分包总数` | 无 | **表示当前分包在整组分包中的顺序位置。**<br>编号从 `1` 开始。<br>不分包时固定填 `1`。<br>接收端不得接受 `分包序号 > 分包总数` 的分包。 |
| 20 | 预留字段 1 | `uint16` | 2B | 当前冻结 `0x0000` | 无 | **预留扩展字段。**<br>当前版本发送端必须写 `0`。<br>接收端不得赋予其业务语义。 |
| 22 | 预留字段 2 | `uint16` | 2B | 当前冻结 `0x0000` | 无 | **预留扩展字段。**<br>当前版本发送端必须写 `0`。<br>接收端不得赋予其业务语义。 |

### 3.3 补充说明

1. `offset = 0..23` 全部属于 B 表专用包头范围。
2. `包头长度 = 24` 是本版格式识别前提之一；接收端在包头长度不为 `24` 时，不得继续按本版解释。
3. `整包长度` 的最小合法值为 `32`，即包体长度为 `0` 时的最小帧长。
4. 分包归组时，接收端不应只依据 `分包序号` 排序，必须先按**包类型 + 包序号**识别业务组，再在组内按 `分包序号` 排列。
5. `预留字段 1/2` 当前版本只能写 `0`。若现场联调收到非 `0` 值，可记录日志，但不得将其作为当前版本业务字段解释。

## 4. 包体规则

### 4.1 结构说明

包体位于包头和包尾之间，为**变长区域**。

其起止范围如下：

1. 包体起始偏移：`offset = 24`
2. 包体结束偏移：`offset = 整包长度 - 9`

### 4.2 长度关系

包体长度由以下公式确定：

**包体长度 = 整包长度 - 包头长度 - 包尾长度**

代入本版固定长度后可写为：

**包体长度 = 整包长度 - 24 - 8**

即：

**包体长度 = 整包长度 - 32**

### 4.3 对齐与补零规则

1. 包体内部字段建议按 4B 自然对齐设计。
2. 若包体实际业务内容不足 4B 对齐要求，可在包体末尾补零。
3. 包体末尾补零字节属于包体的一部分，计入 `整包长度`。
4. 包体末尾补零字节参与整帧 CRC 计算。

### 4.4 补充说明

1. B 表不展开包体内部业务字段定义，只为业务包体提供统一封装边界。
2. 包体是否允许为空，由具体业务类型决定；B 表本身不禁止 `包体长度 = 0`。
3. B 表不规定包体的业务合法性检查，仅规定包体在整帧中的边界和长度关系。

---

## 5. B 表包尾

### 5.1 结构说明

包尾固定长度为 **8B**，位于整帧末尾。

其起止范围如下：

1. 包尾起始偏移：`offset = 整包长度 - 8`
2. 包尾结束偏移：`offset = 整包长度 - 1`

### 5.2 字段表

| 偏移 | 字段名称 | 数据类型 | 字长 | 取值范围 | 量化口径 | 注释说明 |
| -- | -- | -- | --: | -- | -- | -- |
| 包尾起始 + 0 | 整帧校验码 | `uint32` | 4B | `0..0xFFFFFFFF` | CRC32/IEEE 结果值 | **表示当前整帧的 CRC32/IEEE 校验结果。**<br>发送端对指定范围内的数据计算 CRC32/IEEE 后，将结果以小端写入本字段。<br>接收端必须复算并比对本字段。 |
| 包尾起始 + 4 | 帧尾校验字 | `uint32` | 4B | 固定值 `0xAA5555AA` | 无 | **用于辅助判定整帧结束位置。**<br>发送端固定写入 `0xAA5555AA`。<br>接收端不得仅依赖帧尾校验字判定整帧结束，必须结合 `整包长度` 同时判断。 |

### 5.3 补充说明

1. 包尾字段的等价整包偏移如下：<br>`整帧校验码` 位于 `offset = 整包长度 - 8 .. 整包长度 - 5`<br>`帧尾校验字` 位于 `offset = 整包长度 - 4 .. 整包长度 - 1`
2. `整帧校验码` 字段自身不参与 CRC 计算。
3. `帧尾校验字` 不参与 CRC 计算。
4. 接收端建议先依据 `整包长度` 定位包尾，再检查 `帧尾校验字` 是否匹配固定值。

---

## 6. 整帧 CRC32/IEEE

### 6.1 字段与覆盖范围

整帧校验码采用 **CRC32/IEEE**，覆盖范围冻结如下：

1. 起点：从“包头长度”字段开始
2. 终点：到“包体结束”位置为止

若以整包起始字节为零点，则等价偏移范围为：

1. 起始偏移：`offset = 4`
2. 结束偏移：`offset = 整包长度 - 9`

### 6.2 CRC 参数

| 参数项 | 冻结值 |
| -- | -- |
| CRC 名称 | `CRC-32/IEEE` |
| 生成多项式 | `0x04C11DB7` |
| 初始值 | `0xFFFFFFFF` |
| 输入反射 | `true` |
| 输出反射 | `true` |
| 最终异或值 | `0xFFFFFFFF` |
| 结果写入字节序 | 小端写入“整帧校验码”字段 |

### 6.3 补充说明

1. 参与 CRC 计算的内容包括：<br>`offset = 4` 开始的包头字段<br>包体全部字节内容<br>包体末尾的对齐补零字节
2. 不参与 CRC 计算的内容包括：<br>`offset = 0..3` 的帧首校验字<br>整帧校验码字段自身<br>帧尾校验字
3. CRC 输入顺序按报文实际发送字节顺序处理，不做额外字序翻转。
4. 若后续协议版本需要切换为其他 CRC 族，必须通过协议版本升级明确说明，当前版本不允许混用。

## 7. 包类型编码

### 7.1 包类型字段表

| 包类型值 | 名称 | 说明 |
| -- | -- | -- |
| `0x01` | A表一次性装订包 | 用于一次性下装的资源表或模板表 |
| `0x02` | D表波位控制包 | 用于波位级控制下发 |
| `0x03` | C1自然状态回传包 | 用于状态、告警、自检、姿态等信息回传 |
| `0x04` | C2波位数据回传包 | 用于数据控制信息与 IQ 数据回传 |
| `0x05` | 应答结果包 | 用于命令执行结果或装订结果反馈 |
| `0x06` | 维护调试包 | 用于调试、维护、诊断用途 |
| `0xFF` | 预留扩展包 | 用于后续扩展 |

### 7.2 补充说明

1. 接收端必须依据 `包类型` 选择对应业务包体解析方式。
2. 同一组分包内不得混用不同包类型。
3. 对于当前版本未定义的包类型，接收端不得按已知业务表格式强行解释。

---

## 8. 分包与重组约束

### 8.1 发送端约束

1. 不分包时，必须满足：<br>`分包总数 = 1`<br>`分包序号 = 1`
2. 分包传输时，必须满足：<br>`分包总数 >= 2`<br>`分包序号` 从 `1` 开始连续编号到 `分包总数`
3. 同一业务组内所有分包必须保持以下字段完全一致：<br>`包类型`<br>`包序号`<br>`分包总数`
4. 同一业务组内每个 `分包序号` 只能出现 1 个合法分包。

### 8.2 接收端约束

1. 接收端应先按 `包类型 + 包序号` 识别同一业务组。
2. 在同一业务组内，应按 `分包序号` 对分包排序。
3. 若同一业务组内出现多个不同的 `分包总数`，应判为非法组。
4. 若出现 `分包序号 = 0` 或 `分包序号 > 分包总数`，应拒绝该分包。
5. 若业务组声明 `分包总数 = N`，则仅当收到 `1..N` 全部合法分包后，才可视为分包组完整。

### 8.3 补充说明

1. B 表只定义分包标识方式，不规定具体业务的重组超时策略。
2. B 表只定义分包格式一致性要求，不规定业务层对缺包、乱序、重包的处置策略。
3. 业务层若需要超时重组、丢包作废、局部补零等策略，应在对应业务表或实现代码中另行规定。

---

## 9. 使用与校验建议

1. 发送端建议按以下顺序生成整帧：<br>先填写包头字段<br>再填写包体<br>必要时对包体末尾补零对齐<br>再计算并写入整帧校验码<br>最后写入帧尾校验字
2. 接收端建议按以下顺序校验整帧：<br>先校验帧首校验字<br>再校验包头长度与整包长度<br>再定位并校验帧尾校验字<br>再复算 CRC32/IEEE<br>全部通过后再将包体交给对应业务表解析
3. 若存在分包，建议在完成单帧合法性校验后，再进入业务组重组流程。
