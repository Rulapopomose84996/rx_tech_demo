## 8. 网络拓扑与端口映射适配要求

本项目目标是比较不同接收后端在相同流量模型下的公平性，因此网络拓扑必须固定。

建议约束如下：

- 单阵面对应单物理口或单 RX queue
- 三阵面测试优先采用三物理口或三独立 queue 映射
- 所有对比测试必须固定 MTU、端口速率、流量模型、包长分布和发流源
- 需要在测试记录中明确“阵面 -> 网口 -> queue -> CPU -> NUMA”映射关系

推荐优先使用 Intel X710 的三个端口或三个 queue 承载三阵面对比场景，原因是：

- 队列资源更充足
- 10GbE 更符合高带宽验证目标
- 更利于后续扩展 AF_XDP / DPDK

## 9. NUMA 与 CPU 绑定要求

### 9.1 绑定原则

- 网卡所在 NUMA 节点优先匹配接收线程 CPU
- 后处理线程应尽量与对应接收线程位于同 NUMA 节点
- 不同阵面之间避免共享接收核
- 三阵面并发场景下，不同阵面不得复用同一个核心

### 9.2 推荐绑核策略

单阵面：

- 1 个接收核
- 1 个解析或消费核
- 所有相关线程位于网卡所在 NUMA 节点

三阵面：

- 每个阵面至少 1 个接收核
- 如使用 `SPSC` 模式，每个阵面额外 1 个消费核
- 共计建议 `6` 个物理核起步

### 9.3 IRQ / RSS 约束

- `socket` 模式必须记录 RSS 与 IRQ affinity 配置
- `AF_XDP` 模式必须明确 queue id 和 XDP redirect 关系
- `DPDK` 模式必须明确 lcore、port、queue 绑定关系

## 16. 已知问题、限制与风险

### 16.1 当前已确认问题

- `libxdp` 已通过共享缓存前缀补齐，但默认系统路径仍不会自动发现该库
- `dpdk-hugepages.py` 仍未提供，但不阻断当前 DPDK 最小闭环
- 当前目标网口默认未挂载常驻 XDP program，AF_XDP attach/detach 流程已验证通过
- Intel X710 在该内核组合下已确认 `zerocopy` 不可用，强制 `zerocopy` 时 `xsk_socket__create failed`
- DPDK 具备 VFIO、独立 IOMMU group、hugepage、用户态工具链与 `testpmd` 闭环能力
- AF_XDP 最小 bind 已成功，且 zero-copy / copy 模式判定已完成
- 收包级别 AF_XDP RX PoC 已成功运行，当前已确认 `xsk_mode=copy`
- 使用 `enP1s25f0 -> enP1s25f3` 的受控 raw UDP 发流时，发送成功，但 `enP1s25f3` 的主机侧 `rx_packets` / `rx-0.packets` 仍未增长，说明当前 ingress 路径尚未打通到 queue 0

### 16.2 风险说明

- 若 AF_XDP 驱动仅支持 copy mode，则收益可能有限
- 若 DPDK 驱动绑定需要改动现网网卡归属，则需严格隔离验证窗口
- 若 NUMA、IRQ、queue、绑核不固定，对比结果会失真
- 若直接使用 WX1860A4 作为 AF_XDP/DPDK 首批验证对象，风险高于 Intel X710

## 17. 回退方案

当高性能后端条件不满足时，建议按以下顺序回退：

1. DPDK 不满足前置条件时，回退到 AF_XDP
2. AF_XDP 不满足前置条件时，回退到 `socket + recvmmsg`
3. 若共享缓存依赖未准备完成，优先保证 `socket` 基线能在服务器原生构建和测试

这也是本项目当前最合理的实施顺序。

### 18.5 当前已确认决策

#### 18.5.1 AF_XDP 测试口

推荐：

- `enP1s25f3`

推荐理由：

- 当前接口处于 `UP`
- 同属 Intel X710，具备 32 队列能力
- 从当前项目语境看，更适合作为实验口而不是主数据口
- 先拿同一个实验口完成单口单队列 AF_XDP PoC，风险最低

执行约束：

- `2026-03-17` 已确认 `enP1s25f3` 不承载现网关键业务，并长期作为专用实验口
- 若后续现场网络角色发生变化，应先更新 `docs/AF_XDP_DPDK_准备与分工.md`，再调整本节结论

#### 18.5.2 DPDK 解绑测试口

推荐：

- `enP1s25f3`

推荐理由：

- DPDK 验证需要驱动重绑，风险高于 AF_XDP
- 将 AF_XDP 与 DPDK 都集中在同一个“专用实验口”上，最容易控制影响范围
- 避免误碰更可能承担现网数据面的端口，例如 `enP1s25f0`

执行约束：

- `2026-03-17` 已确认允许在正式 `DPDK` 压测窗口继续对 `enP1s25f3` 执行驱动重绑
- 必须提前准备从 `vfio-pci` 绑回原驱动 `i40e` 的回退命令
- 不得把同类操作扩展到其他 X710 端口，除非先更新基线文档并重新确认

#### 18.5.3 依赖管理方式

推荐：

- `AF_XDP`：优先采用“`libbpf` 系统安装 + `libxdp` 共享缓存前缀”
- `DPDK`：优先采用“共享缓存离线化 + 项目自管版本”

推荐理由：

`AF_XDP`：

- 更贴近内核与系统态网络栈
- `libbpf` 已能稳定走系统安装
- `libxdp` 在当前麒麟仓库中无现成系统包，改由共享缓存前缀管理更可控
- 需要时只需导出前缀环境，便于快速拉起最小 PoC

`DPDK`：

- 对版本、驱动、工具链和构建参数更敏感
- 更适合项目内锁版本，避免不同项目之间相互污染
- 共享缓存模式更利于在麒麟 V10 ARM64 上复用和追溯

执行约束：

- 无论哪种方式，都必须把“实际安装版本”和“获取方式”写入项目文档
- AF_XDP 当前已固化为“`libbpf` 系统安装 + `libxdp 1.2.9` 共享缓存前缀”模式

#### 18.5.4 是否允许启用 hugepage

推荐：

- 允许启用

推荐理由：

- 不启用 hugepage，DPDK 路线无法进入正式验证
- 当前服务器已挂载 `hugetlbfs`，说明系统层面已经为这条路留了入口
- 当前主要缺的是“页数分配”，而不是“平台完全不支持”

执行约束：

- 当前 `2 x 512MB` hugepage 基线已可用于最小闭环和正式压测入口
- 如压测规模扩大，可按测试规模调整页数，但调整后必须重新验收 `nr_hugepages` 与 `free_hugepages`
- 如调整 hugepage 策略，应先更新基线文档
