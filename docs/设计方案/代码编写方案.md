## Plan: 新分支——监控/配置/协议/核心四路并进

**TL;DR：** 在新分支 `dev/phase3-infra` 上，按协议事实裁剪监控、将配置全面切到文件驱动、将协议常量参数化以解耦，并按设计文档开始实现真正的热路径对象模型（CpiContext、SlotWriter、ProgressTracker 等）。

---

### Phase A — 监控层裁剪（Sidecar Simplification）

**背景**：当前 `UdpPayloadFrame` 携带 `ts_ns / port_id / queue_id / face_id`；CSV index 和 `RunSummary` 里记录了协议中根本不存在的字段。按协议事实（doc 08），包中只有 4 个业务字段：`cpi(2B) / channel(2B) / prt(2B) / packet_index(2B)`。

**步骤**

1. **精简 `UdpPayloadFrame`**（udp_payload_assembler.h）：保留 `source/dest IP/port`，将 `ts_ns / port_id / queue_id / face_id` 移出结构体或降级为可选的 DPDK 后端私有信息（不进入协议层）
2. **精简 `SamplePacketView`**（[src/receiver/protocol/rxtech/sample_packet_parser.h](src/receiver/protocol/rxtech/sample_packet_parser.h)）：去除 `ip_fragment_offset / more_ip_fragments / header_offset / frame_length / error_reason`（这些是传输层细节，不属于业务视图）；业务字段仅保留 `{valid, kind, cpi, channel, prt, packet_index, tail, payload_ptr, payload_len}`
3. **精简 CSV index 格式**（owner_loop.cpp）：CSV 只输出 `{cpi, channel, prt, packet_index, packet_kind, payload_len, valid}`，移除 `sequence / offset / ts_ns / port_id / queue_id / face_id / validation`
4. **精简 `RunSummary` 协议部分**（metrics.h）：`ProtocolChannelSummary` 只留 `{channel, channel_name, data_packets, iq_count}`；`ProtocolCpiSummary` 只留 `{cpi, control_table_packets, data_packets, prt_count}`；移除 `zero_padding_bytes / duplicate_fragments` 等细节
5. 同步更新 `IMetricsCollector` 接口，移除 `on_port_packet / on_reassembled_block / on_missing_fragments / on_duplicate_fragment / on_invalid_header`，替换为单一 `on_protocol_event(kind, cpi, channel, prt, packet_index)` 或若干语义清晰的简化方法

### Phase B — 配置系统重构（Config File–First）

**背景**：当前有 CLI positional args + named flags + key:value 文件三套机制，混乱且 CLI 承担了过多职责。目标：只保留 `--config path` 和 `--dry-run`，其余全部通过配置文件。

**步骤**

1. **扩展 `RxConfig`**（[src/receiver/runtime/rxtech/rx_config.h](src/receiver/runtime/rxtech/rx_config.h)）新增字段：
   - `log_level` (string: "debug"/"info"/"warn"/"error")
   - `log_output` (string: "stdout"/"file")
   - `log_file_path`
   - `capture_output_dir`（替代现在的 `output_dir`，语义更明确）
   - `capture_enabled` (bool)
   - `capture_index_filename` / `capture_data_filename`
2. **升级 `load_config_file`**（[src/receiver/runtime/src/rx_config.cpp](src/receiver/runtime/src/rx_config.cpp)）：不引入外部 YAML 库，使用当前已有的 `key=value` / `key: value` 格式，增加 section 支持（`[dpdk]` / `[network]` / `[capture]` / `[protocol]` / `[runtime]`），类似 INI/TOML 风格。这样零依赖，适配 CMake 3.16 环境
3. **精简 `CliArgs`**（[src/receiver/app/cli/cli_args.h](src/receiver/app/cli/cli_args.h)）：只保留 `--config FILE`，`--dry-run`，`--help`；移除 `--output / --iface / --queue / --duration / --max-burst / --cores` 等（这些全部进配置文件）
4. **更新两个示例 conf 文件**（dpdk_single_face.conf, dpdk_receiver0_replay.conf）：补全新字段，增加 section 分组注释
5. **更新 `app_main_common.cpp`**：移除 CLI override 合并逻辑，改为 "默认值 → conf 文件（必须提供）→ 仅 --dry-run 例外"

### Phase C — 协议层解耦（Protocol Parameterization）

**背景**：当前 `SamplePacketParser / SamplePacketValidator / ProtocolSequenceInterpreter` 里硬编码了 magic constants、包大小、通道数、每通道包数等。协议字段会随项目推进大面积修改，必须参数化。

**步骤**

1. **新建 `ProtocolSpec` 结构体**（新文件 `src/receiver/protocol/rxtech/protocol_spec.h`）：
   ```
   magic_control  = 0x55AAFF00
   magic_data     = 0x55AAFF03
   magic_tail     = 0x55AAFF30
   magic_ctrl_table = 0x55AAFF00
   udp_packet_size       = 2048
   packet_header_size    = 16
   packet_data_size      = 2032
   channels_per_prt      = 4
   packets_per_channel   = 9
   iq_per_full_packet    = 508
   iq_per_last_packet    = 476
   control_table_size    = 2048
   ```
2. **将 `ProtocolSpec` 纳入 `RxConfig`**：在 `[protocol]` section 中可覆盖所有字段
3. **重构 Parser / Validator / Interpreter**：构造时注入 `const ProtocolSpec&`，移除所有 `constexpr` 魔数。`SamplePacketParser` 改为 `PacketParser`，`SamplePacketView` 改为 `ParsedPacketView`（与设计文档对齐）
4. **`ProtocolSequenceInterpreter` 计算逻辑参数化**：`27 = 3 × 9`（注意：doc 08 说是 4 通道，当前代码里是 3 通道，这是需要对齐的差异，见下方"待确认"）
5. 保持 `IRxBackend` / `ReceiveContext` 等不变，仅协议层内部重构

### Phase D — 核心热路径推进（CpiContext + SlotWriter + ProgressTracker）

**背景**：当前 `OwnerLoop` 是一个"平铺"实现，把解析/验证/录制/统计全混在一起。设计文档定义了 9 模块架构，现在开始按图索骥落地关键模块。**此 Phase 建立骨架，不要求完整闭环**。

**步骤**

1. **新增目录 `src/receiver/storage/`**：
   - `rxtech/cpi_context.h`：实现 `CpiHotHeader` + `PrtSummary[N_PRT_MAX]` + `slot_valid_bytes[]` 前段；`payload[N_PRT_MAX*36][2048]` 后段；`CpiState` 枚举（ACTIVE/DECIDING/SEALED/TOMBSTONE/RECYCLED）
   - `rxtech/cpi_context_pool.h`：固定深度对象池，启动时一次分配，`N_PRT_MAX` 和 `CTX_POOL_DEPTH` 为编译期常量或 startup 参数
   - `src/slot_writer.cpp`：实现 `SlotWriter::write()`：slot_index 计算、bitmap 查重、首次写入的 memcpy 序列
   - `src/progress_tracker.cpp`：实现 `ProgressTracker::advance()`：bitmap 更新、ch_recv_count、ready_prt_count、FULL_READY/TAIL_OBSERVED 标志

2. **新增 `src/receiver/admit/`**：
   - `rxtech/cpi_admission.h` + `src/cpi_admission.cpp`：实现 `CpiAdmission::judge()`，返回 `{WRITE_ACTIVE, TRIGGER_CPI_SWITCH, LATE_TO_CLOSED, DROP}`；包含 `RecentClosedRing`（固定深度 8）

3. **新增 `src/receiver/output/`**：
   - `rxtech/cpi_output.h`：定义 `CpiOutput`（只读视图 + decision + aggregate），`ReleaseToken`
   - `rxtech/output_pool.h`：对应 ctx_pool 的输出对象池

4. **重构 `OwnerLoop`**（owner_loop.cpp）：将当前的平铺逻辑拆分为调用链：`PacketParser → PacketValidator → CpiAdmission → SlotWriter → ProgressTracker → CpiFinalizer`；OwnerLoop 本身只做循环驱动和 `OwnerState` 持有，不再内联业务逻辑

5. **`CpiFinalizer` 骨架**（finalize，目录已存在）：实现状态机转换逻辑，先做 `FULL_READY` 和 `CPI_SWITCH` 两个触发条件；`TIMEOUT` 和 `WAVE_END` 先存桩

6. **更新 CMakeLists.txt**：新增 `storage/`、`admit/`、`output/` 三个子目录的编译目标

### 待确认事项

1. **通道数差异**：doc 08 明确"4 通道（和/俯仰差/方位差/匿隐）"，但当前 `ProtocolSequenceInterpreter` 代码里用的是 3 通道 × 9 包 = 27。Phase C 要对齐为 `4 × 9 = 36`，这会影响 validator、slot 计算和 payload 大小（N_PRT_MAX × 36 × 2048）。由于sender没有第四个通道，所以暂时确认为 3 通道 × 9 包 = 27。

1. **配置文件格式**：方案建议用扩展 INI/section 格式（零依赖），不引入 yaml-cpp。


2. **Phase D 边界**：本分支目标是"建立 CpiContext 内存模型 + SlotWriter + ProgressTracker + Admission + 骨架 Finalizer"，**不包括** SPSC 队列接入、下游消费者、完整闭环录制。
