# 软件任务书

## 一、项目名称

高性能服务器接收端技术验证 Demo 开发项目

## 二、项目背景

现有系统采用基于 Linux UDP socket 的接收方案，整体架构已具备独立阵面收包、批量接收、线程绑核、内存池和后处理解耦等高性能设计基础，接收线程与后续处理线程之间通过 SPSC 队列进行隔离，收包热路径已尽量简化  。
随着单阵面接收带宽提升至不低于 4.8 Gbps，需进一步评估 AF_XDP、DPDK 等高性能接收技术在本项目场景下的适用性、性能收益和工程代价。

为降低对现网主程序的影响，拟开发独立的技术验证 Demo，用于建立统一的接收性能评测框架，对比以下三类接收模式：

1. 现有基线方案：socket + recvmmsg
2. AF_XDP 接收方案
3. DPDK 接收方案

## 三、项目目标

开发一套独立的服务器接收端技术验证 Demo，通过统一测试框架，对不同接收后端在相同硬件环境、相同流量模型和相同处理逻辑下的性能进行对比测试，为后续工程选型提供依据。

项目目标包括：

1. 建立统一的接收后端抽象接口；
2. 实现 socket、AF_XDP、DPDK 三种接收模式；
3. 建立统一的测试场景、统一的性能指标和统一的结果输出格式；
4. 对单阵面和三阵面接收能力进行对比测试；
5. 评估不同方案在吞吐、CPU 占用、丢包率、突发恢复能力和工程复杂度方面的差异；
6. 形成技术验证报告，为后续是否迁移至 AF_XDP/DPDK 提供决策依据。

## 四、工作内容

### 1. Demo 总体框架设计

设计并实现统一的 benchmark 框架，要求不同接收技术仅替换“收包后端”，其余处理流程保持一致，确保对比结果公平可信。

框架应包括以下模块：

* 配置管理模块
* 接收后端抽象接口模块
* 公共解析模块
* 公共统计与指标采集模块
* 公共结果输出模块
* 测试场景驱动模块

### 2. 接收后端开发

开发以下三类接收后端：

#### （1）基线后端

基于现有 UDP socket + recvmmsg 模式实现，作为对比基准。
`recvmmsg()` 支持单次系统调用接收多条消息，适合作为 Linux UDP 接收基线。([man7.org](https://www.man7.org/linux/man-pages/man2/recvmmsg.2.html))

#### （2）AF_XDP 后端

基于 AF_XDP 实现单端口/单队列接收后端，支持 XSK、UMEM、fill/completion ring 管理，验证其在本项目场景下的吞吐和 CPU 优势。AF_XDP 支持将 socket 绑定到特定 `netdev + queue_id`，适合构建“一阵面一队列一线程”的接收模型。([docs.kernel.org](https://docs.kernel.org/6.5/networking/af_xdp.html))

#### （3）DPDK 后端

基于 DPDK PMD 实现用户态轮询接收后端，支持 mbuf 管理、RX queue 轮询和批量收包，对比其极限吞吐能力和部署复杂度。DPDK PMD 采用用户态轮询模型，适合高吞吐低抖动场景。([doc.dpdk.org](https://doc.dpdk.org/guides-24.03/prog_guide/poll_mode_drv.html))

### 3. 公共处理链开发

为保证测试公平性，三类接收后端共用统一后处理逻辑。
Demo 至少支持以下三类处理模式：

1. **RX-only 模式**
   仅接收并计数，不做业务处理，用于测纯接收极限。

2. **RX + 轻量解析模式**
   对报文头进行最小化解析、校验和分类，用于贴近真实热路径。

3. **RX + SPSC handoff 模式**
   将接收到的数据通过 SPSC 队列转交消费线程，用于模拟当前项目中的接收-处理解耦架构。

有条件时，可扩展：

4. **接入当前完整后处理链模式**
   与现有系统后续处理链对接，用于验证方案可落地性。

### 4. 测试场景开发

建立统一的压测场景，至少包括：

1. 单阵面稳态接收测试
2. 三阵面并发稳态接收测试
3. 单阵面突发流量测试
4. 三阵面并发突发流量测试
5. 不同固定包长测试
6. 真实报文长度分布测试

### 5. 指标采集与报告生成

实现统一指标采集与结果输出，输出 JSON/CSV 格式测试结果，并支持自动生成对比报告。

## 五、任务指标

### 1. 功能指标

1. 支持三类接收后端：

   * socket + recvmmsg
   * AF_XDP
   * DPDK

2. 支持三类测试模式：

   * RX-only
   * RX + 轻量解析
   * RX + SPSC handoff

3. 支持以下测试维度：

   * 单阵面
   * 三阵面并发
   * 稳态流量
   * 突发流量
   * 固定包长
   * 真实包长分布

4. 支持统一配置加载与统一结果输出。

### 2. 性能测试指标

Demo 应至少输出以下指标：

* 接收吞吐率（Gbps）
* 包接收速率（Mpps）
* 丢包率
* CPU 占用率
* 单核峰值占用率
* 平均 batch 大小和 batch 分布
* p50/p99 接收处理时延
* 队列高水位
* 缓冲池耗尽次数
* 各后端错误统计

### 3. 对比验证指标

在相同测试条件下，应完成以下验证：

1. 单阵面 4.8 Gbps 场景下的达标性验证；
2. 三阵面同时运行场景下的稳定性验证；
3. 小包场景下的吞吐能力对比；
4. 突发场景下的丢包与恢复能力对比；
5. 各方案单位吞吐 CPU 消耗对比；
6. 各方案工程复杂度、部署成本和可维护性评估。

### 4. 验收判据

项目完成后，应满足以下验收条件：

1. 三类接收后端能够独立运行；
2. 三类接收后端在统一框架下完成对比测试；
3. 能输出统一格式的测试结果；
4. 能完成单阵面和三阵面场景测试；
5. 能形成完整测试报告；
6. 能明确给出以下结论之一：

   * 当前实现已满足需求，暂无必要迁移；
   * AF_XDP 具备明显收益，建议进一步工程化；
   * DPDK 具备显著收益，建议作为后续高性能路线；
   * 各方案收益有限，不建议投入迁移成本。

## 六、技术路线

### 1. 总体技术路线

采用“统一 benchmark core + 多接收后端”的设计方式：

* 公共层负责配置、解析、统计、报告
* 接收后端仅负责不同收包技术实现
* 后处理链尽量共用

### 2. 演进路线

分阶段实施：

#### 阶段一

完成 demo 基础框架和 socket 基线方案，实现统一 benchmark 主体。

#### 阶段二

接入 AF_XDP 后端，完成单端口/单队列 PoC 和性能对比。

#### 阶段三

接入 DPDK 后端，完成用户态轮询 PoC 和性能对比。

#### 阶段四

扩展到三阵面并发场景，形成最终报告。

项目交付物包括但不限于：

3. 测试脚本
4. 测试配置文件
5. 测试结果原始数据（JSON/CSV）
6. 技术验证报告
7. 接收技术选型建议书

## 八、进度建议

建议按以下阶段推进：

1. 需求梳理与方案设计
2. 基线框架与 socket 后端开发
3. AF_XDP 后端开发与测试
4. DPDK 后端开发与测试
5. 场景压测与数据整理
6. 技术报告编写与结论输出

## 九、风险说明

本项目的主要风险包括：

1. 网卡驱动对 AF_XDP zero-copy 支持不足；
2. DPDK 驱动绑定和部署环境改造成本较高；
3. 不同后端运行时模型差异较大，若抽象层设计不当，可能影响公平对比；
4. FPGA 前端若不能稳定提供可控流量模型，会影响测试结果可信度；
5. NUMA、RSS、IRQ affinity 等系统配置若不统一，会导致测试结果失真。

## 十、预期成果

通过本项目，应形成以下能力和成果：

1. 明确当前实现与 AF_XDP/DPDK 在真实接收场景下的性能差异；
2. 明确单阵面 4.8 Gbps、三阵面并发场景下的最优接收技术路线；
3. 明确各方案的工程代价、系统改造点和落地风险；
4. 为后续接收模块架构升级提供量化依据。
