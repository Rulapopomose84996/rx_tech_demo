# 同类项目构建与组织架构重构指南（麒麟V10 ARM64）

> 适用范围：部署在同一类服务器环境上的 C++/CMake 项目  
> 目标环境：Kylin Linux Advanced Server V10 (GFB), aarch64, GCC 7.3.0, CMake 3.16.5  
> 用途：为其他项目提供可复用的构建组织方式、部署思路和旧环境适配策略

---

## 1. 先说结论

如果是同一台服务器或同一类麒麟 V10 ARM64 环境上的项目，建议默认采用这套模式：

1. 顶层 CMake 只做“模块编排”，不要堆业务细节
2. 第三方依赖尽量项目内离线化，不依赖服务器在线安装开发包
3. 构建方式分成三条：
   - 本地 Docker 开发构建
   - WSL/Linux 交叉构建
   - 服务器原生构建
4. 服务器正式验收以“原生构建 + 指定测试子目录”作为标准流程
5. 旧环境下不要过度依赖新标准库特性，特别是 `std::filesystem`
6. 运行时数据、数据库、日志必须和源码目录隔离

## 2. 本项目的推荐目录组织

这个项目最终形成的组织方式，适合作为同类项目模板：

```text
rma_manager/
├── CMakeLists.txt                 # 顶层编排
├── cmake/                         # 选项、告警、工具链、查找模块
├── src/
│   ├── rma_core/                  # 核心库
│   └── rma_service/               # 服务层
├── tests/
│   ├── unit/                      # 单元测试
│   └── integration/               # 集成测试
├── web/                           # 前端
├── profiles/                      # YAML 业务配置
├── deps_offline/                  # 离线依赖包
├── scripts/                       # 构建脚本
├── deploy/                        # 部署资产
├── docs/
│   ├── 设计文档/
│   ├── 部署基线/
│   ├── 项目规划/
│   └── 项目进展/
└── data/                          # 本地运行时数据（不入仓）
```

### 为什么这样分

- `src/rma_core` 和 `src/rma_service` 分开，便于把核心库单独测试和复用
- `profiles/` 独立出来，避免把业务配置硬编码进代码
- `deps_offline/` 独立出来，是旧环境成功构建的关键
- `deploy/` 独立出来，便于把“源码构建”和“部署落地”分层
- `data/` 独立出来，避免 SQLite/日志污染仓库根目录

## 3. 顶层 CMake 应该怎么组织

本项目的顶层做法是推荐范式：

```cmake
cmake_minimum_required(VERSION 3.16)
project(rma_manager VERSION 0.1.0 LANGUAGES C CXX)

set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CXX_STANDARD_REQUIRED ON)
set(CMAKE_CXX_EXTENSIONS OFF)

include(cmake/options.cmake)
include(cmake/warnings.cmake)

add_subdirectory(third_party)
add_subdirectory(src/rma_core)

if(BUILD_SERVICE)
    add_subdirectory(src/rma_service)
endif()

if(BUILD_TOOLS)
    add_subdirectory(tools/mock_dacs)
endif()

if(BUILD_TESTS)
    enable_testing()
    add_subdirectory(tests)
endif()
```

### 关键原则

- 顶层只负责：
  - 编译标准
  - 全局选项
  - 子模块装配
- 子模块各自维护自己的 `CMakeLists.txt`
- 不要把具体源码文件列表堆在顶层

## 4. 构建方式怎么分层

这个项目实际跑通后，建议把构建固定成三层。

### 4.1 本地 Docker 开发构建

适合：

- 本机快速验证
- 不依赖服务器权限
- 不直接碰麒麟系统

入口：

- [docker/Dockerfile.dev](/d:/WorkSpace/Company/Tower/rma_manager/docker/Dockerfile.dev)
- [scripts/docker-build-dev.ps1](/d:/WorkSpace/Company/Tower/rma_manager/scripts/docker-build-dev.ps1)
- [scripts/docker-shell-dev.ps1](/d:/WorkSpace/Company/Tower/rma_manager/scripts/docker-shell-dev.ps1)

特点：

- 默认本地开发走 `linux/amd64`
- 只复现“编译环境”，不复现服务器内核、网卡、systemd、GPU
- 更适合作为开发加速层，而不是最终可信发布层

### 4.2 WSL/Linux 交叉构建

适合：

- 在 Linux/WSL 上做 ARM64 交叉编译
- 需要比 Docker 更贴近交叉工具链

入口：

- [scripts/dev_build_wsl_cross.sh](/d:/WorkSpace/Company/Tower/rma_manager/scripts/dev_build_wsl_cross.sh)

特点：

- 依赖 `aarch64-linux-gnu-gcc/g++`
- 默认构建目录：`build_wsl_cross_dev`
- 可跑单测，但正式测试仍建议显式指定 `tests/unit`

### 4.3 服务器原生构建

适合：

- 最终可信构建
- 验证老编译器和老 CMake 的兼容性
- 服务层最终启动验收

入口：

- [scripts/build_production.sh](/d:/WorkSpace/Company/Tower/rma_manager/scripts/build_production.sh)

特点：

- 直接使用服务器原生 GCC / CMake / Ninja
- 是本项目在麒麟 V10 上的正式基线

## 5. 为什么项目必须坚持离线依赖

本项目在服务器上成功稳定构建，一个核心经验就是：

**第三方依赖必须尽量离线化。**

本项目的做法：

- `deps_offline/` 中直接保存压缩包
- `third_party/CMakeLists.txt` 里统一校验和装配
- `FetchContent` 只允许读取本地包，不允许联网拉取

这样做的好处：

- 服务器不依赖公网
- 旧系统包管理器不可控时也不影响项目构建
- 多个项目可以共用一套离线依赖管理思路

适合推广到同服务器其他项目的依赖：

- `googletest`
- `yaml-cpp`
- `nlohmann/json`
- `spdlog`
- `sqlite amalgamation`
- header-only 库（如 `Crow`、`Asio`）

## 6. 旧环境适配的核心经验

### 6.1 编译器要选 GCC 7.3.0

对于这个服务器环境，项目最终稳定跑通的编译器是：

- `GCC 7.3.0 aarch64-linux-gnu`

不建议把 `CoreX` 的 `clang++` 当作默认编译器，除非项目本身明确依赖 GPU/CUDA 兼容层。

### 6.2 CMake 版本要接受 3.16.5

这个环境的原生 CMake 是 `3.16.5`。  
所以同类项目如果没有强依赖更高版本特性，建议：

- `cmake_minimum_required(VERSION 3.16)`

不要轻易把门槛抬到 3.20+，否则服务器原生构建直接失效。

### 6.3 避免依赖 `std::filesystem`

这是本项目最典型的旧环境坑之一。

在 GCC 7 上：

- `std::filesystem` 支持不稳定
- 常常直接编不过

本项目最终做法：

- 文件大小获取：`stat`
- 路径规范化：`realpath`
- 文件存在/普通文件检查：`stat`
- 文件可读检查：`ifstream`

对于同服务器其他项目，建议默认策略：

- 除非明确验证通过，否则不要在核心路径里用 `std::filesystem`

### 6.4 `-Werror` 会放大所有旧编译器告警

本项目为了保证质量启用了：

- `-Wall -Wextra -Wpedantic -Werror`

结果是很多“新编译器没事、老编译器报错”的问题都会暴露，例如：

- 结构化绑定中的未使用变量
- 布尔默认值引发的 merge 语义问题
- 老编译器更严格/更怪异的警告

这不是坏事，但意味着：

- 同类项目必须愿意为 GCC 7 做兼容修复

## 7. 重点说明：麒麟 V10 上的 CTest 问题

这是这份文档最重要的“经验结论”之一。

### 7.1 现象

在麒麟 V10 + CMake 3.16.5 环境上，本项目遇到过这种情况：

- 测试目标已正常编译
- `tests/unit/CTestTestfile.cmake` 里已经注册了大量 `add_test(...)`
- 但在顶层执行：

```bash
ctest --test-dir build_probe_tests --output-on-failure
```

仍然可能出现：

```text
No tests were found!!!
```

### 7.2 根因判断

从实际排查结果看，问题不在：

- GoogleTest 测试本身
- 测试注册本身
- 测试二进制本身

而在于：

- 旧版 `CTest` 在顶层递归发现/统计测试时行为不稳定

### 7.3 最终结论

**不要把顶层 `ctest` 作为麒麟 V10 上的唯一标准入口。**

本项目的正式测试入口固定为：

```bash
ctest --test-dir build_probe_tests/tests/unit --output-on-failure
```

或者：

```bash
cd build_probe_tests/tests/unit
ctest --output-on-failure
```

### 7.4 对其他项目的建议

如果其他项目也部署在同类环境上，建议直接采用下面策略：

1. 测试分目录明确组织：`tests/unit`、`tests/integration`
2. 构建脚本直接指定测试子目录，不依赖顶层递归
3. README 和部署基线里明确写死“正式测试入口”

也就是：

- 不写“运行 `ctest --test-dir build_xxx`”
- 而写“运行 `ctest --test-dir build_xxx/tests/unit`”

## 8. 服务层项目怎么组织更稳

这个项目的服务层结构也值得作为模板：

```text
src/rma_service/
├── include/service/
│   ├── app.h
│   ├── config.h
│   ├── auth.h
│   ├── dto.h
│   ├── *_api.h
│   └── ...
└── src/
    ├── main.cpp
    ├── app.cpp
    ├── config.cpp
    ├── dto.cpp
    ├── auth.cpp
    ├── target_api.cpp
    ├── job_api.cpp
    ├── profile_api.cpp
    ├── recovery_api.cpp
    └── ...
```

推荐原则：

- `main.cpp` 只做启动和参数解析
- `app.cpp` 负责应用生命周期
- 每类 REST/WS 接口拆成独立 `*_api.cpp`
- DTO 和业务逻辑分离

对其他项目的借鉴意义：

- 如果项目未来要从 CLI 演进成服务层，这种组织方式更容易扩展

## 9. 运行时数据应该怎么放

本项目已经踩过的坑：

- SQLite 文件散落在仓库根目录
- `rma.db`、`rma.db-shm`、`rma.db-wal` 被误提交

最终整理原则：

- 运行时数据统一放 `data/`
- 仓库只保留：
  - `data/.gitkeep`
- `.gitignore` 必须忽略：
  - `data/*.db`
  - `data/*.db-shm`
  - `data/*.db-wal`

这是其他项目非常值得直接复用的一条约束。

## 10. 部署资产应该怎么写

一个可复用的部署项目，至少要具备：

- `deploy/install.sh`
- `deploy/*.service`
- `deploy/service_config_example.yaml`
- `deploy/README.md`

同时部署文档必须和代码实现严格一致：

- 配置字段名一致
- 默认数据库路径一致
- 健康检查返回值一致
- 启动命令一致

不要让“部署文档”和“真实代码行为”长期漂移。

### 可以按需裁剪

## 12. 建议的重构顺序

如果要拿这套经验去重构同服务器上的其他项目，建议按下面顺序：

1. 先整理目录结构
2. 再整理顶层 CMake
3. 然后把依赖迁到 `deps_offline/`
4. 写正式构建脚本
5. 把测试迁到 `tests/unit`
6. 单独验证麒麟 V10 的 `ctest` 行为
7. 最后补部署资产和部署文档

不要一上来先做前端、Docker 或复杂服务化，那样很容易把基础问题掩盖掉。
