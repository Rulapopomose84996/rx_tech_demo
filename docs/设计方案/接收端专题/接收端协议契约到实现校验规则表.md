# 协议契约与实现校验规则表

## 1. 文档定位

本文不再讨论“为什么这么做”，只回答一个问题：

**接收端在实现时，协议字段到底怎么解析，单包到底怎么判，什么时候允许写当前 CPI，什么时候必须直接丢，落位和边界观察到底按什么规则执行。**

本文是“实现级对象模型与内存布局”“模块接口与 ownership 边界”“生命周期、切换条件与异常恢复闭环”的继续下钻文档。
目标不是追求协议解释上的面面俱到，而是形成一套**简单、稳定、易编码、易联调、主链路分支最少**的正式规则表。
这与当前整体冻结方向一致：接收端以 CPI 为最终输出单元，采用单 owner 顺序闭环、固定槽位一次落位、最少必要判断、只把最终 CPI 输出对象跨线程交给下游。

---

## 2. 适用范围

本文覆盖：

* UDP 包字段解析口径
* 单包合法性校验规则
* 当前 CPI 写入判定规则
* 固定槽位定位规则
* 重复写入处理规则
* 尾包观察与边界辅助规则
* 收口触发的协议相关观察规则
* reason code 最小建议集

本文不覆盖：

* 下游信号处理内部算法
* 控制面决策本体
* 长期归档与复杂回放平台
* 复杂异常解释系统

这与需求边界文档和主热路径文档保持一致。

---

## 3. 实现基准口径

这里先把一个容易混淆的问题说清楚。

协议补充说明文档里保留了“哪些内容原始协议事实已明确、哪些内容原始记录尚未明确”的分层；而主热路径方案后续又补充冻结了当前接收端实现可直接采用的解释口径，包括：

* 小端序
* `CPI` 按模 `65536` 递增
* `PRT` 在同一 CPI 内从 `0` 到 `N_PRT - 1`
* `I/Q` 采用 `int16/int16`
* “最后一个 PRT”由绑定快照中的 `N_PRT` 决定，尾包只作辅助

因此，**本文以当前已在主热路径方案中补充冻结的实现口径为准**；若后续上游冻结资产再次变更，应以新的冻结资产为准，并同步回写本文。

---

## 4. 设计原则

本文落表遵循下面 6 条：

1. **先做单包合法性，再做当前 CPI 写入判定。**
2. **主链路只保留三类判断：合法/非法，可写/不可写，首次/重复。**
3. **不把细异常分类扩展成主流程分支树。**
4. **不为“以后可能要分析”而在热路径挂重型状态。**
5. **不做补包、不做补零、不做猜测性修复。**
6. **规则优先保证实现简单和行为确定，再考虑边际性能优化。**

这与当前冻结方向完全一致。

## 5. 解析输入契约表

### 5.1 单包基础结构

当前实现按以下固定格式解析 UDP 包：

| 项目       | 规则                                               |
| -------- | ------------------------------------------------ |
| UDP 包总长度 | 固定 `2048` 字节                                     |
| 包信息区长度   | 固定 `16` 字节                                       |
| 数据区长度    | 固定 `2032` 字节                                     |
| 包头       | 固定 `0x55AAFF03`                                  |
| 包尾       | 普通包填 `0`；仅当前波位最后一个 PRT 的最后一个 UDP 包为 `0x55AAFF30` |
| 字段顺序     | 包头 / CPI / 通道 / PRT / 包计数 / 包尾                   |
| 内部定位键    | `(CPI, PRT, Channel, PacketIndex)`               |

这些规则来自当前协议事实和接收端最小解释约束。

### 5.2 字段解析口径

| 字段          |        偏移 |    长度 | 当前实现口径                                |
| ----------- | --------: | ----: | ------------------------------------- |
| 包头          |     `0~3` |    4B | 按小端读 `uint32`，期望 `0x55AAFF03`         |
| CPI         |     `4~5` |    2B | 按小端读 `uint16`                         |
| 通道          |     `6~7` |    2B | 按小端读 `uint16`，期望 `0~3`                |
| PRT         |     `8~9` |    2B | 按小端读 `uint16`                         |
| PacketIndex |   `10~11` |    2B | 按小端读 `uint16`，期望 `0~8`                |
| 包尾          |   `12~15` |    4B | 按小端读 `uint32`                         |
| payload     | `16~2047` | 2032B | 按 `I, Q, I, Q...` 解释，`I/Q` 均为 `int16` |

这里的小端序、`I/Q` 类型、`CPI/PRT` 规则和最后一个 PRT 的判定口径，本文均以主热路径方案补充冻结结论为准。

## 6. 单包合法性校验规则表

单包合法性校验只回答一个问题：

**这个包是否足够可信，值得进入“当前 CPI 写入判定”阶段。**

非法包必须在这里被拦住，不能再进入归属、落位和输出路径。这个约束来自需求边界文档。

### 6.1 校验顺序

建议固定顺序如下：

1. 长度校验
2. 包头校验
3. 通道范围校验
4. PacketIndex 范围校验
5. 绑定快照存在性校验
6. PRT 范围校验
7. 包尾规则校验
8. 字段组合一致性校验

这样做的原因很简单：

* 先做最便宜、最确定的检查
* 尽量在不依赖上下文的地方提前失败
* 减少后续无意义分支

### 6.2 正式规则表

| 规则编号  | 校验项            | 判定规则                                                       | 失败动作         |
| ----- | -------------- | ---------------------------------------------------------- | ------------ |
| V-001 | 长度             | `udp_len == 2048`                                          | 非法，直接丢弃      |
| V-002 | 包头             | `header == 0x55AAFF03`                                     | 非法，直接丢弃      |
| V-003 | 通道范围           | `channel in [0, 3]`                                        | 非法，直接丢弃      |
| V-004 | PacketIndex 范围 | `packet_index in [0, 8]`                                   | 非法，直接丢弃      |
| V-005 | 绑定快照           | 当前存在有效绑定快照                                                 | 无快照，不可继续主链路  |
| V-006 | PRT 范围         | `0 <= prt < N_PRT`                                         | 非法或不可写当前 CPI |
| V-007 | 包尾取值           | `tail == 0` 或 `tail == 0x55AAFF30`                         | 非法，直接丢弃      |
| V-008 | 包尾位置规则         | `tail == 0x55AAFF30` 时，必须同时满足“当前 CPI 最后一个 PRT + 当前通道最后一个包” | 非法，直接丢弃      |
| V-009 | 普通包尾规则         | 非最后位置的包，`tail` 必须为 `0`                                     | 非法，直接丢弃      |
| V-010 | payload 长度语义   | 数据区固定 `2032B`，不单独从协议字段再取变长长度                               | 固定结构，不进入变长分支 |

这里用“绑定快照 + `N_PRT`”来约束 `PRT` 范围和最后一个 PRT 判定，而不是反过来依赖尾包定义业务边界。这是当前已冻结的口径。

## 7. 字段组合一致性规则表

这里不做复杂化，只保留最有价值、最容易写、最能防污染的几条。

| 规则编号  | 组合项                     | 判定规则                                           | 说明                |
| ----- | ----------------------- | ---------------------------------------------- | ----------------- |
| C-001 | `tail` 与 `packet_index` | `tail == 0x55AAFF30` 时，`packet_index` 必须为 `8`  | 只有通道最后一个包才可能带有效尾包 |
| C-002 | `tail` 与 `prt`          | `tail == 0x55AAFF30` 时，`prt == N_PRT - 1`      | 尾包只允许出现在最后一个 PRT  |
| C-003 | `tail` 与普通位置            | 非最后一个 PRT 或非最后一个 `packet_index`，`tail` 必须为 `0` | 避免误把普通包当边界包       |
| C-004 | `channel` 与固定槽位         | `channel` 必须能映射到固定 4 通道之一                      | 防止槽位越界            |
| C-005 | `packet_index` 与固定槽位    | `packet_index` 必须能映射到每通道 9 包之一                 | 防止槽位越界            |

这里刻意**不**增加“更多更细”的组合树，因为当前主链路冻结方向就是最少必要判断，不把异常解释系统塑造成主架构。

## 8. 当前 CPI 写入判定规则表

这是本文最关键的一张表。

单包合法以后，第二阶段只回答：

**这个包是否允许写当前 `ACTIVE` CPI。**

当前主链路已冻结为同一时刻只保留一个 `ACTIVE` 可写 CPI，不维护“过去 / 当前 / 未来”多窗口并发可写模型。

### 8.1 正式规则表

| 规则编号  | 条件                                      | 结果             | 动作              |
| ----- | --------------------------------------- | -------------- | --------------- |
| A-001 | `pkt.cpi == active_cpi`                 | 命中当前 CPI       | 继续走槽位定位         |
| A-002 | `pkt.cpi == (active_cpi + 1) mod 65536` | 新 CPI 到来       | 触发 `CPI_SWITCH` |
| A-003 | 其他 `cpi`                                | 不命中当前 CPI      | 直接计数丢弃          |
| A-004 | 无 `active_ctx`                          | 当前无可写 CPI      | 直接丢弃或等待新上下文建立   |
| A-005 | 命中 recent closed                        | 迟到包 / 已关闭上下文命中 | 直接计数丢弃          |

这张表直接采用现有主热路径方案里的补充结论。

### 8.2 关键实现口径

1. **合法但不命中当前 CPI 的包，主链路直接丢弃。**
2. **不在主链路展开“过期 / 未来 / 越界 / 不可归属”的复杂业务树。**
3. **这些细分类只允许降级为内部 reason code。**

这与当前热路径方案和需求边界文档一致。

## 9. 固定槽位定位规则表

### 9.1 槽位组织规则

固定槽位采用三层静态组织：

* `PRT`
* `Channel`
* `PacketIndex`

不采用哈希表、`map`、链表、树、动态扩容容器。
目标不是追求极限技巧，而是让**索引路径固定、实现短、排障简单**。这与当前主热路径方案一致。

### 9.2 线性索引公式

建议正式冻结为：

```text
slot_index = ((prt * 4) + channel) * 9 + packet_index
```

其中：

* `prt in [0, N_PRT - 1]`
* `channel in [0, 3]`
* `packet_index in [0, 8]`

### 9.3 规则表

| 规则编号  | 条件         | 结果                |
| ----- | ---------- | ----------------- |
| S-001 | 三个下标都合法    | 计算唯一 `slot_index` |
| S-002 | 任一维越界      | 视为非法或不可写当前 CPI    |
| S-003 | `slot` 未命中 | 允许首次写入            |
| S-004 | `slot` 已命中 | 视为重复写入            |

## 10. 重复写入规则表

重复写入策略必须简单，不能搞成“比对 payload 后再决定覆写”的复杂分支。

### 正式规则

| 规则编号  | 条件          | 动作                          |
| ----- | ----------- | --------------------------- |
| D-001 | `slot` 首次命中 | 拷贝 payload，写长度，置命中位，更新摘要    |
| D-002 | `slot` 再次命中 | 记重复包计数，写轻量 reason code，立即返回 |
| D-003 | 重复写入        | 不覆盖旧 payload                |
| D-004 | 重复写入        | 不回滚已有位图与聚合摘要                |

这与当前冻结结论一致：首次写入获胜，重复包不覆盖原数据。

## 11. 尾包与边界观察规则表

这里必须单独写清楚，避免后续代码里偷偷把尾包当主边界。

### 11.1 基本口径

1. **尾包只作辅助一致性检查，不作业务边界定义依据。**
2. **最后一个 PRT 的判定，统一由绑定快照中的 `N_PRT` 决定。**
3. **`TAIL_OBSERVED` 只进入辅助观察位，不单独驱动完整性成立。**

这与主热路径方案和生命周期方案完全一致。

### 11.2 规则表

| 规则编号  | 条件                          | 动作                   |
| ----- | --------------------------- | -------------------- |
| T-001 | `tail == 0x55AAFF30` 且位置合法  | 置 `TAIL_OBSERVED`    |
| T-002 | `tail == 0x55AAFF30` 但位置不合法 | 直接判非法                |
| T-003 | 未看到尾包但完整位图已到齐               | 可按 `FULL_READY` 进入裁决 |
| T-004 | 看到了尾包但位图未完整                 | 不得直接视为 `COMPLETE_OK` |

## 12. 收口相关协议观察规则表

协议层和主链路真正相关的收口观察，只保留最小集合。

| 规则编号  | 观察项             | 条件                                           | 结果            |
| ----- | --------------- | -------------------------------------------- | ------------- |
| F-001 | `FULL_READY`    | `received_slot_count == expected_slot_count` | 可进入裁决         |
| F-002 | `CPI_SWITCH`    | 观察到 `pkt.cpi == active_cpi + 1 mod 65536`    | 当前 CPI 截止进入裁决 |
| F-003 | `WAVE_END`      | 控制面给出波位结束                                    | 当前 CPI 截止进入裁决 |
| F-004 | `TIMEOUT`       | 当前 CPI 等待超时                                  | 当前 CPI 截止进入裁决 |
| F-005 | `STOP`          | 受控停止                                         | 当前 CPI 截止进入裁决 |
| F-006 | `RESET`         | 受控复位                                         | 当前 CPI 截止进入裁决 |
| F-007 | `TAIL_OBSERVED` | 观测到合法尾包                                      | 只作辅助观察        |

这里不增加“轻微缺失 + 看见尾包 + 某些通道够了就算完整”之类灰区规则。
这样最稳，也最符合你要求的“在实现难度和性能之前取得平衡，不为了高级而复杂”。同时也符合当前文档里“主触发条件保持最小集合，尾包只作辅助”的冻结结论。

## 13. 推荐的最小 reason code 集合

reason code 只做内部计数、日志和联调，不反向塑造主链路结构。建议当前阶段只保留下面这组最小集：

| reason code                 | 含义               |
| --------------------------- | ---------------- |
| `INVALID_LEN`               | 长度错误             |
| `INVALID_HEADER`            | 包头错误             |
| `INVALID_TAIL`              | 包尾取值或包尾位置错误      |
| `INVALID_CHANNEL`           | 通道范围错误           |
| `INVALID_PRT`               | PRT 范围错误         |
| `INVALID_PACKET_INDEX`      | PacketIndex 范围错误 |
| `NO_ACTIVE_CONTEXT`         | 当前无可写 CPI        |
| `NOT_CURRENT_CPI`           | 不命中当前 CPI        |
| `LATE_TO_CLOSED_CPI`        | 命中已关闭 CPI        |
| `DUPLICATE_PACKET`          | 重复写入             |
| `BIND_SNAPSHOT_UNAVAILABLE` | 绑定快照不可用          |
| `INTERNAL_INCONSISTENCY`    | 内部状态不一致          |

这组已经够编码、够联调、够统计。
不建议在当前阶段把 reason code 扩成几十类，因为当前主链路已经明确要弱化复杂异常解释。

## 14. 推荐的实现顺序

实现时建议按下面固定顺序执行，避免流程分散：

### 14.1 单包处理顺序

1. 从 mbuf 读取前 16 字节
2. 解析出 `header / cpi / channel / prt / packet_index / tail`
3. 做单包合法性校验
4. 若非法：计数并丢弃
5. 读取 `active_ctx`
6. 做当前 CPI 写入判定
7. 若不可写：计数并丢弃
8. 计算 `slot_index`
9. 查位图判断首次/重复
10. 若重复：计数并返回
11. 首次写入：一次落位 + 更新长度 + 置命中位 + 更新聚合摘要
12. 检查是否命中 `FULL_READY / CPI_SWITCH / WAVE_END / TIMEOUT / STOP / RESET`
13. 若命中：进入裁决阶段

这个顺序与现有主链路冻结流程一致。

### 14.2 最小伪代码

```cpp
parse(pkt);

if (!validate_basic(pkt, bind)) {
    drop(INVALID_*);
    return;
}

admit = check_active_cpi(pkt, owner);

if (admit == DROP) {
    drop(reason);
    return;
}

if (admit == CPI_SWITCH) {
    finalize_active(CPI_SWITCH);
    open_next_active();
    // 当前包重试一次
}

slot = calc_slot_index(pkt.prt, pkt.channel, pkt.packet_index);

if (is_duplicate(slot)) {
    count_duplicate();
    return;
}

write_payload_once(slot, pkt.payload);
update_bitmaps_and_summary(slot, pkt);

if (should_finalize()) {
    finalize_active(trigger);
}
```

## 15. 明确不做的复杂化设计

为了防止后续实现越写越重，本文再明确禁止下面这些做法：

### 15.1 不做“多阶段复杂校验树”

不要把合法性校验拆成很多层可回退状态机。
当前协议足够固定，固定顺序检查就够了。

### 15.2 不做“先缓存再统一判定”

不要把一批包先存下来，再统一做合法性和归属判定。
当前主链路已经冻结为 owner 内顺序闭环，包到即判、判完即处理。

### 15.3 不做“payload 内容比对后决定覆盖”

重复包直接判重复，不比较 payload 是否一致。
这是最省实现、也最稳定。

### 15.4 不做“尾包驱动业务边界”

尾包只能辅助，不得主导 `COMPLETE_OK` 或最后一个 PRT 判定。

### 15.5 不做“热路径补零 / 补包 / 猜测修复”

缺失就是缺失，进入后续裁决，不在热路径做伪造修补。

### 15.6 不做“为了极限性能牺牲可读性”的微优化

当前最有价值的不是把几条判断压成极难维护的位技巧，而是：

* 规则顺序固定
* reason code 明确
* 返回路径清楚
* 代码可审、可测、可联调

你要求弱化性能优化在代码编写上的边际效益，这里就明确写死：
**当前阶段优先选择“短而清楚的固定流程”，而不是“为了少几个分支而牺牲可维护性”。**

## 16. 建议直接冻结的 10 条实现口径

1. **单包解析固定读取前 16 字节头信息区。**
2. **所有头字段当前实现统一按小端序解析。**
3. **单包合法性校验只保留：长度、包头、包尾、字段范围、字段组合一致性。**
4. **合法后才进入当前 CPI 写入判定。**
5. **当前 CPI 写入判定只保留三种主结果：写当前 CPI、触发 `CPI_SWITCH`、直接丢弃。**
6. **固定槽位索引统一按 `(PRT, Channel, PacketIndex)` 纯计算生成。**
7. **首次写入获胜，重复写入不覆盖原数据。**
8. **尾包只作辅助观察，不作主边界定义。**
