# 一、设计目标

这版生命周期草案只追求三件事：

* **状态少**
* **切换硬**
* **恢复闭环**

不追求：

* 多窗口并发可写
* 精细异常树驱动状态机
* 热路径自动补救
* “看起来聪明”的猜测性恢复

这符合现有主链路目标：接收端不是异常解释系统，而是稳定完成接收、落位、封口和输出的高确定性模块；主链路只保留最少必要判断；旁路能力从属，不得反压主链路。

---

# 二、总的生命周期模型

## 1）状态集合

建议严格冻结为这 5 个状态：

* `ACTIVE`
* `DECIDING`
* `SEALED`
* `TOMBSTONE`
* `RECYCLED`

这与现有文档完全一致。

---

## 2）每个状态的含义

### `ACTIVE`

当前唯一可写 CPI。

允许：

* 接收合法且命中当前 CPI 的包
* 首次写入固定槽位
* 更新位图、长度、聚合摘要
* 置收口触发位
* 进入裁决

不允许：

* 跨线程暴露可写引用
* 生成多个输出对象
* 被第二线程写

---

### `DECIDING`

进入裁决窗口，已禁止继续写入分片，正在做唯一裁决。

允许：

* 冻结读取摘要
* 归一化触发原因
* 形成唯一裁决结果
* 封口

不允许：

* 再接收新分片写入
* 再推进命中位图
* 重复进入裁决

---

### `SEALED`

裁决结果已确定，写入口已关闭，摘要已冻结。

允许：

* 生成最终 CPI 输出对象
* 推进边界
* 登记最近封口索引
* 进入 `TOMBSTONE`

不允许：

* 再写 payload
* 再改裁决结果
* 再改完整性摘要

---

### `TOMBSTONE`

业务上已结束，但对象尚不能立即复用。

允许：

* 等待输出对象消费完成
* 接收迟到包命中并稳定丢弃
* 等待 owner 完成最终回收检查

不允许：

* 恢复成 `ACTIVE`
* 再输出第二个对象
* 再参与主链路写入

---

### `RECYCLED`

已回池，可重新分配。

允许：

* 仅由对象池重新分配为新 CPI

不允许：

* 被视为旧 CPI 继续处理
* 被迟到包重新激活

---

# 三、总状态迁移图

推荐唯一状态迁移路径如下：

```text
NEW(from pool)
  -> ACTIVE
  -> DECIDING
  -> SEALED
  -> TOMBSTONE
  -> RECYCLED
```

只允许这条主链。

**禁止的逆向迁移：**

* `DECIDING -> ACTIVE`
* `SEALED -> ACTIVE`
* `TOMBSTONE -> ACTIVE`
* `RECYCLED -> anything old-key related`

这是最重要的约束。
因为你们当前主方案就是“一个当前可写 CPI + 封口即关闭 + 回收显式闭环”，不是支持回滚或重开上下文的复杂系统。

---

# 四、触发源设计

## 1）触发源冻结

进入裁决阶段的触发源，只保留这几个：

* `FULL_READY`
* `CPI_SWITCH`
* `WAVE_END`
* `TIMEOUT`
* `STOP`
* `RESET`
* `TAIL_OBSERVED`

其中 `TAIL_OBSERVED` 只作辅助观察位，不参与主原因排序。这个口径现有文档已经冻结。

---

## 2）主触发与辅助触发

### 主触发

真正驱动 `ACTIVE -> DECIDING` 的只有 6 个：

* `FULL_READY`
* `CPI_SWITCH`
* `WAVE_END`
* `TIMEOUT`
* `STOP`
* `RESET`

### 辅助触发

* `TAIL_OBSERVED`

`TAIL_OBSERVED` 的作用只是增强 owner 对“逻辑边界已靠近/已观测到尾迹”的确认，不允许单独作为完整性成立的依据。现有文档已经明确：逻辑边界为主，尾包只作辅助。

---

## 3）主触发优先级

建议保持和已有冻结一致：

1. `STOP / RESET`
2. `WAVE_END`
3. `CPI_SWITCH`
4. `TIMEOUT`
5. `FULL_READY`

`TAIL_OBSERVED` 不参与主排序。

这条优先级很重要，原因是：

* `STOP/RESET` 是外部强截止，必须最高优先
* `WAVE_END` 是业务边界强截止
* `CPI_SWITCH` 表示旧 CPI 的逻辑接收窗口已经结束
* `TIMEOUT` 是保底截止
* `FULL_READY` 反而是最自然、最低冲突的收口

---

# 五、单可写 CPI 原则

这是这版生命周期设计的根。

## 冻结结论

**owner 同一时刻只维护一个 `ACTIVE` 可写 CPI。**

不保留：

* “上一个 CPI 仍可补写”
* “当前 CPI 可写”
* “下一个 CPI 预创建并可写”

三窗口模型。

这和现有热路径方案完全一致：主链路同一时刻只保留一个 `ACTIVE` 的可写 CPI，新 CPI 明确到来后，旧 CPI 不再长时间拖尾等待，而是尽快进入裁决。

---

## 直接影响

1. 生命周期逻辑简单
2. `CPI_SWITCH` 语义清晰
3. 不需要多上下文共享写
4. 迟到包稳定落入“过期/已关闭命中”路径
5. 不会因为旧 CPI 拖尾而长期占住可写窗口

---

# 六、推荐的控制对象

建议新增一个很薄的生命周期控制器：

## `CpiLifecycleController`

它不是一个复杂状态机框架，只是把状态推进收口到一个地方。

建议接口：

```cpp
enum class TriggerKind : uint8_t {
    NONE,
    FULL_READY,
    CPI_SWITCH,
    WAVE_END,
    TIMEOUT,
    STOP,
    RESET,
    TAIL_OBSERVED
};

struct TriggerBits {
    uint8_t bits;
};

struct FinalizeIntent {
    TriggerKind primary;
    TriggerBits all_bits;
    uint64_t now_tsc;
};

class CpiLifecycleController {
public:
    bool mark_trigger(CpiContext& ctx, TriggerKind trig) noexcept;
    bool should_enter_deciding(const CpiContext& ctx) const noexcept;
    bool try_enter_deciding(CpiContext& ctx) noexcept;
    bool seal(CpiContext& ctx, CpiDecision decision, uint64_t now_tsc) noexcept;
    void enter_tombstone(CpiContext& ctx) noexcept;
    bool can_recycle(const CpiContext& ctx) const noexcept;
    void recycle(CpiContext& ctx, CtxPool& pool) noexcept;
};
```

## 为什么要单独放这个控制器

不是为了“架构优雅”，而是为了避免：

* `ProgressTracker` 顺手改状态
* `OutputDispatcher` 顺手回收
* `CpiAdmission` 顺手切换上下文
* 多模块同时改 `ctx.state`

生命周期推进点一旦分散，最后一定出事故。

---

# 七、状态允许动作表

这是实现里最应该落成表格的一部分。

## `ACTIVE`

允许：

* `write_packet`
* `update_progress`
* `mark_trigger`
* `try_finalize`

禁止：

* `dispatch_output`（除非先进入裁决并封口）
* `recycle`

---

## `DECIDING`

允许：

* `freeze_summary`
* `normalize_trigger`
* `make_decision`
* `seal`

禁止：

* `write_packet`
* `update_slot`
* `retry_finalize`

---

## `SEALED`

允许：

* `build_output`
* `push_output`
* `record_discard`
* `register_recent_closed`
* `enter_tombstone`

禁止：

* `write_packet`
* `change_decision`
* `change_summary`

---

## `TOMBSTONE`

允许：

* `late_packet_drop`
* `wait_release_token`
* `can_recycle_check`
* `recycle`

禁止：

* `build_output_again`
* `write_packet`
* `reopen`

---

## `RECYCLED`

允许：

* `pool_allocate_new`

禁止：

* 任何旧业务逻辑访问

---

# 八、推荐的状态字段

在上一轮 `CpiContext` 的基础上，生命周期相关字段建议至少保留：

```cpp
struct CpiLifeFields {
    CpiState state;
    CpiDecision decision;

    uint8_t trigger_bits;
    uint8_t flags;

    uint64_t create_tsc;
    uint64_t first_rx_tsc;
    uint64_t last_rx_tsc;
    uint64_t deciding_tsc;
    uint64_t seal_tsc;

    uint32_t output_ref;      // 0/1
    uint32_t finalize_count;  // 仅调试保护，正常应 <= 1
};
```

## 说明

### `trigger_bits`

记录所有已观测触发源。

### `deciding_tsc`

用于观测“裁决入口”延迟。

### `seal_tsc`

用于回收等待、迟到包观测、事件时间戳。

### `output_ref`

控制 `TOMBSTONE -> RECYCLED` 是否允许发生。

### `finalize_count`

正常运行期应该只会成功一次。
这个字段不是业务字段，是防实现错误的保护。

# 九、触发条件实现草案

## 1）`FULL_READY`

含义：按冻结规则完整到齐。

建议触发条件：

* `received_slot_count == expected_slot_count`

或等价的完整性位图成立。

注意：

* `TAIL_OBSERVED` 不能替代 `FULL_READY`
* 缺包但看到了尾包，仍然不是 `FULL_READY`

---

## 2）`CPI_SWITCH`

含义：当前处理到一个合法包，其 `pkt.cpi` 命中“下一个 CPI”。

现有热路径方案里，这本来就是收口主触发之一。

推荐行为：

1. 当前 `ACTIVE` 置 `CPI_SWITCH`
2. 立即尝试 finalize 当前 CPI
3. finalize 完成后创建新 `ACTIVE`
4. 再用当前包重试一次 admission 到新 CPI

**不要**：

* 让旧 `ACTIVE` 和新 `ACTIVE` 同时长期可写
* 让切换逻辑散落在 admission 模块里偷偷完成

---

## 3）`WAVE_END`

含义：当前波位业务边界已结束。

推荐行为：

* 当前 `ACTIVE` 若存在，直接进入 finalize
* 后续同波位旧键包一律走已关闭/过期路径
* 是否立刻创建下一个 `ACTIVE` 由新的控制面快照决定

---

## 4）`TIMEOUT`

含义：当前 `ACTIVE` 在限定时间内未达到完整收口，但继续等待收益不足。

推荐行为：

* 只作为保底截止
* 不用于驱动多次重试裁决
* 一旦命中就直接 finalize

**不建议**：

* “超时后继续等一点”
* “超时后先补零再裁决”
* “超时后保留半关闭态继续收包”

---

## 5）`STOP`

含义：受控停机。

推荐行为：

* 先停止继续收新 burst
* 当前 `ACTIVE` 立即以 `STOP` 为主因进入裁决
* 所有已投递输出继续等待 release 回执
* owner drain 完 release 后再退出

这对应你们需求里“受控启动、停止和复位，确保在途数据有一致的收尾语义”。

---

## 6）`RESET`

含义：受控复位。

推荐行为：

* 与 `STOP` 类似，但最终不是线程退出，而是把 owner 状态恢复到“空闲可重启”
* `ACTIVE` 先 finalize
* `TOMBSTONE` 全部等回收完
* 清空 recent closed ring、owner counters 的运行态部分
* 恢复到 `active_ctx = nullptr`

---

## 7）`TAIL_OBSERVED`

含义：观测到尾包辅助信号。

推荐行为：

* 只置位 `trigger_bits`
* 可用于增强日志或裁决摘要
* 不得单独触发 finalize
* 不得单独判定 `COMPLETE_OK`

---

# 十、推荐的 `ACTIVE -> DECIDING` 入口规则

建议把所有裁决入口统一成一个函数：

```cpp
bool try_finalize_active(OwnerState& owner,
                         TriggerKind primary,
                         uint64_t now_tsc) noexcept;
```

## 入口规则

### 只有满足下面之一才允许调用：

* `FULL_READY`
* `CPI_SWITCH`
* `WAVE_END`
* `TIMEOUT`
* `STOP`
* `RESET`

### 调用后第一步必须做：

* 检查 `active_ctx != nullptr`
* 检查 `ctx.state == ACTIVE`
* 失败则直接返回，不做任何补救性乱改

### 成功进入时：

* `ACTIVE -> DECIDING`
* `deciding_tsc = now_tsc`

这样做的好处是：

* 所有裁决入口统一
* 幂等保护统一
* 任何异常情况下都能明确知道“是否真正进入裁决”

---

# 十一、裁决结果判定规则

基于现有冻结口径，裁决逻辑建议只围绕 5 个布尔量展开：

* `is_complete`
* `has_missing`
* `forced_cutoff`
* `has_effective_data`
* `ctx_trusted`

这个方向和现有热路径文档一致：裁决逻辑围绕少量核心布尔量展开，而不是复杂异常树。

---

## 判定草案

### 1）`COMPLETE_OK`

条件：

* `is_complete == true`
* `ctx_trusted == true`

### 2）`INCOMPLETE_BUT_COMMITTABLE`

条件：

* `is_complete == false`
* `has_missing == true`
* `has_effective_data == true`
* `ctx_trusted == true`
* `forced_cutoff == false` 或不强调外部截止语义

### 3）`ABNORMAL_CUTOFF_COMMIT`

条件：

* `forced_cutoff == true`
* `has_effective_data == true`
* `ctx_trusted == true`

### 4）`DISCARD_INVALID`

条件：

* `has_effective_data == false`
  或
* `ctx_trusted == false`

这和现有裁决语义是一致的：
完整就是完整，不完整但可提交就如实输出不完整事实，外部截止单独区分，没业务价值或上下文不可信就丢弃。

---

# 十二、什么叫“上下文不可信”

这是当前阶段必须补齐但又不能写复杂的地方。

建议最小定义，不要发散。

## 建议最小判定条件

满足任一即可视为 `ctx_trusted == false`：

1. `expected_n_prt == 0`
2. `expected_slot_count == 0`
3. 关键绑定快照缺失
4. 状态字段非法
5. 聚合摘要明显自相矛盾
   例如 `received_slot_count > expected_slot_count`
6. finalize 前发现内部结构破坏
   例如位图和计数严重不一致，且不是简单重复包可解释

**不要**一开始把“不可信”扩展成几十种异常树。
否则生命周期设计会被拖回复杂异常解释系统。

---

# 十三、什么叫“足够业务价值”

这也是当前还没完全冻结的项，建议先给一个最小工程门槛。

## 建议最小门槛

满足任一即可视为 `has_effective_data == true`：

### 方案 A：按 PRT

* 至少收到 1 个完整 PRT

### 方案 B：按槽位比例

* `received_slot_count >= expected_slot_count * ratio_min`

### 方案 C：按通道完整度

* 至少 1 个 PRT 的 4 通道都形成有效摘要

在当前阶段，我更推荐 **方案 A：至少 1 个完整 PRT**。
原因是简单、容易解释、与下游的 CPI/PRT 语义更一致。

但这条你们后续仍需结合下游最小可用输入门槛再拍板。现有文档也明确，这一项还需要后续细化。

---

# 十四、迟到包处理闭环

这是生命周期方案里必须写死的一条。

## 冻结规则

命中 `SEALED / TOMBSTONE` 的同键包，稳定判为过期或命中已关闭上下文，不得重建旧 CPI。

---

## 推荐实现

结合上一轮对象模型，使用：

* `active_ctx`
* `recent_closed_ring`

处理逻辑：

### A. 命中 `active_ctx`

正常继续 admission / write

### B. 未命中 `active_ctx`，但命中 `recent_closed_ring`

* 计 `late_to_closed`
* 写轻量 reason code
* 丢弃

### C. 两者都不命中

* 按 `DROP_NOT_CURRENT` 处理

这样不需要长期保留多个可写/可查旧上下文，也不需要 map。

---

# 十五、推荐的切换流程

## 1）`CPI_SWITCH` 切换流程

```text
收到合法包
 -> admission 发现 pkt.cpi 命中新 CPI
 -> 给当前 active_ctx 置 CPI_SWITCH
 -> try_finalize_active(CPI_SWITCH)
 -> 若 should_output，则 dispatch output
 -> 当前 ctx 进入 TOMBSTONE
 -> 从 ctx_pool 申请 new_ctx
 -> 绑定新快照并置 ACTIVE
 -> 对当前包重试一次 admission + write
```

### 关键约束

* 旧 CPI 先 finalize，再切新 `ACTIVE`
* 当前包最多重试一次
* 不允许旧、新两个 CPI 长时间并行可写

---

## 2）`WAVE_END` 切换流程

```text
owner 收到 wave_end 事件
 -> 若 active_ctx 存在，置 WAVE_END
 -> try_finalize_active(WAVE_END)
 -> dispatch 或 discard
 -> 进入 TOMBSTONE
 -> 等待后续是否有新 wave 快照
```

---

## 3）`STOP` 流程

```text
收到 stop 命令
 -> 停止 poll 新 burst
 -> 若 active_ctx 存在，try_finalize_active(STOP)
 -> dispatch 或 discard
 -> owner 继续处理 release_spsc
 -> 所有 TOMBSTONE 均满足回收条件后退出
```

---

## 4）`RESET` 流程

```text
收到 reset 命令
 -> 停止新写入
 -> 若 active_ctx 存在，try_finalize_active(RESET)
 -> dispatch 或 discard
 -> drain 所有 release_spsc
 -> 回收所有 ctx/output
 -> clear recent_closed_ring
 -> owner 状态清零
 -> 返回可重启态
```

---

# 十六、异常恢复方案

这里不讲“系统级灾备”，只讲接收端 owner 内必须保证的恢复闭环。

---

## 异常一：重复触发裁决

### 现象

同一个 `ctx` 被多次调用 finalize。

### 处理

* 若 `state == DECIDING`：直接返回，记内部重复进入
* 若 `state == SEALED / TOMBSTONE`：直接返回，记重复触发
* 若 `state == RECYCLED`：记严重内部错误

这和现有裁决文档的幂等要求一致。

---

## 异常二：输出池申请失败

这是实现里必须正面回答的，不要回避。

### 推荐处理

**优先建议：输出池容量必须设计到“正常工况不失败”。**
但如果真的失败，建议：

* 当前 `ctx` 已经 `SEALED`
* 不能回到 `ACTIVE`
* 记录严重事件 `output_pool_exhausted`
* 将该 CPI 置为“已裁决但未成功分发”
* 直接进入 `TOMBSTONE`
* 由 owner 走受控丢弃统计路径

### 为什么不建议阻塞等待

因为会反压主链路，违背当前方向。
当前文档已经明确：主数据路径只保留最少必要队列，旁路和其他能力不得反压主链路。

### 为什么不建议深拷贝临时兜底

因为会把主链路引入突发大拷贝和复杂分支。

---

## 异常三：下游长时间不归还 `ReleaseToken`

### 现象

`TOMBSTONE` 长时间堆积，`ctx_pool` 逼近耗尽。

### 推荐处理

* owner 持续统计 `tombstone_depth`
* 超过水位时报警
* 不主动复用未释放的 `ctx`
* 当 `ctx_pool` 空时，新的 CPI 无法创建，则按系统级严重退化处理：

  * 新包继续计数丢弃
  * 主链路不崩、不乱写、不复用脏内存

### 关键原则

**宁可丢，也不能乱复用。**

---

## 异常四：旁路失败

### 处理

* 只记事件
* 不改主链路语义
* 不影响 `ACTIVE/DECIDING/SEALED/TOMBSTONE/RECYCLED`

这和需求冻结一致：旁路失败不得改变接收、校验、落位、裁决和输出语义。

---

## 异常五：迟到包大量涌入

### 处理

* 只查 `recent_closed_ring`
* 命中则计数丢弃
* 不重建旧上下文
* 不触发额外复杂恢复

---

## 异常六：内部状态损坏

### 现象

发现 `received_slot_count > expected_slot_count`、位图和计数严重冲突等。

### 处理

* 若仍在 `ACTIVE`，立即尝试 `RESET` 风格截止当前 `ctx`
* 以 `ctx_trusted = false` 进入 `DISCARD_INVALID`
* 封口、TOMBSTONE、回收仍要走完

**不要**：

* 悄悄修正
* 重置计数后继续收
* 让损坏上下文继续参与输出

---

# 十七、回收条件设计

## 1）进入 `TOMBSTONE` 的时机

无论输出还是作废，裁决结束后都进入 `TOMBSTONE`。这个口径与现有文档一致。

---

## 2）允许回收的最小条件

建议 `can_recycle(ctx)` 只检查三项：

1. `ctx.state == TOMBSTONE`
2. `ctx.output_ref == 0`
3. owner 不再需要它用于 recent closed
   这里建议 recent closed 只存 brief，不持有整个 `ctx`

满足后即可：

* `ctx.state = RECYCLED`
* 清理必要字段
* 回收到 `ctx_pool`

---

## 3）为什么 `SEALED` 不能直接回收

因为：

* 输出对象可能还没生成
* 下游可能还在读
* recent closed 还未登记完成

所以 `SEALED` 只是“已封口”，不是“可复用”。

---

# 十八、推荐的 owner 周期性动作

owner 主循环里建议有三个周期性动作：

## 1）`check_timeout_active(now_tsc)`

若当前 `ACTIVE` 超时，则置 `TIMEOUT` 并 finalize。

## 2）`drain_release_tokens()`

处理下游回收回执，把满足条件的 `TOMBSTONE` 送入 `RECYCLED`。

## 3）`flush_metrics()`

输出轻量统计，不阻塞主链路。

---

# 十九、推荐的关键计数器

为了验证生命周期闭环，建议至少保留这些 owner 级指标：

* `active_ctx_count`
  正常应始终为 `0/1`
* `enter_deciding_count`
* `seal_count`
* `tombstone_count`
* `recycled_count`
* `late_to_closed_count`
* `duplicate_finalize_count`
* `output_pool_exhausted_count`
* `ctx_pool_exhausted_count`
* `release_token_pending_count`

这些指标能直接告诉你：

* 状态是否重复推进
* 是否有封口不回收
* 是否有迟到包压力
* 是否有下游拖慢导致对象池吃紧

---

# 二十、这一版方案最重要的 8 条冻结结论

我建议你们把生命周期、切换条件与异常恢复闭环，直接冻结成下面这 8 条：

1. **同一时刻只允许一个 `ACTIVE` 可写 CPI。**
2. **生命周期只保留 `ACTIVE / DECIDING / SEALED / TOMBSTONE / RECYCLED` 五态。**
3. **进入裁决的主触发只保留 `FULL_READY / CPI_SWITCH / WAVE_END / TIMEOUT / STOP / RESET`，`TAIL_OBSERVED` 只作辅助。**
4. **任何生命周期推进只能由 owner 线程执行。**
5. **同一 CPI 只能完成一次有效裁决；重复触发只记内部异常，不重入主流程。**
6. **迟到包命中已封口 CPI 时稳定丢弃，不得重建旧上下文。**
7. **输出对象与上下文分离；下游只读输出对象，消费完成后用回执显式返还，由 owner 完成最终回收。**
8. **异常恢复不做补包、不做补零、不做状态回滚，只做受控封口、受控输出或受控作废。**
