# 接收端主热路径设计方案

## 1. 文档目的

本文围绕接收端主热路径，从需求约束推进到可实现方案，形成一套简单、高性能、可落地、与当前冻结口径一致的实现方案。

---

## 2. 当前已知冻结前提

### 2.1 主链路冻结方向

- 数据面采用 DPDK 原生 i40e PMD
- 数据面网口由 vfio-pci 接管
- 单 owner 线程 run-to-completion
- 同一 CPI 上下文单线程独占写入
- 分片归属成功后一次落位到 CPI 固定槽位
- 最终 CPI 输出对象通过 SPSC 交给下游

### 2.2 主链路最少必要判断

主链路只保留三类判断：

- 合法 / 非法
- 可写当前 CPI / 不可写当前 CPI
- 首次写入 / 重复写入

更细失败原因只允许降级为内部 reason code 或旁路调试信息。

### 2.3 当前已知协议事实

- 单个 UDP 包长度固定为 2048 字节
- 包信息区固定 16 字节
- 字段包括：包头、CPI 计数、通道计数、PRT 计数、包计数、包尾
- 一个 PRT 固定发送 36 个 UDP 包
- 共 4 个通道，每通道 9 个 UDP 包
- 包计数范围当前记录为 0~8
- 接收端可按 (CPI, PRT, Channel, PacketIndex) 形成 CPI 内部相对位置

### 2.4 当前不得擅自冻结的协议事项

当前仍未明确（后续方案不得写成已知事实）：

- 字节序
- CPI / PRT 计数递增规则
- I、Q 的数值类型与量化格式
- "当前波位最后一个 PRT"的判定边界

## 3. 本文收敛范围

**讨论范围：**

- 收包后的协议解析
- 单包合法性校验
- 是否允许写入当前 CPI 的判定
- 固定槽位定位
- 一次落位
- 位图 / 长度 / 聚合摘要推进
- 进入裁决前的最小收口条件

**不展开：**

- 下游信号处理内部实现
- 控制面决策本体
- 长期归档平台
- 复杂异常解释系统

## 6. 当前 CPI 的写入判定边界

### 6.1 收敛结论

#### 结论一：主链路同一时刻只保留一个 ACTIVE 的可写 CPI

owner 热路径不维护多 CPI 并发可写模型，不引入"过去 / 当前 / 未来"多个可写窗口的共享写复杂度。

#### 结论二：合法但不命中当前 CPI 的包，主链路直接丢弃

主链路统一处理为：

- 计数
- 写轻量 reason code
- 结束处理

不进入重组，不污染当前 CPI，不为了联调诉求引入复杂分支树。

#### 结论三：主链路不做补零修复

- 接收端主链路不补包
- 接收端主链路不补零伪造原始分片
- 缺失事实只体现在位图、长度、通道 / PRT / CPI 聚合摘要中
- 是否允许带缺失提交，由后续 CPI 裁决阶段依据冻结规则决定

缺失是"状态事实"，不是"接收端热路径即时修复动作"。

#### 结论四：当前阶段不在主链路展开精细业务语义分类

"过期 / 未来 / 越界 / 不可归属"等细分类，当前统一降级为内部 reason code。

热路径只保留：

- 合法 / 非法
- 可写当前 CPI / 不可写当前 CPI
- 首次写入 / 重复写入

### 6.2 对实现的直接影响

1. owner 线程只维护一个当前可写 CPI 上下文句柄。
2. 归属判定阶段的出口只保留两类：允许写当前 CPI / 不允许写当前 CPI。
3. 不允许写当前 CPI 的包：直接计数、记录轻量 reason code、立即返回。
4. 不在热路径引入"轻微缺失就补零"的写入逻辑。
5. 缺失与不完整信息只进入聚合摘要，留给后续裁决阶段决定。
6. 槽位只表达"是否收到真实分片"，不表达"是否由接收端伪造补零"。

---

## 7. 固定槽位索引规则

### 7.1 收敛结论

#### 结论一：固定槽位采用三层静态数组，不采用稀疏容器

- CPI 内部按 `PRT -> Channel -> PacketIndex` 组织固定槽位
- 热路径不使用哈希表、map、链表、树或动态扩容结构

#### 结论二：每个分片槽位只保存最少必要元数据和一份 payload

槽位只表达"这个分片是否真实收到，以及收到后的最少必要事实"。

默认最小集合为：

- 命中状态
- 实际有效载荷长度
- 必要时间戳或接收时间
- payload 存储区

#### 结论三：首次写入获胜，后续重复包不覆盖原数据

- 后续同索引包统一按重复包处理
- 只做重复计数或轻量事件记录
- 不覆盖原 payload
- 不回滚已有位图与聚合摘要

### 7.2 对实现的直接影响

1. CPI 上下文内的数据区做成连续内存块或规则数组，而不是对象图。
2. 槽位定位收敛为纯计算索引，而不是查表 + 分配。
3. 重复包路径极短：命中位已置位 → 计数 → 返回。
4. 首次写入路径固定：计算槽位地址 → 拷贝 payload → 置命中位 → 更新长度与最小摘要。

---

## 8. 槽位最小字段与位图推进

### 8.1 收敛结论

#### 结论一：分片槽位保持极简，只保存分片事实

- 槽位命中状态
- 实际有效长度
- 接收时间信息
- payload 区

槽位不承载复杂联调字段，不承载异常解释树，不承载高层统计。

#### 结论二：统计与诊断信息尽量上提到 CPI / owner 级摘要

重复包计数、非法包计数、丢弃计数等不挂在分片槽位上，默认放置在：

- CPI 级摘要
- owner 级计数器
- 必要旁路事件

#### 结论三：首次写入成功后，热路径只推进最小必要状态

- 分片命中位图
- Channel / PRT / CPI 已收计数或等价聚合摘要
- 用于判断是否需要进入裁决阶段的最小检查位

### 8.2 对实现的直接影响

1. 分片槽位结构小而定长，利于连续布局和缓存命中。
2. 热路径更新动作固定：拷贝 payload → 写长度 → 写时间 → 置命中位 → 更新上层摘要。
3. 统计口径集中到上层后，更便于做旁路导出和批量观测。
4. CPI 上下文总内存布局更可控，更容易预估对象池大小。

---

## 9. 进入裁决前的收口条件

### 9.1 收敛结论

#### 结论一：进入裁决阶段的主触发条件保持最小集合

- `FULL_READY`（完整到齐）
- `CPI_SWITCH`（新 CPI 到来，当前 CPI 需要截止）
- `WAVE_END`（波位结束）
- `TIMEOUT`（等待超时）
- `STOP / RESET`（受控停止或复位）

#### 结论二：尾包观测只作为辅助信号

- 逻辑边界为主
- 尾包只作辅助
- 辅助信号不反向支配主链路语义

#### 结论三：新 CPI 明确到来后，旧 CPI 不再长时间拖尾等待

- 不再长期等待旧 CPI 自然补齐
- 尽快进入裁决阶段
- 由裁决结果决定：提交、异常截止提交或作废

### 9.2 对实现的直接影响

1. 收口触发源做成固定小集合。
2. `TAIL_OBSERVED` 或等价尾包观测位只进入辅助摘要，不直接驱动完整性判断。
3. 当前 CPI 与下一 CPI 的切换逻辑明确：新 CPI 到来 → 旧 CPI 截止进入裁决 → 不维持长尾等待模式。
4. 热路径保持确定性，不因旧 CPI 缺失分片持续拖占可写窗口。

---

## 10. 四类裁决结果的判定规则

### 10.1 收敛结论

#### 结论一：`COMPLETE_OK` 只表达"按冻结规则完整到齐"

- 该 CPI 所需分片已按冻结规则到齐
- 完整性摘要满足完整提交条件
- 上下文可信

不夹带补偿语义，不允许把"轻微缺失但可接受"伪装成完整。

#### 结论二：`INCOMPLETE_BUT_COMMITTABLE` 表达"保留事实的不完整提交"

- 存在缺失
- 但仍形成有效业务数据
- 可由下游按不完整数据语义继续处理

接收端不负责把它修补成完整，只负责如实输出不完整事实。

#### 结论三：`ABNORMAL_CUTOFF_COMMIT` 用于"因外部截止因素被迫收口但仍可提交"

用于区分：

- 数据不是自然完整到齐后收口
- 而是由于 `STOP / RESET / WAVE_END / CPI_SWITCH / TIMEOUT` 等触发被迫截止
- 但当前 CPI 仍保留可交付价值

#### 结论四：`DISCARD_INVALID` 范围收紧

只在两类情况下进入：

- 当前 CPI 未形成足够业务价值
- 或上下文已不可信，继续输出风险大于收益

### 10.2 对实现的直接影响

1. 裁决逻辑围绕少量核心布尔量展开：是否完整、是否存在缺失、是否因外部截止收口、是否存在有效业务数据、上下文是否可信。
2. 不再需要"轻微缺失但算完整"这类灰区判定。
3. 下游契约清晰：
   - `COMPLETE_OK` = 完整事实
   - `INCOMPLETE_BUT_COMMITTABLE` = 不完整但保留事实
   - `ABNORMAL_CUTOFF_COMMIT` = 外部截止下的受控提交
   - `DISCARD_INVALID` = 不输出对象

---

## 11. 最终输出对象与 ownership 闭环

### 11.1 收敛结论

#### 结论一：最终 CPI 输出对象保持只读最小集

默认至少包含：

- CPI 标识与必要元数据
- 裁决结果
- 完整性 / 缺失摘要
- 指向已封口数据区的只读视图或等价引用

不向下游暴露 owner 内部可写状态。

#### 结论二：下游与接收端保持严格单向交接

- 下游只读取最终 CPI 输出对象
- 不回写接收端上下文
- 不参与接收端生命周期推进
- 不反向修改裁决结果

#### 结论三：输出池与上下文池分离，回收路径显式闭环

- 可写 CPI 上下文来自接收端上下文池
- 最终输出对象来自独立输出池
- 输出对象经 SPSC 交给下游后，由消费完成方显式归还输出池
- 可写 CPI 上下文在进入 `SEALED -> TOMBSTONE -> RECYCLED` 后按接收端规则回收

### 11.2 对实现的直接影响

1. 接收端内部可写上下文保持面向写入优化。
2. 输出对象针对跨线程只读消费单独设计，字段集更稳定。
3. 分别估算主链路热路径内存占用和下游消费滞后带来的输出积压上界。
4. 避免过早回收、重复回收、下游仍在读而接收端已复用内存。

## 12. 当前已形成的主方案摘要

截至目前，方案已收敛为：

1. 主链路同一时刻只保留一个可写 CPI。
2. 非法包或不命中当前 CPI 的包，主链路直接计数并丢弃。
3. 接收端主热路径不补零、不补包，只保留原始接收事实。
4. CPI 内部按 `PRT -> Channel -> PacketIndex` 三层静态数组组织固定槽位。
5. 每个槽位只保留最小必要元数据和一份 payload。
6. 首次写入获胜，重复包不覆盖原数据。
7. 首次写入后只推进最少必要位图与聚合摘要。
8. 收口触发条件保持最小集合，逻辑边界优先，尾包只作辅助。
9. 新 CPI 明确到来后，旧 CPI 不做长时间拖尾等待。
10. 四类裁决结果严格区分完整提交、不完整提交、异常截止提交和作废。
11. 最终输出对象只读化，与可写上下文分离。
12. 输出池与上下文池分离，生命周期与回收路径显式闭环。

## 13. 标准方案稿（可评审版）

### 13.1 方案结论

接收端主链路采用单 owner 顺序闭环模型。

owner 线程持续接收报文，完成协议解析、单包合法性校验、是否允许写入当前 CPI 的判定，并将命中当前 CPI 的分片一次性落到固定槽位。落位后只推进最少必要位图与聚合摘要。当满足完整到齐、CPI 切换、波位结束、超时、停止或复位等少量收口条件时，当前 CPI 进入裁决阶段。裁决结果限定为 `COMPLETE_OK`、`INCOMPLETE_BUT_COMMITTABLE`、`ABNORMAL_CUTOFF_COMMIT`、`DISCARD_INVALID` 四类。前三类生成只读 CPI 输出对象，经 SPSC 交给下游；下游只读，不回写接收端上下文。接收端上下文池与输出对象池分离，生命周期和回收路径显式闭环。

### 13.2 设计目标

本方案的目标不是构造复杂异常解释系统，而是在接收端边界内形成一条简单、高性能、可落地、易于验证的主链路。

主链路只保留最少必要判断：

- 合法 / 非法
- 可写当前 CPI / 不可写当前 CPI
- 首次写入 / 重复写入

更细失败原因统一降级为内部 reason code 或旁路调试信息，不反向塑造主链路对象模型。

### 13.3 模块边界

**接收端负责：**

- 网络入口收包
- 协议解析
- 单包合法性校验
- 当前 CPI 写入判定
- 固定槽位一次落位
- 位图与聚合摘要推进
- CPI 裁决、封口、输出准备
- 轻量统计与从属旁路

**接收端不负责：**

- 补包
- 伪造尾包
- 伪造完整性
- 在热路径补零修复原始分片
- 让下游回写接收端上下文

### 13.4 主链路执行模型

主链路同一时刻只保留一个可写 CPI。

owner 线程对每个报文按固定顺序处理：

1. 收包
2. 协议解析
3. 单包合法性校验
4. 判断是否允许写入当前 CPI
5. 计算 `(PRT, Channel, PacketIndex)` 相对位置
6. 定位唯一固定槽位
7. 若首次写入，则一次落位并推进最小摘要
8. 若重复写入，则计数并立即返回
9. 检查是否满足进入裁决的收口条件

非法包或不命中当前 CPI 的包直接计数丢弃，不进入重组和输出路径。

### 13.5 数据组织方案

CPI 内部固定槽位按 `PRT -> Channel -> PacketIndex` 三层静态数组组织。

默认不使用哈希表、map、链表或动态扩容容器。这样可以保证：

- 索引路径固定
- 分支少
- 缓存友好
- 实现简单

每个分片槽位只保存最少必要字段：

- 命中状态
- 实际有效长度
- 接收时间信息
- payload 区

统计与诊断信息尽量上提到 Channel / PRT / CPI / owner 级摘要中维护。

### 13.6 写入与重复包策略

同一 `(PRT, Channel, PacketIndex)` 槽位采用"首次写入获胜"策略。

- **首次写入成功：** 拷贝 payload，写长度与时间，置命中位，更新聚合摘要
- **后续同索引包：** 按重复包处理，只做计数或轻量事件记录，不覆盖原数据，不回滚摘要

### 13.7 缺失与不完整处理

接收端主热路径不补零、不补包。

缺失只作为接收事实体现在：

- 槽位未命中
- 位图缺口
- 通道 / PRT / CPI 聚合摘要缺失

是否允许带缺失提交，由裁决阶段决定，而不是由热路径即时修补。

### 13.8 收口与裁决

进入裁决阶段的主触发条件保持最小集合：

- `FULL_READY`
- `CPI_SWITCH`
- `WAVE_END`
- `TIMEOUT`
- `STOP / RESET`

尾包观测只作为辅助信号，不单独决定业务完整性。

新 CPI 一旦明确到来，旧 CPI 不再长时间拖尾等待，而是尽快进入裁决。

裁决结果统一为：

- `COMPLETE_OK`：按冻结规则完整到齐
- `INCOMPLETE_BUT_COMMITTABLE`：存在缺失，但仍保留有效业务数据
- `ABNORMAL_CUTOFF_COMMIT`：因外部截止因素被迫收口，但仍可提交
- `DISCARD_INVALID`：无足够业务价值，或上下文已不可信

### 13.9 输出对象与下游交接

只有前三类裁决结果允许生成最终 CPI 输出对象。

最终输出对象保持只读最小集，默认至少包含：

- CPI 标识与必要元数据
- 裁决结果
- 完整性 / 缺失摘要
- 指向已封口数据区的只读视图或等价引用

下游只读取该对象，不回写接收端上下文，也不参与接收端生命周期推进。

### 13.10 生命周期与回收

可写 CPI 上下文与最终输出对象分离：

- 可写 CPI 上下文来自接收端上下文池
- 最终输出对象来自独立输出池

可写 CPI 上下文按接收端生命周期推进：

- `ACTIVE`
- `DECIDING`
- `SEALED`
- `TOMBSTONE`
- `RECYCLED`

最终输出对象经 SPSC 交给下游后，由消费完成方显式归还输出池。

两类对象不混用，不共享生命周期。

### 13.11 工程优点

该方案的主要优点是：

1. 主链路短，状态少，容易保持确定性。
2. 数据结构静态，利于缓存命中与容量估算。
3. 热路径不做猜测性修复，不污染原始接收事实。
4. 裁决结果边界清晰，便于下游消费。
5. ownership 清晰，回收闭环明确，便于排障与验证。

### 13.12 尚需后续补充的实现细项

以下内容仍需在实现阶段继续细化，但不影响当前主方案成立：

1. `N_PRT` 的工程上界。
2. `TIMEOUT` 的工程阈值。
3. "足够业务价值"的最小门槛。
4. "上下文不可信"的最小判定条件。
5. 输出对象只读视图的具体封装形式。
6. 输出池容量按下游最坏滞后场景的估算方式。

## 14. 本文结论

当前方案已经满足"简单、高性能、可落地"的收敛目标，可以作为接收端主热路径的标准评审文档基础版本。

## 15. 补充结论

结论直接冻结如下：

**1. 字节序**

* 统一按**小端序**解析。
* 适用范围：包头、包尾、CPI 计数、通道计数、PRT 计数、包计数、payload 中的 I/Q。

**2. CPI 计数规则**

* 字段类型：`uint16`
* 同一 CPI 内所有 UDP 包的 `CPI` 值相同。
* 新 CPI 开始时，`CPI = (上一 CPI + 1) mod 65536`
* 回卷规则：`65535 -> 0`

**3. PRT 计数规则**

* 字段类型：`uint16`
* 同一 PRT 内所有 UDP 包的 `PRT` 值相同。
* 在同一 CPI 内，`PRT` 从 `0` 开始递增。
* 当前 CPI 的最后一个 PRT 固定定义为：`PRT = N_PRT - 1`
* 进入下一个 CPI 后，PRT 重新从 `0` 开始计数。

**4. 当前波位最后一个 PRT 的判定边界**

* 不靠尾包反推。
* 统一由**创建该 CPI 上下文时绑定的波位安排快照中的 `N_PRT`**决定。
* 判定规则：`prt == N_PRT - 1` 即命中当前波位最后一个 PRT。
* 尾包 `0x55AAFF30` 只作为**辅助一致性检查**，不是边界定义依据。

**5. I/Q 数据类型与复数样点格式**

* `I = int16`
* `Q = int16`
* 单个复数样点大小 = **4 Bytes**
* payload 内按 `I, Q, I, Q ...` 解释。

**7. 接收端当前 CPI 写入判定可直接采用**

* `pkt.cpi == active_cpi`：允许继续判定写入当前 CPI
* `pkt.cpi == (active_cpi + 1) mod 65536`：视为新 CPI 到来，可触发 `CPI_SWITCH`
* 其他 `cpi`：默认不命中当前 CPI，直接计数丢弃

**8. PRT 合法范围**

* `0 <= prt < N_PRT`
* 超出范围：直接判非法或不可写当前 CPI

**9. PacketIndex 合法范围**

* `0 <= packet_index <= 8`
* 超出范围：直接判非法

**10. 最终决策口径**

* 协议契约现在按下面四条正式闭环：

  * **小端序**
  * **CPI 按模 65536 递增；PRT 在 CPI 内从 0 到 N_PRT-1**
  * **复数样点格式为 int16/int16，单样点 4 Bytes**
  * **最后一个 PRT 由绑定快照中的 N_PRT 决定，尾包仅辅助**
