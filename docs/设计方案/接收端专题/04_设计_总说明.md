## 2. 一句话说明

接收端是一个以 CPI 为最终输出单元、由单 owner 线程顺序完成收包、校验、归属、固定槽位落位、裁决封口和输出准备的数据面入口模块。

---

## 3. 当前冻结的主链路

```text
DPDK RX
  -> owner 线程
  -> 协议解析
  -> 单包合法性校验
  -> 判断是否允许写入当前 CPI
  -> 分片一次落位到 CPI 固定槽位
  -> 更新位图 / 长度 / 状态 / 进度
  -> CPI 裁决与封口
  -> 最终 CPI 输出对象
  -> SPSC
  -> 下游消费者
```

主链路中不存在默认的“解析线程池 / 归属线程池 / 重组线程池”设定。
只有最终 CPI 输出对象允许跨线程交接。

---

## 4. 模块职责概览

接收端负责：

* 接收 UDP 数据报文
* 解析协议字段
* 完成单包合法性校验
* 判断报文是否允许写入当前 CPI
* 在 owner 内完成固定槽位落位和进度推进
* 对 CPI 做裁决、封口和输出
* 产生轻量统计、事件和必要旁路信息

接收端不负责：

* 信号处理算法
* 控制面决策
* 显控业务解释
* 复杂长期归档系统

---

## 5. 数据组织方式

### 5.1 最终输出单元

接收端对外最小输出单元固定为 CPI。

### 5.2 内部组织粒度

接收端内部继续保留：

* PRT
* 通道
* 分片

这些粒度只用于：

* 槽位定位
* 位图推进
* 完整性统计
* 输出摘要生成

### 5.3 一次落位原则

分片归属成功后，payload 只拷贝一次，直接落到 CPI owner 固定槽位。

后续只维护：

* 分片位图
* 通道状态
* PRT 状态
* CPI 聚合摘要
* 输出对象引用关系

### 5.4 为什么这样组织

这种组织方式的核心价值在于：

* 原始 IQ 数据只保留一份
* 主链路避免重复拷贝和大块副本
* PRT / 通道 / 分片仍可提供足够的内部定位和完整性信息
* CPI 级输出契约可以保持稳定

---

## 6. 判断与失败路径

主链路只保留最少必要判断：

* 合法 / 非法
* 可写当前 CPI / 不可写当前 CPI
* 首次写入 / 重复写入

失败路径统一口径为：

> owner 内直接计数并丢弃，必要时写轻量 reason code；不进入重组，不污染当前 CPI。

“过期 / 未来 / 越界 / 不可归属”等可以作为内部 reason code 存在，但不再作为主文档叙事中心。

---

## 7. 裁决与输出

当 CPI 达到收口条件时，由同一 owner 完成：

* 读取聚合摘要
* 形成唯一裁决结果
* 关闭写入口
* 推进边界
* 生成最终 CPI 输出对象

裁决结果统一为：

* `COMPLETE_OK`
* `INCOMPLETE_BUT_COMMITTABLE`
* `ABNORMAL_CUTOFF_COMMIT`
* `DISCARD_INVALID`

最终交付给下游的是只读的 CPI 输出对象，而不是可写接收上下文。

---

## 8. 观测与旁路

主链路只做轻量计数和必要事件打点。

旁路、录制、日志、导出都必须满足：

* 只消费最终对象或必要副本
* 不接触 owner 可写上下文
* 不反压主链路

主链路观测重点是：

* 收包速率
* 字节速率
* 非法包计数
* 重复包计数
* 丢弃计数
* CPI 完整提交数
* CPI 不完整提交数
* CPI 作废数

---

## 9. 如何理解这份设计

如果不看代码，最容易理解接收端的方式是把它拆成四句话：

1. 它先把高速 UDP 原始输入稳定接住。
2. 它只做最少必要判断，防止当前 CPI 被污染。
3. 它把归属成功的分片一次落到固定槽位，并逐步推进 CPI 进度。
4. 它在 owner 内完成 CPI 级裁决、封口和输出，然后把只读对象交给下游。

---

## 10. 本文结论

接收端设计的重点，不是“把每一种异常都展开解释”，而是把一条简单且稳定的主链路写清楚：

* CPI 级输出
* 单 owner 闭环
* 固定槽位一次落位
* 最少必要判断
* 轻量观测和从属旁路
