# 一、设计目标与约束

这版草案只追求三件事：

1. **热路径短**
2. **内存布局规则**
3. **生命周期闭环**

不追求：

* 热路径复杂异常解释
* 每槽位重型元数据
* 输出阶段再做大块复制
* 为调试而污染对象模型

这符合你们现有冻结结论：主链路只保留“合法/非法、可写/不可写、首次/重复”三类判断；固定槽位一次落位；输出对象只读化；输出池与上下文池分离。

---

# 二、对象清单

实现上只保留下面 6 类对象。

## 1）报文解析视图 `ParsedPacketView`

**作用**：单包解析后的栈上临时视图。
**生命周期**：只活在当前报文处理函数内。
**是否入池**：否。
**是否跨线程**：否。

它只保存：

* `cpi`
* `prt`
* `channel`
* `packet_index`
* `is_tail`
* `payload_ptr`
* `payload_len`
* `rx_tsc`
* `reason_code`（只在失败路径写）

这个对象**不落堆、不入队、不挂上下文**。
处理完当前报文后，连同报文缓冲对象（message buffer，mbuf）一起结束。

---

## 2）绑定快照 `BoundWaveSnapshotLite`

**作用**：在 CPI 上下文创建时，冻结当前 CPI 的解释依据。
**生命周期**：跟随 `CpiContext`。
**是否跨线程**：只读可见，但通常只在 owner 内使用。

建议只保留最小字段：

* `wave_id`
* `schedule_version`
* `n_prt`
* `timeout_tsc`
* `channel_count`（固定 4，也可不存）
* `packets_per_channel`（固定 9，也可不存）
* `packets_per_prt`（固定 36，也可不存）

这样做是因为裁决阶段必须使用**创建该 CPI 时绑定的依据**，而不是“当前最新快照”。

---

## 3）可写 CPI 上下文 `CpiContext`

这是**主对象**。

**作用**：承载一个 CPI 从创建、接收、落位、摘要推进、裁决、封口到可回收的全过程。
**生命周期状态**：`ACTIVE -> DECIDING -> SEALED -> TOMBSTONE -> RECYCLED`。
**ownership**：从创建到封口只允许归属线程（owner）写。

这个对象包含两部分：

* **前部小元数据区**
* **后部大 payload 区**

后面详细展开。

---

## 4）只读输出对象 `CpiOutput`

**作用**：跨线程交给下游。
**生命周期**：从输出池申请，入 SPSC，消费后归还。
**ownership**：生成在 owner，读取在下游，回收确认最终回到 owner。

它只包含：

* CPI 标识与必要元数据
* 裁决结果
* 完整性/缺失摘要
* 指向 `CpiContext` 已封口数据区的只读视图
* 回收句柄

它**不持有可写上下文语义**，也不允许下游回写接收端状态。

---

## 5）归属线程本地状态 `OwnerState`

**作用**：归属线程本地总控对象。
**生命周期**：线程全程驻留。
**是否跨线程**：否。

建议包含：

* `active_ctx`：当前唯一可写 CPI
* `ctx_pool`
* `output_pool`
* `tx_spsc`：输出给下游
* `release_spsc`：下游释放回执返还给 owner
* `recent_closed_ring`
* owner 级计数器
* 当前波位安排快照句柄

---

## 6）最近封口索引环 `RecentClosedRing`

**作用**：用极小代价处理迟到包/已关闭上下文命中。
**生命周期**：owner 常驻。
**是否跨线程**：否。

每个槽位只保留：

* `wave_id`
* `cpi_id`
* `seal_tsc`
* `decision`
* `schedule_version`

**不要保留整块旧 payload。**

这样即便旧 `CpiContext` 已经回收，也能稳定判定“这是迟到包/已关闭 CPI 命中”，而不必把整块大上下文长期挂着等迟到包。这个思路和“命中 `SEALED/TOMBSTONE` 的迟到包稳定判定为过期”是一致的，但实现更轻。

---

# 三、最关键的内存布局结论

## 结论一：**不要把每个分片做成一个对象**

不要这样：

* `vector<Prt>`
* `Prt` 里再 `vector<Channel>`
* `Channel` 里再 `vector<PacketSlot>`
* 每个 `PacketSlot` 再挂 payload 指针

这会带来：

* 指针跳转多
* 初始化复杂
* 缓存局部性差
* 池化和回收难看
* 热路径有对象图遍历味道

你们当前方案已经明确要求固定槽位、静态数组、纯计算索引，不应回退到对象图。

---

## 结论二：**采用“两段式布局”**

每个 `CpiContext` 采用：

### A. 前部元数据区

放热路径反复访问的内容：

* CPI 头字段
* 生命周期状态
* 触发位
* 聚合摘要
* `PRT/Channel` 位图与计数
* `slot_valid_bytes[]`
* 可选 `slot_rx_tsc[]`

### B. 后部 payload 连续区

放所有真实分片 payload。

这样做的好处是：

* 重复包判定只摸前部位图，不会碰大 payload 区
* 写入时通过线性地址直接 `memcpy`
* 裁决阶段读取摘要不需要扫大块 payload
* 缓存行为更可控

---

## 结论三：**payload 槽位步长固定为 2048 字节**

当前每包数据区有效载荷是 `2032` 字节。建议**不按 2032 做步长，而按 2048 做步长**：

* 有效数据拷入前 `2032` 字节
* 剩余 `16` 字节留空/保留
* 每槽位地址步长固定为 `2048`

这样有三个直接收益：

1. **地址计算极简**
   `offset = slot_index << 11`

2. **2KB 对齐天然规整**

3. **后期如果要做 SIMD/预取/页对齐，也更顺手**

代价只是每槽位浪费 16 字节。
一个 PRT 36 槽，只浪费 `36 * 16 = 576` 字节，完全可接受。

# 四、`CpiContext` 的推荐结构

下面这版是**可直接落 C++17 结构设计**的草案，不是最终代码。

```cpp
enum class CpiState : uint8_t {
    ACTIVE,
    DECIDING,
    SEALED,
    TOMBSTONE,
    RECYCLED
};

enum class CpiDecision : uint8_t {
    NONE,
    COMPLETE_OK,
    INCOMPLETE_BUT_COMMITTABLE,
    ABNORMAL_CUTOFF_COMMIT,
    DISCARD_INVALID
};

struct BoundWaveSnapshotLite {
    uint32_t wave_id;
    uint16_t schedule_version;
    uint16_t n_prt;
    uint64_t timeout_tsc;
};

struct CpiHotHeader {
    uint32_t window_id;
    uint32_t wave_id;
    uint16_t cpi_id;
    uint16_t expected_n_prt;
    uint16_t expected_slot_count;   // n_prt * 36
    uint16_t received_slot_count;

    uint16_t duplicate_count;
    uint16_t reserved0;

    uint64_t first_rx_tsc;
    uint64_t last_rx_tsc;
    uint64_t seal_tsc;

    CpiState state;
    CpiDecision decision;
    uint8_t trigger_bits;   // FULL_READY/CPI_SWITCH/WAVE_END/TIMEOUT/STOP/RESET/TAIL
    uint8_t flags;          // has_tail / output_enqueued / ctx_untrusted ...
};

struct PrtSummary {
    uint16_t ch_pkt_bitmap[4];  // 每通道 9 bit，uint16 足够
    uint8_t  ch_recv_count[4];  // 0~9
    uint8_t  ready_channel_count;
    uint8_t  recv_packet_count; // 0~36
    uint8_t  flags;             // seen_tail / completed / reserved
};

struct CpiAggregate {
    uint16_t ready_prt_count;
    uint16_t missing_slot_count;
    uint16_t repeated_slot_count;
    uint16_t invalid_reason_count;

    uint16_t first_seen_prt;
    uint16_t last_seen_prt;
    uint16_t max_contiguous_ready_prt;
    uint16_t reserved0;
};

struct CpiColdTail {
    uint32_t output_ref;          // 0/1，必要时可扩成引用计数
    uint32_t pool_index;
    uint32_t last_reason_code;
    uint32_t reserved0;
};

struct CpiContext {
    alignas(64) CpiHotHeader hot;
    BoundWaveSnapshotLite bind;
    CpiAggregate agg;

    // N_PRT_MAX 由工程配置给出
    PrtSummary prt[kMaxPrtPerCpi];

    // 每槽位最小事实
    uint16_t slot_valid_bytes[kMaxPrtPerCpi * 36];

    // 如联调需要可打开；默认建议关闭
    // uint64_t slot_rx_tsc[kMaxPrtPerCpi * 36];

    // 每槽位固定 2048 步长，前 2032 字节写有效数据
    alignas(2048) uint8_t payload[kMaxPrtPerCpi * 36][2048];

    CpiColdTail cold;
};
```

# 五、为什么这套结构是合适的

## 1）热字段和冷字段分开

热路径反复访问的是：

* `state`
* `expected_slot_count`
* `received_slot_count`
* `trigger_bits`
* `prt[prt].ch_pkt_bitmap[ch]`
* `prt[prt].recv_packet_count`
* `slot_valid_bytes[slot]`

这些都在前部元数据区，缓存友好。

而冷字段：

* `last_reason_code`
* `pool_index`
* 输出回收相关字段

放尾部，不污染主热路径。

---

## 2）每槽位不单独存 `hit` 字段

命中事实由 **位图** 表示：

* 每个通道 9 个包
* 用 `uint16_t` 的低 9 位表达够了

所以重复包判断只需要：

```cpp
if (prt[prt_idx].ch_pkt_bitmap[ch] & (1u << pkt_idx)) { ... }
```

这样比每槽位一个 `bool hit` 更规整，也更接近你们文档里“位图推进”的冻结口径。

---

## 3）每槽位默认只保留最少必要事实

建议默认只保留：

* `slot_valid_bytes`
* payload

**默认不存每槽位时间戳。**

理由：

* 热路径里每槽位再写一个 `uint64_t rx_tsc`，收益不高
* 大多数裁决不需要“每片接收时刻”
* 你们现在真正需要的是 CPI 级首包/末包时间、超时判断、轻量观测

所以推荐：

* `CpiHotHeader.first_rx_tsc`
* `CpiHotHeader.last_rx_tsc`

够用。
只有在联调阶段明确需要“单槽位时间”，再用编译开关打开 `slot_rx_tsc[]`。

这符合“主链路不为以后分析而加重型状态”的方向。

---

# 六、固定槽位索引规则

这是实现里最关键的一条，必须写死。

## 1）线性槽位索引

定义：

```text
slot_index = ((prt * 4) + channel) * 9 + packet_index
```

前提：

* `channel in [0, 3]`
* `packet_index in [0, 8]`
* `prt in [0, n_prt - 1]`

这和现有协议事实、固定槽位组织方式完全对齐。

---

## 2）payload 地址

定义：

```text
payload_ptr = &ctx->payload[slot_index][0]
```

因为每槽位步长固定 2048，所以本质就是：

```text
payload_base + (slot_index << 11)
```

---

## 3）摘要地址

定义：

```text
prt_summary = &ctx->prt[prt]
bitmap      = &ctx->prt[prt].ch_pkt_bitmap[channel]
valid_bytes = &ctx->slot_valid_bytes[slot_index]
```

整个写入路径没有哈希、没有 map、没有链表、没有查表分配。
完全是**边界检查 + 位运算 + 固定地址 memcpy**。这正是你们当前热路径该有的样子。

---

# 七、热路径写入规则

## 1）只维护一个 `ACTIVE` 上下文

owner 线程只保留一个当前可写 `active_ctx`。
不维护“过去/当前/未来”多可写窗口。

---

## 2）单包处理顺序

建议固定为：

1. 从网卡收包
2. 解析到 `ParsedPacketView`
3. 单包合法性校验
4. 判断是否命中当前 `active_ctx`
5. 不命中则丢弃/计数
6. 命中则计算 `slot_index`
7. 查位图判断首次/重复
8. 首次则一次落位
9. 更新最少必要摘要
10. 判断是否触发裁决

这和现有冻结主链路一致。

---

## 3）首次写入路径

首次写入只做下面几件事：

```text
memcpy(payload_ptr, pkt.payload, pkt.payload_len)
slot_valid_bytes[slot_index] = pkt.payload_len
bitmap |= mask
++ch_recv_count
++recv_packet_count
++ctx.hot.received_slot_count
更新 first_rx_tsc / last_rx_tsc
必要时更新 ready_channel_count / ready_prt_count
必要时置 FULL_READY / TAIL_OBSERVED
```

---

## 4）重复包路径

重复包命中后：

* `duplicate_count++`
* owner 级计数++
* 写轻量 `reason_code`
* 立即返回

**不要覆盖旧 payload。**
这点必须坚持。

---

## 5）触发裁决的条件

触发位建议直接放 `trigger_bits`：

* `FULL_READY`
* `CPI_SWITCH`
* `WAVE_END`
* `TIMEOUT`
* `STOP`
* `RESET`
* `TAIL_OBSERVED`

其中 `TAIL_OBSERVED` 只作辅助位，不作主边界定义。

---

# 八、`CpiOutput` 的推荐结构

只读输出对象建议尽量小。

```cpp
struct CpiReadOnlyView {
    const uint8_t* payload_base;
    const uint16_t* slot_valid_bytes;
    const PrtSummary* prt_summary;
    uint16_t n_prt;
    uint16_t slot_count;
    uint16_t payload_stride;   // 固定 2048
};

struct CpiOutput {
    uint32_t wave_id;
    uint32_t window_id;
    uint16_t cpi_id;
    uint16_t schedule_version;

    CpiDecision decision;
    uint8_t trigger_bits;
    uint8_t flags;

    uint16_t received_slot_count;
    uint16_t missing_slot_count;
    uint16_t duplicate_count;
    uint16_t ready_prt_count;

    uint64_t first_rx_tsc;
    uint64_t last_rx_tsc;
    uint64_t seal_tsc;

    CpiReadOnlyView view;

    CpiContext* sealed_ctx;   // 仅用于回收，不给下游写权限
    uint32_t output_id;
};
```

## 输出对象设计原则

### 原则一：只读

它是给下游消费的，不是给下游回写接收端上下文的。

### 原则二：不复制整块 payload

如果输出时再复制整个 CPI，会直接把这套设计打穿。

### 原则三：只带必要元数据 + 只读视图

下游要看数据，就按视图读。
不需要把接收端内部全部对象树暴露出去。

---

# 九、生命周期与回收闭环

这是这套方案里第二关键的部分。

## 1）为什么不能“输出对象直接复制 payload”

因为那会让输出阶段出现一整次 CPI 级大拷贝，直接破坏“主链路一次落位、原始数据只保留一份”的原则。

---

## 2）为什么不能“输出对象引用上下文后，马上回收上下文”

因为下游还在读。

---

## 3）推荐闭环

推荐这样做：

### A. 封口时

* `ACTIVE -> DECIDING -> SEALED`
* 冻结摘要
* 生成 `CpiOutput`
* `ctx->cold.output_ref = 1`
* `CpiOutput.sealed_ctx = ctx`
* 推入输出 SPSC
* 把 `{wave_id, cpi_id, seal_tsc, decision}` 写入 `RecentClosedRing`
* `ctx.state = TOMBSTONE`

### B. 下游消费完成后

下游不直接回收 `ctx`，而是把 `output_id + ctx_pool_index` 放入**回收回执 SPSC** 返还给 owner。

### C. owner 处理回执

owner 收到回执后：

* 释放 `CpiOutput` 回输出池
* `ctx->cold.output_ref = 0`
* 若 `ctx.state == TOMBSTONE`，则回收到 `ctx_pool`

---

## 4）为什么我建议加一个“回收回执 SPSC”

因为这是最稳的。

如果不用回执环，常见替代方案有两个：

### 方案 A：下游直接写原子标志

问题：

* 引入跨线程原子写
* owner 还得扫 tombstone 列表
* 状态一致性更脆

### 方案 B：输出时深拷贝 payload

问题：

* 代价太大
* 违背当前主方案

所以，**一个轻量的“回收回执 SPSC”是值得的**。
它不是主数据路径，不会改变主链路结构，只是把跨线程释放动作做成显式闭环。
这也更符合“上下文池与输出池分离，回收路径显式闭环”的冻结要求。

---

# 十、最近封口索引环 `RecentClosedRing`

建议固定长度 8 或 16。

```cpp
struct ClosedCpiBrief {
    uint32_t wave_id;
    uint16_t cpi_id;
    uint16_t schedule_version;
    uint64_t seal_tsc;
    CpiDecision decision;
};
```

owner 本地维护一个环：

```cpp
ClosedCpiBrief recent_closed[8];
uint32_t recent_closed_head;
```

## 它的作用

当报文不命中当前 `active_ctx` 时，快速判断：

* 是不是刚封口的旧 CPI 迟到包
* 是不是已关闭上下文命中
* 否则就是 generic “不命中当前 CPI”

## 它的好处

不需要为了迟到包识别而：

* 保留整块旧 payload
* 维护复杂多窗口可写模型
* 建立 map

这非常适合你们当前“一个 `ACTIVE` + 少状态 + 高确定性”的路子。

---

# 十一、对象池设计

## 1）`CpiContextPool`

**建议**：固定深度、NUMA 本地、启动期一次性分配。
使用非一致内存访问（non-uniform memory access，NUMA）本地大页内存。

### 推荐规则

* `N_PRT_MAX` 编译期固定
* `CTX_POOL_DEPTH` 启动参数给定
* 启动时校验实际 `n_prt <= N_PRT_MAX`
* 热路径不做任何 `new/malloc`

---

## 2）`CpiOutputPool`

**建议**：固定深度，与 `CTX_POOL_DEPTH` 同级或略大。
因为一个输出对象通常对应一个 `SEALED/TOMBSTONE` 上下文。

---

## 3）容量建议公式

设：

* `P = N_PRT_MAX`
* 每 PRT 固定 36 包
* 每槽位步长固定 2048

则：

```text
slot_count_max = P * 36
payload_bytes  = slot_count_max * 2048
valid_len_bytes = slot_count_max * 2
prt_summary_bytes ≈ P * 16
context_total ≈ payload_bytes + valid_len_bytes + prt_summary_bytes + header/cold
```

可以近似写成：

```text
context_total ≈ P * 36 * 2048 + 小量元数据
```

也就是：

```text
context_total ≈ P * 73728 bytes + 小量元数据
```

---

## 4）两个示例

### 当 `N_PRT_MAX = 64`

* 槽位数：`64 * 36 = 2304`
* payload 区：`2304 * 2048 = 4,718,592 B`
* 约 **4.50 MiB**
* 加元数据后，约 **4.51 ~ 4.55 MiB / CPI**

### 当 `N_PRT_MAX = 128`

* 槽位数：`4608`
* payload 区：`9,437,184 B`
* 约 **9.00 MiB**
* 加元数据后，约 **9.02 ~ 9.10 MiB / CPI**

所以这套方案的内存消耗主要由 `N_PRT_MAX` 决定。
**`N_PRT_MAX` 必须尽快冻结。**

# 十二、这套方案里哪些字段必须有，哪些不要有

## 必须有

### `CpiContext` 头部

* `cpi_id`
* `wave_id`
* `schedule_version`
* `expected_n_prt`
* `expected_slot_count`
* `received_slot_count`
* `state`
* `decision`
* `trigger_bits`
* `first_rx_tsc`
* `last_rx_tsc`
* `seal_tsc`

### `PRT` 摘要

* `ch_pkt_bitmap[4]`

## 不建议有

### 每槽位不建议放

* reason code
* 重复计数
* 非法计数
* 复杂状态枚举
* 调试字符串
* 链表指针
* 动态容器

### `CpiContext` 不建议放

* 多份原始 IQ 副本
* 面向调试的大块日志区
* 每个异常类别的重型明细
* 下游回写句柄

这和“复杂异常降级到 reason code、调试能力不能支配主链路”的冻结方向一致。

# 十五、这版方案里仍需尽快补冻结的 4 个参数

虽然方案已完整可落，但这 4 个量必须尽快冻结：

1. **`N_PRT_MAX` 工程上界**
2. **`CTX_POOL_DEPTH`**
3. **`OUTPUT_POOL_DEPTH`**
4. **下游“有效业务数据”最小门槛**

其中最关键的是 **`N_PRT_MAX`**。
因为它直接决定单个 `CpiContext` 多大。
